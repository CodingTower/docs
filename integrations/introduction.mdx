**ZenOpus integrations** let you connect your apps to **external tools, MCP servers, and APIs** to add functionality, use real data as context, and integrate third-party services securely.

With ZenOpus, you can integrate:

- **Shared connectors** to add capabilities to deployed apps
- **Personal connectors (MCP servers)** to provide context during app creation
- **Any API** for custom functionality

## ZenOpus integrations overview

| Category | Purpose | Examples |
| --- | --- | --- |
| Shared connector | Add capabilities and external services that your deployed app can call. Configured once by admins, available to everyone in your workspace. | ZenOpus Cloud, ZenOpus AI, Stripe, Shopify, Firecrawl |
| Personal connector (MCP server) | Connect your personal tool accounts to provide context while building. Only you can access your connections. | Linear, Notion, Miro, any custom MCP server |
| Any API | Integrate custom or third-party APIs not built into ZenOpus when your deployed app needs custom functionality | Any API |

<Info>
**What is an API?**  

 API (Application Programming Interface) is a structured way for services or applications to communicate and share data with each other. It defines what you can ask for, what you’ll get back, and how to make the request.
</Info>

## Shared connectors: Add capabilities to your published app

Shared connectors are the easiest way to extend the power of your ZenOpus app. They act like plug-ins that your app can call whenever it needs to perform an action or fetch data.

Shared connectors are ideal when your **deployed app** needs to interact with external services to gain operational capabilities.

### Overview and use cases

- [ZenOpus Cloud](/integrations/cloud): providing built-in backend services, authentication, database, and full-stack hosting
- [ZenOpus AI](/integrations/ai): powering your app with advanced AI capabilities
- [ElevenLabs](/integrations/eleven-labs): generating audio and text-to-speech
- [Firecrawl](/integrations/firecrawl): scraping, crawling, and extracting website content
- [Perplexity](/integrations/perplexity): answering questions and running web-backed research
- [Shopify](/integrations/shopify): managing ecommerce operations
- [Slack](/integrations/slack): sending alerts, reading channels, and posting updates
- [Stripe](/integrations/stripe): handling payments or subscriptions
- [Supabase](/integrations/supabase): authenticating users and storing data
- [Telegram](/integrations/telegram): sending messages and receiving commands through bots
- [Twilio](/integrations/twilio): sending SMS and MMS messages, making voice calls, and adding phone-based communication flows
    <Tip>
    When you need backend functionality, such as authentication, database, and storage, use [ZenOpus Cloud](/integrations/cloud), our built-in full-stack hosting platform.
    </Tip>

### Key characteristics

Shared connectors are workspace-level integrations that provide reusable app capabilities across projects.

- Managed in **Settings → Connectors → Shared connectors**
- Configured once at the workspace level
- Available to everyone in the workspace and reusable across all projects
- Provide functional capabilities your app can call while in use

On **Business and Enterprise plans**, workspace admins and owners can manage shared connector availability for the workspace:

- Enable or disable specific shared connectors using the connector settings (cogwheel icon) in **Settings → Connectors → Shared connectors**

<Note>
ZenOpus Cloud cannot be disabled for the workspace.
</Note>

### Gateway-based connectors

Some shared connectors use a **secure gateway** between your ZenOpus app and the third-party service (for example, Slack).

With gateway-based connectors:

- **Authentication is handled automatically**The gateway manages OAuth and refreshes expiring tokens in the background, so once connected, your integration keeps working without manual re-authentication.
- **Credentials are never exposed in your project**Tokens are stored securely in the gateway and are not visible in project settings or accessible to workspace admins. Your app simply calls the connector, and authentication is handled behind the scenes
- **There is a per-project usage limit**Gateway connectors allow up to **1,000 requests per minute per connector per project**. Each project has its own independent limit, so usage in one project does not affect others. Most apps will not approach this threshold, but if you are building high-volume automations or frequent background operations, you should design with this limit in mind.

Other shared connectors may use different authentication models depending on the service. Some connect directly using API keys, while others are fully managed by ZenOpus. The gateway model is specifically used for OAuth-based services that require secure token handling and automatic refresh.

## Personal connectors: Provide context during app creation

[Personal connectors (MCP servers)](/integrations/mcp-servers) give the ZenOpus Agent access to your internal tools and content, **during app creation**. They provide context so the agent can build apps that reflect your real workflows.

Use personal connectors when your app should be created using the real content your organization already relies on.

### Overview and use cases

- **Amplitude:** using product analytics and feedback to inform app decisions
- **Atlassian (Jira and Confluence):** aligning with tickets, roadmaps, and documentation
- **Granola:** turning meeting notes, decisions, and action items into specs and tasks
- **Linear:** importing issues and specs to generate features
- **Miro:** accessing your Miro boards and diagrams to generate interactive prototypes
- **Notion:** referencing docs and pages to guide app behavior
- **n8n:** mirroring or plugging into existing automation workflows
- **Polar:** using billing, products, and subscription data to scaffold SaaS apps
- **Sanity:** accessing CMS content and schemas to build content-driven apps
- **Custom MCP servers:** connecting internal or third-party systems

### Key characteristics

Personal connectors let you connect your own tools to provide context while building (not app functionality).

- Managed in **Settings → Connectors → Personal connectors**
- Per-user connections, meaning only you can access and manage your connections
- Not shared with other workspace members
- Not included in your deployed app

On **Business and Enterprise plans**, workspace admins and owners can manage personal connector availability for the workspace:

- Enable or disable all **MCP server access** in **Settings → Privacy & security**
- Enable or disable specific MCP servers using the connector settings (cogwheel icon) in **Settings → Connectors → Personal connectors**

## Integrate any API with ZenOpus

ZenOpus can integrate with **any external API**, public or private, authenticated or not. This is the most flexible integration type, allowing you to bring in functionality ZenOpus does not natively support.

When you ask ZenOpus to integrate an API, it automatically chooses the correct approach:

- **If the API requires no authentication**, your app calls the API directly.
- **If the API requires authentication**, ZenOpus prompts you to enable Cloud and add a secret, and automatically creates an Edge Function to protect sensitive credentials.

### APIs without authentication

If an API requires no credentials, there is no setup required. Just ask ZenOpus to integrate the API. For example:

```text
Integrate Chuck Norris Jokes: https://api.chucknorris.io/jokes/random
```

ZenOpus detects that no key is required and adds the feature directly to your app.

### Authenticated APIs (requires secrets)

For APIs that require authentication credentials or API keys, ZenOpus handles the integration securely using ZenOpus Cloud. Credentials are stored as secrets in the linked Cloud project and accessed only through Edge Functions, ensuring they remain protected.

**Setup**

1. Ask ZenOpus to integrate the API  and include key details:
    - Endpoint URLs and HTTP methods
    - Authentication method (API key header, query param, Bearer token, OAuth)
    - Required headers (Content-Type, custom headers)
    - Request and response format examples (JSON, params, bodies)
    - OpenAPI spec or documentation link
1. Enable [ZenOpus Cloud](/integrations/cloud).
1. Add your API key under **Cloud → Secrets.**

**For example:**

```text
Integrate the OpenWeatherMap API. 
Base URL: https://api.openweathermap.org/data/2.5. 
Auth: API key passed as a query parameter appid. 
I need an endpoint to fetch the current weather: GET /weather?q={city}&units=metric&appid={API_KEY}. 
Docs: https://openweathermap.org/current
```

ZenOpus generates the integration with an Edge Function and a weather widget where users can enter a city and fetch weather data.

## FAQ

<AccordionGroup>
<Accordion title="What integrations does ZenOpus support?">
ZenOpus supports three types of integrations: **shared connectors**, **personal connectors (MCP servers)**, and **any API**. Shared connectors add functionality to deployed apps, personal connectors provide context during app creation, and API integrations let you connect to any external service not built into ZenOpus.
</Accordion>
<Accordion title="What’s the difference between shared connectors and personal connectors in ZenOpus?">
**Shared connectors** are used by deployed apps to call external services and are configured once at the workspace level. **Personal connectors (MCP servers)** are used only during app creation to give the ZenOpus Agent access to your tools and content, and are never included in the deployed app.
</Accordion>
<Accordion title="Can ZenOpus integrate with any API?">
Yes. ZenOpus can integrate with **any external API**, public or private. APIs without authentication can be called directly, while authenticated APIs are integrated securely using ZenOpus Cloud, secrets, and Edge Functions to protect credentials.
</Accordion>
</AccordionGroup>
