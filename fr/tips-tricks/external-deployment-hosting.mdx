Ce guide fournit des **exemples pratiques et des instructions** pour déployer et héberger des applications ZenOpus en dehors de ZenOpus Cloud.

Il s'appuie sur les concepts expliqués dans [Options de déploiement, d'hébergement et de propriété avec ZenOpus Cloud](/fr/tips-tricks/deployment-hosting-ownership) et se concentre sur les chemins de migration les plus courants. Vous pouvez adopter n’importe lequel de ces scénarios indépendamment et dans n’importe quel ordre.

Dans ce guide, **hébergement** fait référence à l'endroit où votre application s'exécute, tandis que **déploiement** fait référence à la manière dont votre code est créé et livré dans cet environnement.

Si vous prévoyez de tout exécuter sur ZenOpus Cloud, vous n'avez pas besoin de ce guide.

## Avant de migrer

Ce guide est destiné aux équipes ayant des exigences spécifiques, telles que des contraintes de conformité, des besoins en matière de résidence des données ou des politiques d'infrastructure organisationnelle, qui nécessitent l'exécution de parties d'une application en dehors de ZenOpus Cloud.

ZenOpus Cloud fournit déjà un environnement de développement et de production intégré, comprenant :

- Hébergement de production avec domaines personnalisés et SSL automatique
- Déploiements automatisés et gestion de l'environnement
- Authentification gérée avec sécurité au niveau des lignes
- Stockage de bases de données et de fichiers
- Environnements de prévisualisation automatique pour chaque modification
- Agent IA avec débogage dans les environnements de prévisualisation et de développement
- Accès au fournisseur d'IA géré pour les fonctionnalités d'exécution
- Configuration OAuth intégrée et actualisation des jetons
- Analyse de sécurité et infrastructure prête pour la conformité (SOC 2 Type 2, ISO 27001)

La plupart des équipes n’ont jamais besoin de migrer. Vous pouvez commencer le ZenOpus Cloud et déplacer les composants plus tard si et quand vous rencontrez des contraintes réelles.

## Prérequis : Connectez votre projet à GitHub

Tous les scénarios de ce guide nécessitent [la connexion de votre projet à GitHub](/fr/integrations/github).

Lorsque GitHub est connecté :

- Votre projet ZenOpus reste en synchronisation continue avec votre référentiel
- Les plateformes externes se déploient directement depuis GitHub
- ZenOpus continue de gérer le développement, les aperçus et les outils de manière indépendante

À partir de maintenant, ce guide suppose que votre projet est connecté à GitHub.

## Héberger le frontend de production sur une plateforme gérée

Il s’agit de la première étape la plus courante lorsque l’on quitte ZenOpus Cloud.

Vous déployez l'interface de production sur une plate-forme d'hébergement gérée tout en continuant à utiliser ZenOpus pour le développement et les versions préliminaires. Votre backend et vos données peuvent rester sur ZenOpus Cloud ou s'exécuter ailleurs.

### De quoi vous êtes responsable

Lorsque l'interface de production s'exécute en dehors de ZenOpus Cloud, vous êtes responsable de :

- Pipelines de déploiement frontend et restaurations
- Variables de l'environnement de production
- Comportement CDN et mise en cache
- Disponibilité et disponibilité du frontend
- Journaux de production et historique de déploiement
- Environnements de prévisualisation pour les branches ou les versions de production

ZenOpus ne peut pas surveiller ou déboguer une infrastructure de production qu'il ne contrôle pas.

### Approches communes

- **Plateformes d'hébergement basées sur Git**  
Ces plateformes se connectent directement à votre référentiel GitHub et se construisent et se déploient automatiquement à chaque push :
    - Netlifier
    - Pages Cloudflare
    - Vercel
    - Hébergement AWS Amplify
    - Applications Web statiques Azure
    - Hébergement Google Firebase
- **Stockage d'objets + hébergement CDN**  
Ces plates-formes hébergent des fichiers statiques derrière un CDN mais nécessitent un pipeline de build pour générer et télécharger la sortie `dist/`.
    - **AWS :** S3 + CloudFront
    - **Google Cloud :** Stockage cloud + Cloud CDN
    - **Azure :** Stockage Azure (site Web statique) + Azure CDN ou Front Door

### Déploiement sur une plateforme d'hébergement basée sur Git

Cette approche s'applique aux plateformes qui sont automatiquement créées et déployées à partir de votre référentiel GitHub.

<Accordion title="Exemple : déploiement sur une plate-forme d'hébergement basée sur Git">
<Steps>
<Step title="Connectez votre référentiel">
Connectez votre référentiel GitHub à la plateforme d'hébergement.
</Step>
<Step title="Configurer les paramètres de construction">
La plupart des plateformes les détectent automatiquement :

- **Commande de construction** : `npm run build`
- **Répertoire de sortie** : `dist`
- **Version du nœud** : 22
</Step>
<Step title="Configurer les variables d'environnement">
Les projets sans backend ne nécessitent aucune variable d'environnement.

Les projets utilisant ZenOpus Cloud ont besoin des valeurs du fichier `.env` de votre projet :

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
VITE_SUPABASE_PROJECT_ID
```

Vous pouvez les trouver dans le fichier `.env` dans l'éditeur de code de ZenOpus ou dans votre référentiel GitHub synchronisé.
</Step>
<Step title="Configurer le routage SPA">
Si la navigation URL directe renvoie un `404`, configurez une réécriture de secours afin que toutes les routes servent `/index.html`. La méthode varie selon la plateforme (par exemple, fichier `_redirects` sur Netlify/Cloudflare, `staticwebapp.config.json` sur Azure, règles de réécriture sur Amplify, `vercel.json` sur Vercel, `firebase.json` sur Firebase).
</Step>
<Step title="Mettre à jour les URL de redirection OAuth">
Si votre application utilise la connexion Google ou d'autres fournisseurs OAuth, ajoutez votre nouveau domaine de production aux URL de redirection autorisées par votre fournisseur d'authentification.
</Step>
<Step title="Déployer">
Chaque poussée vers GitHub déclenche un nouveau déploiement de production.
</Step>
</Steps>

**Résultat :** Une interface de production accessible au public servie via HTTPS, connectée à votre backend ZenOpus Cloud.
</Accordion>

### Déploiement sur stockage objet + CDN avec CI/CD

L'hébergement basé sur CDN nécessite un pipeline CI/CD pour créer votre application et télécharger la sortie `dist/`. Les étapes de construction sont identiques d’un fournisseur à l’autre. Le déploiement est spécifique au fournisseur, voir les liens vers la documentation officielle de chaque plateforme.

<Accordion title="Exemple : Déploiement sur stockage objet + CDN avec CI/CD">
Pour l'authentification, tous les principaux fournisseurs de cloud Support **OpenID Connect (OIDC)** avec GitHub Actions, ce qui élimine le besoin de stocker des informations d'identification de longue durée sous forme de secrets. C'est l'approche recommandée.

- [AWS : Utiliser les rôles IAM avec OIDC](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Google Cloud : Fédération d'identité de charge de travail](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)
- [Azure : informations d'identification d'identité fédérée](https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation)

<Steps>
<Step title="Ajouter des secrets GitHub">
Ajoutez les secrets requis dans les paramètres de votre référentiel GitHub (**Paramètres → Secrets et variables → Actions**).

| Secrets | Valeur |
| --- | --- |
| VITE_SUPABASE_URL | URL de votre projet Supabase (à partir du fichier .env de votre projet) |
| VITE_SUPABASE_PUBLISHABLE_KEY | Votre clé publiable Supabase (à partir du fichier .env de votre projet) |
</Step>
<Step title="Créer le fichier de workflow">
Par exemple, créez `.github/workflows/deploy.yml` dans votre référentiel :

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build application
        run: npm run build
        env:
          VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
          VITE_SUPABASE_PUBLISHABLE_KEY: ${{ secrets.VITE_SUPABASE_PUBLISHABLE_KEY }}

      # Deploy step is provider-specific. See documentation links below.
      # The build output is in the dist/ directory.
```
</Step>
<Step title="Ajouter des étapes de déploiement spécifiques au fournisseur">
Après l'étape de création, ajoutez des étapes de déploiement pour votre plateforme. Reportez-vous à la documentation officielle pour connaître les meilleures pratiques actuelles :

| Plateforme | Documents |
| --- | --- |
| AWS S3 + CloudFront | Déploiement sur Amazon S3, actions GitHub pour AWS |
| Stockage Google Cloud | Hébergement d'un site Web statique, GitHub Actions pour Google Cloud |
| Stockage Blob Azure | Hébergement de sites Web statiques dans Azure Storage, GitHub Actions pour Azure |
</Step>
<Step title="Configurer le routage SPA">
Tous les hébergements basés sur CDN nécessitent une configuration pour renvoyer `index.html` avec un code d'état **200** pour les routes qui ne correspondent pas à un fichier. Ceci est généralement configuré comme :

- Une **réponse d'erreur personnalisée renvoyant 200** (CloudFront)
- Une **règle de réécriture d'URL sur un équilibreur de charge** (Cloud CDN)
- Une **règle de réécriture d'URL** (Front Door)

<Warning>
Ne vous fiez pas aux paramètres de la page d'erreur au niveau du stockage, car ils renvoient 404 codes d'état.
</Warning>
</Step>
</Steps>
</Accordion>

## Hébergez le frontend de production sur votre propre infrastructure

Utilisez cette approche lorsque vous avez besoin d’un contrôle total sur l’hébergement frontal, la mise en réseau ou l’environnement d’exécution. Ceci est parfois appelé **auto-hébergement**.

Le frontend est construit à partir de votre référentiel GitHub et déployé sur l'infrastructure que vous gérez. Le backend et les données peuvent rester sur ZenOpus Cloud ou s'exécuter ailleurs.

### De quoi vous êtes responsable

Lorsque l'interface de production s'exécute sur l'infrastructure que vous gérez, vous êtes responsable de :

- Automatisation de la création et du déploiement
- Configuration SSL/TLS
- Configuration du CDN et du proxy inverse
- Surveillance, journalisation et disponibilité
- Mises à jour et sécurité de l'infrastructure
- Aperçu des environnements pour les branches ou les versions

ZenOpus ne peut pas surveiller ou déboguer une infrastructure de production qu'il ne contrôle pas.

### Approches communes

- **Déploiements basés sur des conteneurs**  
Par exemple : Docker déployé via Kubernetes (EKS, GKE, AKS), ECS, Nomad ou des plateformes de conteneurs internes
- **Machines virtuelles derrière un serveur Web**  
Par exemple : machines virtuelles Linux exécutant Nginx ou Apache, gérées via la gestion de la configuration ou des outils internes
- **Plateformes PaaS internes**  
Par exemple : plateformes de déploiement internes à l’entreprise ou solutions PaaS dans le cloud privé

### Exigences de construction

Les projets ZenOpus sont des applications Vite standard et sont construits comme des interfaces statiques.

- **Commande de construction :**`npm run build`
- **Répertoire de sortie :**`dist/`
- **Version du nœud :** 22 recommandées

Les variables d'environnement préfixées par `VITE_` sont intégrées au **temps de construction**, pas au moment de l'exécution.

Si vous utilisez ZenOpus Cloud comme backend, vous devez les définir avant d'exécuter `npm run build` :

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_PUBLISHABLE_KEY`

Vous pouvez trouver ces valeurs dans le fichier `.env` de votre projet.

### Déploiement basé sur des conteneurs (Docker)

<Note>
Vous pouvez générer ces configurations Docker directement avec l'agent ZenOpus. Consultez [Utilisation de ZenOpus pour générer des déploiements Docker](#utiliser-zenopus-pour-générer-des-déploiements-docker) ci-dessous.
</Note>

<AccordionGroup>
<Accordion title="Exemple : déploiement manuel de Docker">
Il s'agit de l'approche la plus courante pour le déploiement sur Kubernetes, ECS, Cloud Run ou toute autre plateforme d'orchestration de conteneurs. Cela fonctionne également pour exécuter le frontend en tant que conteneur autonome sur une seule VM.

<Steps>
<Step title="Clonez votre référentiel GitHub">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Créer un fichier Docker">
Les projets ZenOpus peuvent être conteneurisés à l'aide d'une construction Docker standard en plusieurs étapes : la première étape installe les dépendances et exécute `npm run build`, et la deuxième étape sert la sortie statique avec nginx.

Voici un exemple de Dockerfile que vous pouvez adapter :

```docker
# Build stage
FROM node:22-alpine AS builder

WORKDIR /app

# Copy package files and install dependencies
COPY package*.json ./
RUN npm ci

# Copy source code
COPY . .

# Environment variables are embedded at build time
ARG VITE_SUPABASE_URL
ARG VITE_SUPABASE_PUBLISHABLE_KEY
ENV VITE_SUPABASE_URL=$VITE_SUPABASE_URL
ENV VITE_SUPABASE_PUBLISHABLE_KEY=$VITE_SUPABASE_PUBLISHABLE_KEY

# Build the application
RUN npm run build

# Production stage
FROM nginx:alpine

# Copy built assets to nginx
COPY --from=builder /app/dist /usr/share/nginx/html

# Copy nginx configuration for SPA routing
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```
</Step>
<Step title="Créer un fichier de configuration nginx">
Les projets ZenOpus utilisent le routage côté client (React Router avec BrowserRouter), votre serveur Web doit donc renvoyer `index.html` pour toutes les routes. Créez un fichier `nginx.conf` :

```nginx
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    # Cache static assets
    location /assets/ {
    	expires 1y;
    	add_header Cache-Control "public, immutable";
	}

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```
</Step>
<Step title="Construire l'image du conteneur">
Définissez vos variables d'environnement comme arguments de construction :

```bash
docker build \
  --build-arg VITE_SUPABASE_URL=<your-url> \
  --build-arg VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
  -t my-zenopus-app .
```
</Step>
<Step title="Transférer vers votre registre de conteneurs">
Par exemple :

```bash
docker tag my-zenopus-app your-registry/my-zenopus-app:latest
docker push your-registry/my-zenopus-app:latest
```
</Step>
<Step title="Déployez à l'aide de votre plateforme d'orchestration">
Déployez le conteneur à l'aide de Kubernetes, ECS, Cloud Run, Nomad ou de votre plateforme interne. Aucune variable d'environnement d'exécution n'est nécessaire, elles sont déjà intégrées dans la version.
</Step>
</Steps>

**Résultat :** Un conteneur en cours d'exécution servant votre interface via HTTP sur le port 80.

<Warning>
Les variables d'environnement sont intégrées au moment de la construction. Pour les modifier, vous devez reconstruire l'image du conteneur.
</Warning>
</Accordion>
<Accordion title="Exemple : déploiement automatisé de Docker (CI/CD)">
Ce workflow crée une image Docker et la transfère vers un registre de conteneurs pour le déploiement sur Kubernetes, ECS ou d'autres plates-formes d'orchestration. Cela automatise les étapes de déploiement manuel basées sur des conteneurs décrites ci-dessus.

<Note>
Vous devrez probablement l'adapter à vos exigences de sécurité, aux pipelines existants et aux politiques organisationnelles.
</Note>

Pour l'authentification, les principaux registres de conteneurs Support **OpenID Connect (OIDC)** avec GitHub Actions, ce qui élimine le besoin de stocker des informations d'identification de longue durée sous forme de secrets. C'est l'approche recommandée.

- [AWS ECR : Utiliser les rôles IAM avec OIDC](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Google Artifact Registry : Fédération d'identité de charge de travail](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)
- [Azure Container Registry : informations d'identification d'identité fédérée](https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation)
- [GitHub Container Registry : utilisez GITHUB_TOKEN](https://docs.github.com/en/packages/working-with-a-github-packages-registry/working-with-the-container-registry) (aucun secret requis)

**Prérequis :**

- Un registre de conteneurs (par exemple, GitHub Container Registry, AWS ECR, Google Artifact Registry, Azure Container Registry, Docker Hub)
- Les fichiers `Dockerfile` et `nginx.conf` de la section de déploiement manuel basé sur un conteneur ci-dessus
- Une plateforme d'orchestration pour déployer le conteneur (par exemple, Kubernetes, ECS)

<Steps>
<Step title="Ajouter des secrets GitHub">
Ajoutez les secrets requis dans les paramètres de votre référentiel GitHub (**Paramètres → Secrets et variables → Actions**).

| Secrets | Valeur |
| --- | --- |
| VITE_SUPABASE_URL | URL de votre projet Supabase (à partir du fichier .env de votre projet) |

</Step>
<Step title="Créer le fichier de workflow">
Par exemple, créez `.github/workflows/deploy-container.yml` dans votre référentiel :

```yaml

  push:
    branches: [main]

env:
  IMAGE_NAME: my-zenopus-app
  REGISTRY: ghcr.io/your-org  # Replace with your registry URL

jobs:
  build-and-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # TODO: Add registry authentication step here.
      # This workflow will not run until you configure authentication.
      # See "Registry-specific authentication" table below.

      - name: Build and push Docker image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          build-args: |
            VITE_SUPABASE_URL=${{ secrets.VITE_SUPABASE_URL }}
            VITE_SUPABASE_PUBLISHABLE_KEY=${{ secrets.VITE_SUPABASE_PUBLISHABLE_KEY }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```
</Step>
<Step title="Configurer l'authentification du registre">
Ajoutez l'étape d'authentification appropriée avant **Créer et envoyer une image Docker** dans le flux de travail. Reportez-vous à la documentation officielle pour connaître les meilleures pratiques actuelles :

| Centre Docker | Action de connexion Docker |
</Step>

### Déploiement de VM ou de serveur statique

<Accordion title="Exemple : Déploiement manuel d'une VM ou d'un serveur statique">
Cette approche est destinée au déploiement sur des machines virtuelles Linux, des serveurs nus ou toute machine exécutant un serveur Web comme Nginx ou Apache.

<Steps>
<Step title="Clonez votre référentiel GitHub">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Installer les dépendances">
```bash
npm ci
```
</Step>
<Step title="Construire l'application avec des variables d'environnement">
Définissez les variables d'environnement avant d'exécuter la build :

```bash
VITE_SUPABASE_URL=<your-url> \
VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
npm run build
```

La sortie de build se trouve dans le répertoire `dist/`.
</Step>
<Step title="Téléchargez la sortie de build sur votre serveur">
Par exemple, en utilisant `scp` :

```bash
scp -r dist/* user@your-server:/var/www/html/
```

Ou en utilisant `rsync` :

```bash
rsync -avz dist/ user@your-server:/var/www/html/
```
</Step>
<Step title="Configurez votre serveur Web">
Les projets ZenOpus utilisent le routage côté client (React Router avec BrowserRouter), votre serveur Web doit donc renvoyer `index.html` pour toutes les routes.

**Exemple de configuration nginx :**

```nginx
server {
    listen 80;
    server_name your-domain.com;
    root /var/www/html;
    index index.html;

    # Cache static assets
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

**Exemple de configuration Apache (.htaccess) :**

```apache
RewriteEngine On
RewriteBase /
RewriteRule ^index\.html$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /index.html [L]
```
</Step>
<Step title="Configurer TLS (recommandé)">
Utilisez [Certbot](https://certbot.eff.org/) pour ajouter un certificat Let's Encrypt gratuit ou configurer vos certificats SSL existants.
</Step>
</Steps>

**Résultat :** Votre application est diffusée via HTTP(S) depuis votre serveur.

<Warning>
Les variables d'environnement sont intégrées dans la sortie de construction. Aucune configuration d'exécution n'est nécessaire sur le serveur. Pour modifier les variables d'environnement, reconstruisez l'application et téléchargez à nouveau la sortie `dist/`.
</Warning>
</Accordion>

## Héberger le backend et les données sur un fournisseur géré (exemple Supabase)

Cette option est généralement choisie lorsque vous avez besoin d'un accès direct à la base de données, de fonctionnalités de base de données avancées ou d'une séparation plus claire de la propriété de l'infrastructure, sans assumer l'entière responsabilité opérationnelle.

Vous déplacez vos services backend et votre base de données vers un fournisseur backend géré. Le chemin de migration le plus direct consiste à **Supabase géré**, qui correspond étroitement à l'architecture backend par défaut de ZenOpus Cloud. L'interface de production peut s'exécuter sur ZenOpus Cloud ou ailleurs.

Après la migration :

- Le frontend de production doit pointer vers le nouveau backend
- Vous pouvez continuer à utiliser l'éditeur ZenOpus et les environnements de prévisualisation pendant le développement.

Ce guide utilise **Supabase** comme chemin de migration de référence, car les applications ZenOpus s'appuient sur des services compatibles avec Supabase (authentification, stockage, temps réel, fonctions Edge et sécurité au niveau des lignes).

La migration vers d'autres plates-formes back-end est possible, mais peut nécessiter la mise en œuvre de services d'authentification, de stockage et de back-end équivalents en fonction de votre fournisseur.

Les étapes détaillées ci-dessous décrivent comment migrer un projet de ZenOpus Cloud vers une instance Supabase gérée.

### De quoi vous êtes responsable

Lorsque votre backend s'exécute en dehors de ZenOpus Cloud, vous êtes responsable des fonctionnalités backend précédemment gérées par ZenOpus, notamment :

- Disponibilité, mise à l'échelle et sauvegardes de la base de données
- Surveillance back-end et réponse aux incidents
- Configuration et maintenance de la sécurité au niveau des lignes
- Configuration du fournisseur d'authentification
- Identifiants OAuth, URL de redirection et rotation des secrets
- Variables d'environnement back-end et configuration
- Analyse de sécurité pour détecter les erreurs de configuration et les secrets exposés
- Statut de conformité de votre infrastructure backend

ZenOpus ne peut pas surveiller ou déboguer l'infrastructure backend qu'il ne contrôle pas. La configuration OAuth gérée et l'actualisation automatique des jetons ne sont disponibles que lorsque le backend s'exécute sur ZenOpus Cloud.

### Qu'est-ce qui migre et comment

| Composants de l'application | Méthode de migration | Remarques |
| --- | --- | --- |
| Schéma de base de données | Automatique via les migrations SQL | Comprend des tables, des colonnes, des index, des politiques RLS, des fonctions et des déclencheurs |
| Seaux de stockage | Automatique via les migrations SQL | Inclut les politiques d'accès |
| Fournisseurs d'authentification | Manuel | Reconfigurez l'authentification (par exemple, Google OAuth, GitHub) dans votre nouvel environnement d'hébergement |
| Variables d'environnement et secrets | Manuel | Reconfigurez les clés API, les jetons ou les informations d'identification pour les services externes (par exemple, Stripe) dans votre nouvel environnement d'hébergement |
| Données (tableau des matières) | Manuel | Exporter/importer au format CSV |
| Fichiers de stockage | Manuel | Télécharger/télécharger manuellement |
| Comptes utilisateurs | Manuel partiel | Vous pouvez exporter les données utilisateur de la base de données, mais vous ne pouvez pas exporter les mots de passe utilisateur, vous devez donc déclencher un flux de réinitialisation de mot de passe. Nous vous recommandons de planifier la migration avant d'intégrer de vrais utilisateurs. |

## Hébergez le backend et les données sur votre propre infrastructure (exemple Supabase)

Cette option est destinée aux exigences strictes de conformité, de résidence des données ou de contrôle de l’infrastructure.

Vous exécutez le backend et la base de données sur l’infrastructure que vous exploitez. Le chemin auto-hébergé le plus direct est **Supabase auto-hébergé**, qui fournit les services d'authentification, de stockage, de temps réel et de fonction Edge dont dépendent les applications ZenOpus.

L'interface de production peut rester sur ZenOpus Cloud ou ailleurs. ZenOpus peut toujours être utilisé pour le développement, ou le développement peut passer complètement à d'autres outils.

Exécuter uniquement une base de données PostgreSQL autonome n'est pas suffisant, sauf si vous implémentez des services équivalents d'authentification, de stockage, de temps réel et de périphérie.

### De quoi vous êtes responsable

Lorsque le backend s'exécute sur l'infrastructure que vous exploitez, vous êtes responsable des fonctionnalités backend ZenOpus précédemment gérées, notamment :

- Opérations PostgreSQL, sauvegardes et reprise après sinistre
- Authentification, stockage et disponibilité et fiabilité des services en temps réel
- Conception et application de la sécurité au niveau des lignes
- Application des correctifs de sécurité et gestion des mises à niveau de version
- Déploiement et exécution des fonctions Edge
- Optimisation et mise à l'échelle des performances
- Surveillance, alerte et réponse aux incidents
- Certification de conformité de votre infrastructure

ZenOpus ne surveille, n'exploite ni ne débogue aucune partie de l'infrastructure auto-hébergée, et les aperçus de production ne sont pas générés pour les environnements de production auto-hébergés.

**Lorsque vous auto-hébergez Supabase, vous :**

- Déployez Supabase dans votre propre infrastructure à l'aide du [guide officiel d'auto-hébergement de Supabase avec Docker](https://supabase.com/docs/guides/self-hosting/docker).
    <Note>
    Vous pouvez générer ces configurations Docker directement avec l'agent ZenOpus. Consultez [Utilisation de ZenOpus pour générer des déploiements Docker](#utiliser-zenopus-pour-générer-des-déploiements-docker) ci-dessous.
    </Note>
- Configurez PostgreSQL, l'authentification, le stockage et les services requis.
- Appliquez les fichiers de migration de votre projet ZenOpus (`supabase/migrations/`) à votre instance auto-hébergée.
- Mettez à jour les variables d'environnement de votre application pour qu'elles pointent vers votre Supabase auto-hébergée :
    ```text
    VITE_SUPABASE_URL=https://votre-domaine-auto-hébergé.com
    VITE_SUPABASE_PUBLISHABLE_KEY=votre-anon-clé
    ```

Pour une configuration détaillée de l'infrastructure et des conseils opérationnels, suivez la [documentation officielle d'auto-hébergement de Supabase](https://supabase.com/docs/guides/self-hosting).

## Utiliser ZenOpus pour générer des déploiements Docker

Si vous préférez un déploiement conteneurisé, vous pouvez prompt ZenOpus pour générer des configurations Docker et Docker Compose pour votre projet. L'agent fournira les détails des fichiers générés, les ports de service et exécutera les commandes dans le chat. Vous trouverez ci-dessous les trois modèles pris en charge.

### Front-end uniquement

Packagez uniquement l'application React en tant que site statique servi par Nginx. Utilisez-le lorsque vous souhaitez uniquement déplacer votre interface.

Par exemple :

```text
Dockerisez ce projet pour un déploiement frontend uniquement avec Nginx
```

### Backend uniquement (Subabase auto-hébergée)

Exécutez la pile Supabase complète localement sans regrouper le frontend. Utilisez-le lorsque vous souhaitez développer le frontend séparément ou le servir à partir d'un autre hôte.

Par exemple :

```text
Créer une configuration Docker Compose pour un backend Supabase auto-hébergé uniquement (pas de conteneur frontend)
```

### Full stack (frontend + Supabase auto-hébergée)

Regroupez le frontend et une pile Supabase auto-hébergée dans une seule configuration Docker Compose. Utilisez-le pour les déploiements entièrement autonomes.

Par exemple :

```text
Dockerisez ce projet avec un backend Supabase auto-hébergé fourni dans Docker Compose
```

### Configuration manuelle requise pour Supabase auto-hébergée

L'agent génère des secrets d'espace réservé qui doivent être remplacés avant utilisation :

1. Générez un secret JWT : `openssl rand -base64 32`
1. Générez des clés API à l'aide de votre secret JWT. Suivez la [documentation d'auto-hébergement Supabase pour générer des clés API](https://supabase.com/docs/guides/self-hosting/docker#generate-api-keys).
1. Remplacez les valeurs suivantes dans `docker-compose.yml` :
    - `JWT_SECRET`
    - `POSTGRES_PASSWORD`
    - `ANON_KEY`
    - `SERVICE_ROLE_KEY`

### Limites

- L'agent ne peut pas exécuter ou tester les builds Docker. Vous devez vérifier localement.
- L'agent ne peut pas générer de vrais secrets. Les espaces réservés doivent être remplacés manuellement.
