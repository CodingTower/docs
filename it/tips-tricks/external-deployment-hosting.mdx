Questa guida fornisce **esempi pratici e istruzioni** per la distribuzione e l'hosting di applicazioni ZenOpus all'esterno di ZenOpus Cloud.

Si basa sui concetti spiegati in [Opzioni di distribuzione, hosting e proprietà con ZenOpus Cloud](/it/tips-tricks/deployment-hosting-ownership) e si concentra sui percorsi di migrazione più comuni. È possibile adottare uno qualsiasi di questi scenari in modo indipendente e in qualsiasi ordine.

In questa guida, **hosting** si riferisce al luogo in cui viene eseguita la tua applicazione, mentre **deployment** si riferisce al modo in cui il tuo codice viene creato e distribuito in tale ambiente.

Se prevedi di eseguire tutto su ZenOpus Cloud, non hai bisogno di questa guida.

## Prima della migrazione

Questa guida è destinata ai team con requisiti specifici, come vincoli di conformità, esigenze di residenza dei dati o policy dell'infrastruttura organizzativa, che richiedono l'esecuzione di parti di un'applicazione all'esterno di ZenOpus Cloud.

ZenOpus Cloud fornisce già un ambiente integrato di sviluppo e produzione, tra cui:

- Hosting di produzione con domini personalizzati e SSL automatico
- Distribuzioni automatizzate e gestione dell'ambiente
- Autenticazione gestita con sicurezza a livello di riga
- Database e archiviazione di file
- Ambienti di anteprima automatici per ogni modifica
- Agente AI con debug negli ambienti di anteprima e sviluppo
- Accesso gestito al provider di intelligenza artificiale per funzionalità di runtime
- Configurazione OAuth integrata e aggiornamento del token
- Scansione di sicurezza e infrastruttura pronta per la conformità (SOC 2 Tipo 2, ISO 27001)

La maggior parte dei team non ha mai bisogno di migrare. Puoi iniziare il ZenOpus Cloud e spostare i componenti in un secondo momento se e quando raggiungi vincoli reali.

## Prerequisito: collega il tuo progetto a GitHub

Tutti gli scenari descritti in questa guida richiedono [la connessione del progetto a GitHub](/it/integrations/github).

Quando GitHub è connesso:

- Il tuo progetto ZenOpus rimane costantemente sincronizzato con il tuo repository
- Le piattaforme esterne vengono distribuite direttamente da GitHub
- ZenOpus continua a gestire lo sviluppo, le anteprime e gli strumenti in modo indipendente

Da qui in poi, questa guida presuppone che il tuo progetto sia connesso a GitHub.

## Ospita il frontend di produzione su una piattaforma gestita

Questo è il primo passo più comune quando ci si sposta all'esterno di ZenOpus Cloud.

Distribuisci il frontend di produzione su una piattaforma di hosting gestita continuando a utilizzare ZenOpus per lo sviluppo e le anteprime. Il tuo backend e i tuoi dati possono rimanere su ZenOpus Cloud o essere eseguiti altrove.

### Di cosa sei responsabile

Quando il frontend di produzione viene eseguito all'esterno di ZenOpus Cloud, sei responsabile di:

- Pipeline di distribuzione e rollback del frontend
- Variabili dell'ambiente produttivo
- Comportamento e memorizzazione nella cache della CDN
- Disponibilità e uptime del frontend
- Registri di produzione e cronologia di distribuzione
- Ambienti di anteprima per rami o versioni di produzione

ZenOpus non può monitorare o eseguire il debug dell'infrastruttura di produzione che non controlla.

### Approcci comuni

- **Piattaforme di hosting basate su Git**  
Queste piattaforme si connettono direttamente al tuo repository GitHub e si creano e distribuiscono automaticamente a ogni push:
    - Netlify
    - Pagine di Cloudflare
    - Vercello
    - Hosting AWS Amplificare
    - App Web statiche di Azure
    - Hosting Google Firebase
- **Archiviazione di oggetti + hosting CDN**  
Queste piattaforme ospitano file statici dietro una CDN ma richiedono una pipeline di compilazione per generare e caricare l'output `dist/`.
    - **AWS:** S3 + CloudFront
    - **Google Cloud:** Archiviazione nel cloud + CDN nel cloud
    - **Azure:** Archiviazione di Azure (sito Web statico) + CDN o front door di Azure

### Distribuzione su una piattaforma di hosting basata su Git

Questo approccio si applica alle piattaforme che vengono create e distribuite automaticamente dal repository GitHub.

<Accordion title="Esempio: distribuzione su una piattaforma di hosting basata su Git">
<Steps>
<Step title="Connetti il tuo archivio">
Connetti il tuo repository GitHub alla piattaforma di hosting.
</Step>
<Step title="Configura le impostazioni di creazione">
La maggior parte delle piattaforme rileva automaticamente questi:

- **Comando di creazione**: `npm run build`
- **Directory di output**: `dist`
- **Versione del nodo**: 22
</Step>
<Step title="Configurare le variabili d'ambiente">
I progetti senza backend non richiedono variabili di ambiente.

I progetti che utilizzano ZenOpus Cloud necessitano dei valori del file `.env` del tuo progetto:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
VITE_SUPABASE_PROJECT_ID
```

Puoi trovarli nel file `.env` nell'editor di codice di ZenOpus o nel tuo repository GitHub sincronizzato.
</Step>
<Step title="Configura il percorso SPA">
Se la navigazione diretta dell'URL restituisce un `404`, configura una riscrittura di fallback in modo che tutti i percorsi servano `/index.html`. Il metodo varia in base alla piattaforma (ad esempio, file `_redirects` su Netlify/Cloudflare, `staticwebapp.config.json` su Azure, riscrivi regole su Amplify, `vercel.json` su Vercel, `firebase.json` su Firebase).
</Step>
<Step title="Aggiorna gli URL di reindirizzamento OAuth">
Se la tua app utilizza l'accesso con Google o altri provider OAuth, aggiungi il tuo nuovo dominio di produzione agli URL di reindirizzamento consentiti dal provider di autenticazione.
</Step>
<Step title="Distribuire">
Ogni push su GitHub attiva una nuova distribuzione di produzione.
</Step>
</Steps>

**Risultato:** un frontend di produzione accessibile pubblicamente servito tramite HTTPS, connesso al tuo backend ZenOpus Cloud.
</Accordion>

### Distribuzione nello storage di oggetti + CDN con CI/CD

L'hosting supportato da CDN richiede una pipeline CI/CD per creare la tua app e caricare l'output `dist/`. I passaggi di creazione sono identici tra i fornitori. La distribuzione è specifica del provider, vedere i collegamenti alla documentazione ufficiale per ciascuna piattaforma.

<Accordion title="Esempio: distribuzione nello storage di oggetti + CDN con CI/CD">
Per l'autenticazione, tutti i principali provider cloud Support **OpenID Connect (OIDC)** con GitHub Actions, che elimina la necessità di archiviare credenziali di lunga durata come segreti. Questo è l'approccio consigliato.

- [AWS: utilizza i ruoli IAM con OIDC](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Google Cloud: Federazione delle identità del carico di lavoro](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)
- [Azure: credenziali di identità federate](https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation)

<Steps>
<Step title="Aggiungi segreti GitHub">
Aggiungi i segreti richiesti nelle impostazioni del repository GitHub (**Impostazioni → Segreti e variabili → Azioni**).

| Segreto | Valore |
| --- | --- |
| VITE_SUPABASE_URL | L'URL del tuo progetto Supabase (dal file .env del tuo progetto) |
| VITE_SUPABASE_PUBLISHABLE_KEY | La tua chiave pubblicabile Supabase (dal file .env del tuo progetto) |
</Step>
<Step title="Creare il file del flusso di lavoro">
Ad esempio, crea `.github/workflows/deploy.yml` nel tuo repository:

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build application
        run: npm run build
        env:
          VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
          VITE_SUPABASE_PUBLISHABLE_KEY: ${{ secrets.VITE_SUPABASE_PUBLISHABLE_KEY }}

      # Deploy step is provider-specific. See documentation links below.
      # The build output is in the dist/ directory.
```
</Step>
<Step title="Aggiungi passaggi di distribuzione specifici del provider">
Dopo la fase di creazione, aggiungi i passaggi di distribuzione per la tua piattaforma. Fare riferimento alla documentazione ufficiale per le migliori pratiche attuali:

| Piattaforma | Documentazione |
| --- | --- |
| AWS S3 + CloudFront | Distribuzione su Amazon S3, GitHub Actions per AWS |
| Archiviazione Google Cloud | Hosting di un sito Web statico, GitHub Actions per Google Cloud |
| Archiviazione BLOB di Azure | Hosting di siti Web statici in Archiviazione di Azure, Azioni GitHub per Azure |
</Step>
<Step title="Configura il percorso SPA">
Tutto l'hosting supportato da CDN richiede la configurazione per restituire `index.html` con un codice di stato **200** per i percorsi che non corrispondono a un file. Questo è tipicamente configurato come:

- Una **risposta di errore personalizzata che restituisce 200** (CloudFront)
- Una **regola di riscrittura dell'URL su un bilanciatore del carico** (Cloud CDN)
- A **Regola di riscrittura URL** (Porta anteriore)

<Warning>
Non fare affidamento sulle impostazioni della pagina di errore a livello di archiviazione, poiché restituiscono codici di stato 404.
</Warning>
</Step>
</Steps>
</Accordion>

## Ospita il frontend di produzione sulla tua infrastruttura

Utilizza questo approccio quando hai bisogno del controllo completo sull'hosting frontend, sulla rete o sull'ambiente runtime. Questo a volte viene definito **self-hosting**.

Il frontend è creato dal tuo repository GitHub e distribuito nell'infrastruttura che gestisci. Il backend e i dati possono rimanere su ZenOpus Cloud o essere eseguiti altrove.

### Di cosa sei responsabile

Quando il frontend di produzione viene eseguito sull'infrastruttura che gestisci, sei responsabile di:

- Automazione della creazione e della distribuzione
- Configurazione SSL/TLS
- Configurazione CDN e proxy inverso
- Monitoraggio, registrazione e uptime
- Aggiornamenti e sicurezza dell'infrastruttura
- Anteprima degli ambienti per rami o versioni

ZenOpus non può monitorare o eseguire il debug dell'infrastruttura di produzione che non controlla.

### Approcci comuni

- **Distribuzioni basate su container**  
Ad esempio: Docker distribuito tramite Kubernetes (EKS, GKE, AKS), ECS, Nomad o piattaforme di contenitori interni
- **Macchine virtuali dietro un server web**  
Ad esempio: VM Linux che eseguono Nginx o Apache, gestite tramite la gestione della configurazione o strumenti interni
- **Piattaforme PaaS interne**  
Ad esempio: piattaforme di distribuzione interne all'azienda o soluzioni PaaS di cloud privato

### Costruire i requisiti

I progetti ZenOpus sono applicazioni Vite standard e vengono creati come frontend statici.

- **Comando di creazione:**`npm run build`
- **Directory di output:**`dist/`
- **Versione nodo:** 22 consigliata

Le variabili di ambiente con il prefisso `VITE_` sono incorporate in **buildtime**, non in fase di runtime.

Se utilizzi ZenOpus Cloud come backend, devi impostarli prima di eseguire `npm run build`:

- __CODICE0__
- __CODICE0__

Puoi trovare questi valori nel file `.env` del tuo progetto.

### Distribuzione basata su container (Docker)

<Note>
Puoi generare queste configurazioni Docker direttamente con l'agente ZenOpus. Consulta [Utilizzo di ZenOpus per generare distribuzioni Docker](#utilizzo-di-zenopus-per-generare-distribuzioni-docker) di seguito.
</Note>

<AccordionGroup>
<Accordion title="Esempio: distribuzione manuale di Docker">
Questo è l'approccio più comune per la distribuzione su Kubernetes, ECS, Cloud Run o qualsiasi piattaforma di orchestrazione di container. Funziona anche per eseguire il frontend come contenitore autonomo su una singola VM.

<Steps>
<Step title="Clona il tuo repository GitHub">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Crea un file Docker">
I progetti ZenOpus possono essere containerizzati utilizzando una build Docker multifase standard: la prima fase installa le dipendenze ed esegue `npm run build`, mentre la seconda fase fornisce l'output statico con nginx.

Ecco un esempio di Dockerfile che puoi adattare:

```docker
# Build stage
FROM node:22-alpine AS builder

WORKDIR /app

# Copy package files and install dependencies
COPY package*.json ./
RUN npm ci

# Copy source code
COPY . .

# Environment variables are embedded at build time
ARG VITE_SUPABASE_URL
ARG VITE_SUPABASE_PUBLISHABLE_KEY
ENV VITE_SUPABASE_URL=$VITE_SUPABASE_URL
ENV VITE_SUPABASE_PUBLISHABLE_KEY=$VITE_SUPABASE_PUBLISHABLE_KEY

# Build the application
RUN npm run build

# Production stage
FROM nginx:alpine

# Copy built assets to nginx
COPY --from=builder /app/dist /usr/share/nginx/html

# Copy nginx configuration for SPA routing
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```
</Step>
<Step title="Crea un file di configurazione nginx">
I progetti ZenOpus utilizzano il routing lato client (React Router con BrowserRouter), quindi il tuo server web deve restituire `index.html` per tutti i percorsi. Crea un file `nginx.conf`:

```nginx
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    # Cache static assets
    location /assets/ {
    	expires 1y;
    	add_header Cache-Control "public, immutable";
	}

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```
</Step>
<Step title="Costruisci l'immagine del contenitore">
Imposta le variabili di ambiente come argomenti di compilazione:

```bash
docker build \
  --build-arg VITE_SUPABASE_URL=<your-url> \
  --build-arg VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
  -t my-zenopus-app .
```
</Step>
<Step title="Effettua il push al registro dei contenitori">
Ad esempio:

```bash
docker tag my-zenopus-app your-registry/my-zenopus-app:latest
docker push your-registry/my-zenopus-app:latest
```
</Step>
<Step title="Distribuisci utilizzando la tua piattaforma di orchestrazione">
Distribuisci il container utilizzando Kubernetes, ECS, Cloud Run, Nomad o la tua piattaforma interna. Non sono necessarie variabili di ambiente runtime, sono già integrate nel build.
</Step>
</Steps>

**Risultato:** un contenitore in esecuzione che serve il tuo frontend su HTTP sulla porta 80.

<Warning>
Le variabili di ambiente sono incorporate in fase di compilazione. Per modificarli, è necessario ricostruire l'immagine del contenitore.
</Warning>
</Accordion>
<Accordion title="Esempio: distribuzione automatizzata di Docker (CI/CD)">
Questo flusso di lavoro crea un'immagine Docker e la inserisce in un registro dei contenitori per la distribuzione su Kubernetes, ECS o altre piattaforme di orchestrazione. Ciò automatizza i passaggi di distribuzione manuale basati su contenitori descritti sopra.

<Note>
Probabilmente dovrai adattarlo ai tuoi requisiti di sicurezza, alle pipeline esistenti e ai criteri organizzativi.
</Note>

Per l'autenticazione, i principali registri di contenitori Support **OpenID Connect (OIDC)** con GitHub Actions, che elimina la necessità di archiviare credenziali di lunga durata come segreti. Questo è l'approccio consigliato.

<Steps>
<Step title="Aggiungi segreti GitHub">

| Segreto | Valore |

</Step>

```yaml

    branches: [main]

  IMAGE_NAME: my-zenopus-app
  REGISTRY: ghcr.io/your-org  # Replace with your registry URL

    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # TODO: Add registry authentication step here.
      # This workflow will not run until you configure authentication.
      # See "Registry-specific authentication" table below.

      - name: Build and push Docker image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true

            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest

          build-args: |
            VITE_SUPABASE_URL=${{ secrets.VITE_SUPABASE_URL }}
```

</Step>

### Distribuzione di VM o server statico

<Accordion title="Esempio: distribuzione manuale di VM o server statico">
Questo approccio riguarda la distribuzione su VM Linux, server bare metal o qualsiasi macchina che esegue un server Web come Nginx o Apache.

<Steps>
<Step title="Clona il tuo repository GitHub">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Installa le dipendenze">
```bash
npm ci
```
</Step>
<Step title="Costruisci l'applicazione con variabili di ambiente">
Imposta le variabili di ambiente prima di eseguire la build:

```bash
VITE_SUPABASE_URL=<your-url> \
VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
npm run build
```

L'output della compilazione si trova nella directory `dist/`.
</Step>
<Step title="Carica l'output della build sul tuo server">
Ad esempio, utilizzando `scp`:

```bash
scp -r dist/* user@your-server:/var/www/html/
```

Oppure utilizzando `rsync`:

```bash
rsync -avz dist/ user@your-server:/var/www/html/
```
</Step>
<Step title="Configura il tuo server web">
I progetti ZenOpus utilizzano il routing lato client (React Router con BrowserRouter), quindi il tuo server web deve restituire `index.html` per tutti i percorsi.

**Esempio di configurazione nginx:**

```nginx
server {
    listen 80;
    server_name your-domain.com;
    root /var/www/html;
    index index.html;

    # Cache static assets
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

**Esempio di configurazione Apache (.htaccess):**

```apache
RewriteEngine On
RewriteBase /
RewriteRule ^index\.html$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /index.html [L]
```
</Step>
<Step title="Configura TLS (consigliato)">
Utilizza [Certbot](https://certbot.eff.org/) per aggiungere un certificato Let's Encrypt gratuito o configurare i tuoi certificati SSL esistenti.
</Step>
</Steps>

**Risultato:** la tua app viene servita tramite HTTP(S) dal tuo server.

<Warning>
Le variabili di ambiente vengono integrate nell'output di compilazione. Non è necessaria alcuna configurazione di runtime sul server. Per modificare le variabili di ambiente, ricostruisci l'applicazione e ricarica l'output `dist/`.
</Warning>
</Accordion>

## Backend host e dati su un provider gestito (esempio Supabase)

Questa opzione viene generalmente scelta quando è necessario l'accesso diretto al database, funzionalità avanzate del database o una separazione più chiara della proprietà dell'infrastruttura, senza assumersi la piena responsabilità operativa.

Sposti i servizi di backend e il database su un provider di backend gestito. Il percorso di migrazione più diretto è verso **Supabase gestito**, che corrisponde strettamente all'architettura backend predefinita di ZenOpus Cloud. Il frontend di produzione può essere eseguito su ZenOpus Cloud o altrove.

Dopo la migrazione:

- Il frontend di produzione deve puntare al nuovo backend
- Puoi continuare a utilizzare l'editor ZenOpus e visualizzare in anteprima gli ambienti durante lo sviluppo

Questa guida utilizza **Supabase** come percorso di migrazione di riferimento perché le applicazioni ZenOpus si basano su servizi compatibili con Supabase (autenticazione, archiviazione, tempo reale, funzioni edge e sicurezza a livello di riga).

La migrazione ad altre piattaforme di backend è possibile, ma potrebbe richiedere l'implementazione di servizi di autenticazione, archiviazione e backend equivalenti a seconda del provider.

I passaggi dettagliati di seguito descrivono come migrare un progetto da ZenOpus Cloud a un'istanza Supabase gestita.

### Di cosa sei responsabile

Quando il tuo backend viene eseguito all'esterno di ZenOpus Cloud, sei responsabile delle funzionalità di backend ZenOpus precedentemente gestite, tra cui:

- Disponibilità, scalabilità e backup del database
- Monitoraggio del backend e risposta agli incidenti
- Configurazione e manutenzione della sicurezza a livello di riga
- Configurazione del provider di autenticazione
- Credenziali OAuth, URL di reindirizzamento e rotazione segreta
- Variabili e configurazione dell'ambiente backend
- Scansione di sicurezza per configurazioni errate e segreti esposti
- Stato di conformità della tua infrastruttura backend

ZenOpus non può monitorare o eseguire il debug dell'infrastruttura backend che non controlla. La configurazione OAuth gestita e l'aggiornamento automatico del token sono disponibili solo quando il backend viene eseguito su ZenOpus Cloud.

### Cosa migra e come

| Componenti dell'app | Metodo di migrazione | Note |
| --- | --- | --- |
| Schema del database | Automatico tramite migrazioni SQL | Include tabelle, colonne, indici, criteri RLS, funzioni, trigger |
| Secchi di stoccaggio | Automatico tramite migrazioni SQL | Include politiche di accesso |

## Ospita backend e dati sulla tua infrastruttura (esempio Supabase)

Questa opzione è destinata a rigorosi requisiti di conformità, residenza dei dati o controllo dell'infrastruttura.

Esegui il backend e il database sull'infrastruttura che gestisci. Il percorso self-hosted più diretto è **Supabase self-hosted**, che fornisce i servizi di autenticazione, archiviazione, tempo reale e funzioni edge da cui dipendono le applicazioni ZenOpus.

Il frontend di produzione può rimanere su ZenOpus Cloud o altrove. ZenOpus può ancora essere utilizzato per lo sviluppo oppure lo sviluppo può passare completamente ad altri strumenti.

Eseguire solo un database PostgreSQL autonomo non è sufficiente a meno che non si implementino servizi equivalenti di autenticazione, archiviazione, tempo reale e edge.

### Di cosa sei responsabile

Quando il backend viene eseguito sull'infrastruttura che gestisci, sei responsabile delle funzionalità di backend ZenOpus precedentemente gestite, tra cui:

- Operazioni PostgreSQL, backup e ripristino di emergenza
- Autenticazione, archiviazione e disponibilità e affidabilità del servizio in tempo reale
- Progettazione e applicazione della sicurezza a livello di riga
- Applicazione di patch di sicurezza e gestione degli aggiornamenti di versione
- Distribuzione ed esecuzione delle funzioni Edge
- Ottimizzazione e ridimensionamento delle prestazioni
- Monitoraggio, avvisi e risposta agli incidenti
- Certificazione di conformità della tua infrastruttura

ZenOpus non monitora, gestisce o esegue il debug di alcuna parte dell'infrastruttura self-hosted e le anteprime di produzione non vengono generate per ambienti di produzione self-hosted.

**Quando esegui l'hosting autonomo di Supabase, in genere:**

- Distribuisci Supabase nella tua infrastruttura utilizzando la [guida ufficiale all'hosting autonomo di Supabase con Docker](https://supabase.com/docs/guides/self-hosting/docker).
    <Note>
    Puoi generare queste configurazioni Docker direttamente con l'agente ZenOpus. Consulta [Utilizzo di ZenOpus per generare distribuzioni Docker](#utilizzo-di-zenopus-per-generare-distribuzioni-docker) di seguito.
    </Note>
- Configura PostgreSQL, l'autenticazione, l'archiviazione e i servizi richiesti.
- Applica i file di migrazione dal tuo progetto ZenOpus (`supabase/migrations/`) alla tua istanza self-hosted.
- Aggiorna le variabili di ambiente dell'applicazione in modo che puntino al tuo Supabase self-hosted:
    ```text
    VITE_SUPABASE_URL=https://your-self-hosted-domain.com
    VITE_SUPABASE_PUBLISHABLE_KEY=la tua chiave non-anon
    ```

Per la configurazione dettagliata dell'infrastruttura e le indicazioni operative, segui la [documentazione ufficiale di self-hosting di Supabase](https://supabase.com/docs/guides/self-hosting).

## Utilizzo di ZenOpus per generare distribuzioni Docker

Se preferisci una distribuzione in contenitori, puoi prompt ZenOpus generare configurazioni Docker e Docker Compose per il tuo progetto. L'agente fornirà i dettagli del file generato, le porte del servizio ed eseguirà i comandi nella chat. Di seguito sono riportati i tre modelli supportati.

### Solo frontend

Crea il pacchetto solo per l'app React come sito statico servito da Nginx. Usalo quando vuoi spostare solo il frontend.

Ad esempio:

```text
Dockerizza questo progetto per la distribuzione solo frontend con Nginx
```

### Solo backend (Supabase self-hosted)

Esegui l'intero stack Supabase localmente senza raggruppare il frontend. Usalo quando vuoi sviluppare il frontend separatamente o servirlo da un altro host.

Ad esempio:

```text
Crea una configurazione Docker Compose solo per un backend Supabase self-hosted (nessun contenitore frontend)
```

### Stack completo (frontend + Supabase self-hosted)

Raggruppa il frontend e uno stack Supabase self-hosted in un'unica configurazione Docker Compose. Usalo per distribuzioni completamente autonome.

Ad esempio:

```text

```

### Configurazione manuale richiesta per Supabase self-hosted

L'agente genera segreti segnaposto che devono essere sostituiti prima dell'uso:

1. Genera un segreto JWT: `openssl rand -base64 32`
1. Genera chiavi API utilizzando il tuo segreto JWT. Segui la [documentazione di self-hosting di Supabase per la generazione delle chiavi API](https://supabase.com/docs/guides/self-hosting/docker#generate-api-keys).
1. Sostituisci i seguenti valori in `docker-compose.yml`:
    - __CODICE0__
    - __CODICE0__
    - __CODICE0__
    - __CODICE0__

### Limitazioni

- L'agente non può eseguire o testare build Docker. È necessario verificare localmente.
- L'agente non può generare veri segreti. I segnaposto richiedono la sostituzione manuale.
