Esta guía proporciona **ejemplos prácticos e instrucciones** para implementar y alojar aplicaciones ZenOpus fuera de ZenOpus Cloud.

Se basa en los conceptos explicados en [Opciones de implementación, alojamiento y propiedad con ZenOpus Cloud](/es/tips-tricks/deployment-hosting-ownership) y se centra en las rutas de migración más comunes. Puede adoptar cualquiera de estos escenarios de forma independiente y en cualquier orden.

En esta guía, **alojamiento** se refiere a dónde se ejecuta su aplicación, mientras que **implementación** se refiere a cómo se construye y entrega su código a ese entorno.

Si planea ejecutar todo en ZenOpus Cloud, no necesita esta guía.

## Antes de migrar

Esta guía está destinada a equipos con requisitos específicos, como restricciones de cumplimiento, necesidades de residencia de datos o políticas de infraestructura organizacional, que requieren ejecutar partes de una aplicación fuera de ZenOpus Cloud.

ZenOpus Cloud ya proporciona un entorno integrado de desarrollo y producción, que incluye:

- Alojamiento de producción con dominios personalizados y SSL automático
- Implementaciones automatizadas y gestión del entorno.
- Autenticación administrada con seguridad a nivel de fila
- Almacenamiento de archivos y bases de datos
- Entornos de vista previa automática para cada cambio.
- Agente de IA con depuración en entornos de vista previa y desarrollo
- Acceso de proveedor de IA administrado para funciones de tiempo de ejecución
- Configuración de OAuth integrada y actualización de tokens
- Escaneo de seguridad e infraestructura preparada para el cumplimiento (SOC 2 Tipo 2, ISO 27001)

La mayoría de los equipos nunca necesitan migrar. Puede comenzar en ZenOpus Cloud y mover componentes más tarde si alcanza restricciones reales.

## Requisito previo: conecte su proyecto a GitHub

Todos los escenarios de esta guía requieren [conectar su proyecto a GitHub](/es/integrations/github).

Cuando GitHub está conectado:

- Su proyecto ZenOpus permanece sincronizado continuamente con su repositorio
- Las plataformas externas se implementan directamente desde GitHub
- ZenOpus continúa administrando el desarrollo, las vistas previas y las herramientas de forma independiente.

De aquí en adelante, esta guía asume que su proyecto está conectado a GitHub.

## Aloje la interfaz de producción en una plataforma administrada

Este es el primer paso más común al mudarse fuera de ZenOpus Cloud.

Implementa la interfaz de producción en una plataforma de alojamiento administrada mientras continúa usando ZenOpus para el desarrollo y las vistas previas. Su backend y sus datos pueden permanecer en ZenOpus Cloud o ejecutarse en otro lugar.

### De qué eres responsable

Cuando la interfaz de producción se ejecuta fuera de ZenOpus Cloud, usted es responsable de:

- Canalizaciones de implementación frontend y reversiones
- Variables del entorno de producción.
- Comportamiento de CDN y almacenamiento en caché
- Disponibilidad y tiempo de actividad del frontend
- Registros de producción e historial de implementación
- Vista previa de entornos para ramas o lanzamientos de producción

ZenOpus no puede monitorear ni depurar la infraestructura de producción que no controla.

### Enfoques comunes

- **Plataformas de alojamiento basadas en Git**  
Estas plataformas se conectan directamente a su repositorio de GitHub y se construyen e implementan automáticamente en cada inserción:
    - Netlificar
    - Páginas de Cloudflare
    - Vercel
    - Alojamiento de AWS Amplify
    - Aplicaciones web estáticas de Azure
    - Alojamiento de Google Firebase
- **Almacenamiento de objetos + alojamiento CDN**  
Estas plataformas alojan archivos estáticos detrás de una CDN, pero requieren una canalización de compilación para generar y cargar la salida `dist/`.
    - **AWS:** S3 + NubeFront
    - **Google Cloud:** Almacenamiento en la nube + CDN en la nube
    - **Azure:** Azure Storage (sitio web estático) + Azure CDN o Front Door

### Implementación en una plataforma de alojamiento basada en Git

Este enfoque se aplica a plataformas que se crean e implementan automáticamente desde su repositorio de GitHub.

<Accordion title="Ejemplo: implementación en una plataforma de alojamiento basada en Git">
<Steps>
<Step title="Conecta tu repositorio">
Conecte su repositorio de GitHub a la plataforma de alojamiento.
</Step>
<Step title="Configurar los ajustes de compilación">
La mayoría de las plataformas los detectan automáticamente:

- **Comando de compilación**: `npm run build`
- **Directorio de salida**: `dist`
- **Versión de nodo**: 22
</Step>
<Step title="Configurar variables de entorno">
Los proyectos sin backend no requieren variables de entorno.

Los proyectos que utilizan ZenOpus Cloud necesitan los valores del archivo `.env` de su proyecto:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
VITE_SUPABASE_PROJECT_ID
```

Puede encontrarlos en el archivo `.env` en el editor de código de ZenOpus o en su repositorio sincronizado de GitHub.
</Step>
<Step title="Configurar el enrutamiento SPA">
Si la navegación URL directa devuelve un `404`, configure una reescritura alternativa para que todas las rutas sirvan `/index.html`. El método varía según la plataforma (por ejemplo, archivo `_redirects` en Netlify/Cloudflare, `staticwebapp.config.json` en Azure, reglas de reescritura en Amplify, `vercel.json` en Vercel, `firebase.json` en Firebase).
</Step>
<Step title="Actualizar las URL de redireccionamiento de OAuth">
Si su aplicación utiliza el inicio de sesión de Google u otros proveedores de OAuth, agregue su nuevo dominio de producción a las URL de redireccionamiento permitidas de su proveedor de autenticación.
</Step>
<Step title="Implementar">
Cada envío a GitHub desencadena una nueva implementación de producción.
</Step>
</Steps>

**Resultado:** Un frontend de producción de acceso público servido a través de HTTPS, conectado a su backend ZenOpus Cloud.
</Accordion>

### Implementación en almacenamiento de objetos + CDN con CI/CD

El alojamiento respaldado por CDN requiere una canalización de CI/CD para crear su aplicación y cargar la salida `dist/`. Los pasos de compilación son idénticos en todos los proveedores. La implementación es específica del proveedor; consulte los enlaces a la documentación oficial de cada plataforma.

<Accordion title="Ejemplo: implementación en almacenamiento de objetos + CDN con CI/CD">
Para la autenticación, todos los principales proveedores de nube Support **OpenID Connect (OIDC)** con GitHub Actions, lo que elimina la necesidad de almacenar credenciales de larga duración como secretos. Este es el enfoque recomendado.

- [AWS: Utilice roles de IAM con OIDC](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Google Cloud: Federación de identidades de cargas de trabajo](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)
- [Azure: credenciales de identidad federadas](https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation)

<Steps>
<Step title="Agregar secretos de GitHub">
Agregue los secretos requeridos en la configuración de su repositorio de GitHub (**Configuración → Secretos y variables → Acciones**).

| Secreto | Valor |
| --- | --- |
| VITE_SUPABASE_URL | La URL de su proyecto Supabase (desde el archivo .env de su proyecto) |
| VITE_SUPABASE_PUBLISHABLE_KEY | Su clave publicable de Supabase (del archivo .env de su proyecto) |
</Step>
<Step title="Crear el archivo de flujo de trabajo">
Por ejemplo, cree `.github/workflows/deploy.yml` en su repositorio:

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build application
        run: npm run build
        env:
          VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
          VITE_SUPABASE_PUBLISHABLE_KEY: ${{ secrets.VITE_SUPABASE_PUBLISHABLE_KEY }}

      # Deploy step is provider-specific. See documentation links below.
      # The build output is in the dist/ directory.
```
</Step>
<Step title="Agregar pasos de implementación específicos del proveedor">
Después del paso de compilación, agregue pasos de implementación para su plataforma. Consulte la documentación oficial para conocer las mejores prácticas actuales:

| Plataforma | Documentación |
| --- | --- |
| AWS S3 + NubeFront | Implementación en Amazon S3, GitHub Actions para AWS |
| Almacenamiento en la nube de Google | Alojamiento de un sitio web estático, GitHub Actions para Google Cloud |
| Almacenamiento de blobs de Azure | Alojamiento de sitios web estáticos en Azure Storage, GitHub Actions para Azure |
</Step>
<Step title="Configurar el enrutamiento SPA">
Todo alojamiento respaldado por CDN requiere configuración para devolver `index.html` con un código de estado **200** para rutas que no coinciden con un archivo. Normalmente esto se configura como:

- Una **respuesta de error personalizada que devuelve 200** (CloudFront)
- Una **regla de reescritura de URL en un balanceador de carga** (Cloud CDN)
- Una **regla de reescritura de URL** (Puerta principal)

<Warning>
No confíe en la configuración de la página de error a nivel de almacenamiento, ya que devuelven códigos de estado 404.
</Warning>
</Step>
</Steps>
</Accordion>

## Aloje la interfaz de producción en su propia infraestructura

Utilice este enfoque cuando necesite control total sobre el alojamiento frontend, las redes o el entorno de ejecución. A veces esto se denomina **autohospedaje**.

La interfaz se crea a partir de su repositorio de GitHub y se implementa en la infraestructura que administra. El backend y los datos pueden permanecer en ZenOpus Cloud o ejecutarse en otro lugar.

### De qué eres responsable

Cuando la interfaz de producción se ejecuta en la infraestructura que usted administra, usted es responsable de:

- Automatización de construcción e implementación
- Configuración SSL/TLS
- Configuración de CDN y proxy inverso
- Monitoreo, registro y tiempo de actividad
- Actualizaciones de infraestructura y seguridad.
- Vista previa de entornos para sucursales o versiones

ZenOpus no puede monitorear ni depurar la infraestructura de producción que no controla.

### Enfoques comunes

- **Implementaciones basadas en contenedores**  
Por ejemplo: Docker implementado a través de Kubernetes (EKS, GKE, AKS), ECS, Nomad o plataformas de contenedores internos.
- **Máquinas virtuales detrás de un servidor web**  
Por ejemplo: máquinas virtuales Linux que ejecutan Nginx o Apache, administradas mediante gestión de configuración o herramientas internas.
- **Plataformas PaaS internas**  
Por ejemplo: plataformas de implementación interna de la empresa o soluciones PaaS en la nube privada

### Requisitos de construcción

Los proyectos ZenOpus son aplicaciones estándar de Vite y se construyen como interfaces estáticas.

- **Comando de compilación:**`npm run build`
- **Directorio de salida:**`dist/`
- **Versión de nodo:** 22 recomendados

Las variables de entorno con el prefijo `VITE_` se incrustan en el **tiempo de compilación**, no en el tiempo de ejecución.

Si utiliza ZenOpus Cloud como backend, debe configurar esto antes de ejecutar `npm run build`:

- __CÓDIGO0__
- __CÓDIGO0__

Puede encontrar estos valores en el archivo `.env` de su proyecto.

### Implementación basada en contenedores (Docker)

<Note>
Puede generar estas configuraciones de Docker directamente con el agente ZenOpus. Consulte [Uso de ZenOpus para generar implementaciones de Docker](#usando-zenopus-para-generar-implementaciones-de-docker) a continuación.
</Note>

<AccordionGroup>
<Accordion title="Ejemplo: implementación manual de Docker">
Este es el enfoque más común para la implementación en Kubernetes, ECS, Cloud Run o cualquier plataforma de orquestación de contenedores. También funciona para ejecutar la interfaz como un contenedor independiente en una única máquina virtual.

<Steps>
<Step title="Clona tu repositorio de GitHub">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Crear un archivo Docker">
Los proyectos ZenOpus se pueden contener utilizando una compilación Docker estándar de varias etapas: la primera etapa instala dependencias y ejecuta `npm run build`, y la segunda etapa proporciona la salida estática con nginx.

A continuación se muestra un Dockerfile de ejemplo que puede adaptar:

```docker
# Build stage
FROM node:22-alpine AS builder

WORKDIR /app

# Copy package files and install dependencies
COPY package*.json ./
RUN npm ci

# Copy source code
COPY . .

# Environment variables are embedded at build time
ARG VITE_SUPABASE_URL
ARG VITE_SUPABASE_PUBLISHABLE_KEY
ENV VITE_SUPABASE_URL=$VITE_SUPABASE_URL
ENV VITE_SUPABASE_PUBLISHABLE_KEY=$VITE_SUPABASE_PUBLISHABLE_KEY

# Build the application
RUN npm run build

# Production stage
FROM nginx:alpine

# Copy built assets to nginx
COPY --from=builder /app/dist /usr/share/nginx/html

# Copy nginx configuration for SPA routing
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```
</Step>
<Step title="Crear un archivo de configuración nginx">
Los proyectos ZenOpus usan enrutamiento del lado del cliente (React Router con BrowserRouter), por lo que su servidor web debe devolver `index.html` para todas las rutas. Cree un archivo `nginx.conf`:

```nginx
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    # Cache static assets
    location /assets/ {
    	expires 1y;
    	add_header Cache-Control "public, immutable";
	}

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```
</Step>
<Step title="Construya la imagen del contenedor">
Configure sus variables de entorno como argumentos de compilación:

```bash
docker build \
  --build-arg VITE_SUPABASE_URL=<your-url> \
  --build-arg VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
  -t my-zenopus-app .
```
</Step>
<Step title="Enviar a su registro de contenedores">
Por ejemplo:

```bash
docker tag my-zenopus-app your-registry/my-zenopus-app:latest
docker push your-registry/my-zenopus-app:latest
```
</Step>
<Step title="Implemente utilizando su plataforma de orquestación">
Implemente el contenedor utilizando Kubernetes, ECS, Cloud Run, Nomad o su plataforma interna. No se necesitan variables de entorno de ejecución, ya están integradas en la compilación.
</Step>
</Steps>

**Resultado:** Un contenedor en ejecución que sirve su interfaz a través de HTTP en el puerto 80.

<Warning>
Las variables de entorno se integran en el momento de la compilación. Para cambiarlos, debe reconstruir la imagen del contenedor.
</Warning>
</Accordion>
<Accordion title="Ejemplo: implementación automatizada de Docker (CI/CD)">
Este flujo de trabajo crea una imagen de Docker y la envía a un registro de contenedor para su implementación en Kubernetes, ECS u otras plataformas de orquestación. Esto automatiza los pasos de implementación manual basados ​​en contenedores descritos anteriormente.

<Note>
Probablemente necesitará adaptarlo a sus requisitos de seguridad, canalizaciones existentes y políticas organizativas.
</Note>

Para la autenticación, los principales registros de contenedores Support **OpenID Connect (OIDC)** con GitHub Actions, lo que elimina la necesidad de almacenar credenciales de larga duración como secretos. Este es el enfoque recomendado.

- [AWS ECR: utilizar roles de IAM con OIDC](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Registro de artefactos de Google: Federación de identidades de cargas de trabajo](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)

**Requisitos previos:**

- Un registro de contenedor (por ejemplo, GitHub Container Registry, AWS ECR, Google Artifact Registry, Azure Container Registry, Docker Hub)

<Steps>

| Secreto | Valor |
| --- | --- |

  REGISTRY: ghcr.io/your-org  # Replace with your registry URL

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # TODO: Add registry authentication step here.
      # This workflow will not run until you configure authentication.
      # See "Registry-specific authentication" table below.

      - name: Build and push Docker image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true

            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          build-args: |

### Implementación de VM o servidor estático

<Accordion title="Ejemplo: implementación manual de VM o servidor estático">
Este enfoque es para implementar en máquinas virtuales Linux, servidores básicos o cualquier máquina que ejecute un servidor web como Nginx o Apache.

<Steps>
<Step title="Clona tu repositorio de GitHub">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Instalar dependencias">
```bash
npm ci
```
</Step>
<Step title="Construya la aplicación con variables de entorno.">
Establezca variables de entorno antes de ejecutar la compilación:

```bash
VITE_SUPABASE_URL=<your-url> \
VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
npm run build
```

El resultado de la compilación está en el directorio `dist/`.
</Step>
<Step title="Cargue el resultado de la compilación en su servidor">
Por ejemplo, usando `scp`:

```bash
scp -r dist/* user@your-server:/var/www/html/
```

O usando `rsync`:

```bash
rsync -avz dist/ user@your-server:/var/www/html/
```
</Step>
<Step title="Configura tu servidor web">
Los proyectos ZenOpus usan enrutamiento del lado del cliente (React Router con BrowserRouter), por lo que su servidor web debe devolver `index.html` para todas las rutas.

**Ejemplo de configuración de nginx:**

```nginx
server {
    listen 80;
    server_name your-domain.com;
    root /var/www/html;
    index index.html;

    # Cache static assets
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

**Ejemplo de configuración de Apache (.htaccess):**

```apache
RewriteEngine On
RewriteBase /
RewriteRule ^index\.html$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /index.html [L]
```
</Step>
<Step title="Configurar TLS (recomendado)">
Utilice [Certbot](https://certbot.eff.org/) para agregar un certificado Let's Encrypt gratuito o configurar sus certificados SSL existentes.
</Step>
</Steps>

**Resultado:** Su aplicación se entrega a través de HTTP(S) desde su servidor.

<Warning>
Las variables de entorno están integradas en el resultado de la compilación. No se necesita ninguna configuración de tiempo de ejecución en el servidor. Para cambiar las variables de entorno, reconstruya la aplicación y vuelva a cargar la salida `dist/`.
</Warning>
</Accordion>

## Host de backend y datos en un proveedor administrado (ejemplo de Supabase)

Esta opción generalmente se elige cuando necesita acceso directo a la base de datos, funciones avanzadas de la base de datos o una separación más clara de la propiedad de la infraestructura, sin asumir la responsabilidad operativa total.

Mueve sus servicios de backend y su base de datos a un proveedor de backend administrado. La ruta de migración más directa es a **Supabase administrada**, que se asemeja mucho a la arquitectura de backend predeterminada de ZenOpus Cloud. La interfaz de producción puede ejecutarse en ZenOpus Cloud o en otro lugar.

Después de la migración:

- El frontend de producción debe apuntar al nuevo backend.
- Puede continuar usando el editor ZenOpus y obtener una vista previa de los entornos durante el desarrollo.

Esta guía utiliza **Supabase** como ruta de migración de referencia porque las aplicaciones ZenOpus dependen de servicios compatibles con Supabase (autenticación, almacenamiento, tiempo real, funciones perimetrales y seguridad a nivel de fila).

La migración a otras plataformas backend es posible, pero puede requerir la implementación de servicios de autenticación, almacenamiento y backend equivalentes según su proveedor.

Los pasos detallados a continuación describen cómo migrar un proyecto de ZenOpus Cloud a una instancia administrada de Supabase.

### De qué eres responsable

Cuando su backend se ejecuta fuera de ZenOpus Cloud, usted es responsable de las capacidades de backend que ZenOpus administró previamente, incluidas:

- Disponibilidad, escalamiento y copias de seguridad de la base de datos
- Monitoreo de backend y respuesta a incidentes
- Configuración y mantenimiento de seguridad a nivel de fila
- Configuración del proveedor de autenticación
- Credenciales OAuth, URL de redireccionamiento y rotación de secretos
- Variables de entorno backend y configuración
- Escaneo de seguridad en busca de configuraciones erróneas y secretos expuestos
- Postura de cumplimiento de su infraestructura backend

ZenOpus no puede monitorear ni depurar la infraestructura backend que no controla. La configuración de OAuth administrada y la actualización automática del token solo están disponibles cuando el backend se ejecuta en ZenOpus Cloud.

### Qué migra y cómo

| Componentes de la aplicación | Método de migración | Notas |
| --- | --- | --- |
| Esquema de base de datos | Migraciones automáticas vía SQL | Incluye tablas, columnas, índices, políticas RLS, funciones, activadores |
| Cubos de almacenamiento | Migraciones automáticas vía SQL | Incluye políticas de acceso |
| Proveedores de autenticación | manuales | Reconfigure la autenticación (por ejemplo, Google OAuth, GitHub) en su nuevo entorno de alojamiento |
| Variables y secretos de entorno | manuales | Reconfigure cualquier clave API, token o credencial para servicios externos (por ejemplo, Stripe) en su nuevo entorno de hosting |
| Datos (contenido de la tabla) | manuales | Exportar/importar como CSV |
| Archivos de almacenamiento | manuales | Descargar/cargar manualmente |

## Aloje el backend y los datos en su propia infraestructura (ejemplo de Supabase)

Esta opción está destinada a requisitos estrictos de cumplimiento, residencia de datos o control de infraestructura.

Usted ejecuta el backend y la base de datos en la infraestructura que opera. La ruta autohospedada más directa es **Supabase autohospedada**, que proporciona los servicios de autenticación, almacenamiento, tiempo real y funciones perimetrales de los que dependen las aplicaciones ZenOpus.

La interfaz de producción puede permanecer en ZenOpus Cloud o en otro lugar. ZenOpus aún se puede utilizar para el desarrollo, o el desarrollo puede realizar una transición completa a otras herramientas.

Ejecutar solo una base de datos PostgreSQL independiente no es suficiente a menos que implemente servicios perimetrales, de almacenamiento, en tiempo real y de autenticación equivalentes.

### De qué eres responsable

Cuando el backend se ejecuta en la infraestructura que usted opera, usted es responsable de las capacidades del backend ZenOpus administradas previamente, incluidas:

- Operaciones, copias de seguridad y recuperación ante desastres de PostgreSQL
- Autenticación, almacenamiento y disponibilidad y confiabilidad del servicio en tiempo real
- Diseño y aplicación de la seguridad a nivel de fila
- Aplicar parches de seguridad y gestionar actualizaciones de versiones
- Implementación y ejecución de funciones perimetrales
- Ajuste y escalado del rendimiento
- Monitoreo, alertas y respuesta a incidentes
- Certificación de cumplimiento de su infraestructura

ZenOpus no monitorea, opera ni depura ninguna parte de la infraestructura autohospedada y no se generan vistas previas de producción para entornos de producción autohospedados.

**Cuando autohospedas Supabase, normalmente:**

- Implemente Supabase en su propia infraestructura utilizando la [guía oficial de autohospedaje de Supabase con Docker](https://supabase.com/docs/guides/self-hosting/docker).
    <Note>
    Puede generar estas configuraciones de Docker directamente con el agente ZenOpus. Consulte [Uso de ZenOpus para generar implementaciones de Docker](#usando-zenopus-para-generar-implementaciones-de-docker) a continuación.
    </Note>
- Configure PostgreSQL, autenticación, almacenamiento y servicios requeridos.
- Aplique los archivos de migración de su proyecto ZenOpus (`supabase/migrations/`) a su instancia autohospedada.
- Actualice las variables de entorno de su aplicación para que apunten a su Supabase autohospedada:
    ```text
    VITE_SUPABASE_URL=https://tu-dominio-hospedado-propio.com
    VITE_SUPABASE_PUBLISHABLE_KEY=tu-clave-anónima
    ```

Para obtener una configuración detallada de la infraestructura y orientación operativa, siga la [documentación oficial de autohospedaje de Supabase](https://supabase.com/docs/guides/self-hosting).

## Usando ZenOpus para generar implementaciones de Docker

Si prefiere una implementación en contenedores, puede prompt ZenOpus para generar configuraciones de Docker y Docker Compose para su proyecto. El agente proporcionará detalles del archivo generado, puertos de servicio y ejecutará comandos en el chat. A continuación se muestran los tres patrones admitidos.

### Solo interfaz

Empaquete solo la aplicación React como un sitio estático atendido por Nginx. Utilízalo cuando sólo quieras mover tu interfaz.

Por ejemplo:

```text
Dockerize este proyecto para implementación solo frontend con Nginx
```

### Solo backend (Supabase autohospedado)

Ejecute la pila completa de Supabase localmente sin agrupar la interfaz. Úselo cuando desee desarrollar la interfaz por separado o servirla desde otro host.

Por ejemplo:

```text
Cree una configuración de Docker Compose solo para un backend de Supabase autohospedado (sin contenedor de frontend)
```

### Pila completa (frontend + Supabase autohospedado)

Combine la interfaz y una pila de Supabase autohospedada en una única configuración de Docker Compose. Úselo para implementaciones totalmente autónomas.

Por ejemplo:

```text
Dockerize este proyecto con un backend Supabase autohospedado incluido en Docker Compose
```

### Se requiere configuración manual para Supabase autohospedado

El agente genera secretos de marcador de posición que deben reemplazarse antes de su uso:

1. Generar un secreto JWT: `openssl rand -base64 32`
1. Genere claves API utilizando su secreto JWT. Siga la [documentación de autohospedaje de Supabase para generar claves API](https://supabase.com/docs/guides/self-hosting/docker#generate-api-keys).
1. Reemplace los siguientes valores en `docker-compose.yml`:
    - __CÓDIGO0__
    - __CÓDIGO0__
    - __CÓDIGO0__
    - __CÓDIGO0__

### Limitaciones

- El agente no puede ejecutar ni probar compilaciones de Docker. Debes verificar localmente.
- El agente no puede generar secretos reales. Los marcadores de posición requieren reemplazo manual.
