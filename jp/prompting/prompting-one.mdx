このプレイブックには、ZenOpus で最高の結果を得るために最も効果的なテクニック、戦略、原則がまとめられています。プロンプトを初めて使用する場合でも、スキルを磨く準備ができている場合でも、アイデアを迅速かつ一貫して洗練された UI に変えるのに役立つ実用的なアドバイスと明確な例が見つかります。

## ZenOpus に明確な質問をさせる

ZenOpus からより良い結果を得る最も効果的な方法の 1 つは、コードを記述する前にギャップを埋めることです。必要な機能または変更を指定した後、プロンプトの最後に次のような行を追加します。

> 「私がこの機能に何を望んでいるのか、そして私がそれをどのように思い描いているのかを完全に理解するために、必要な質問があれば何でも聞いてください。」

このアプローチには Plan mode を使用することをお勧めします。

ZenOpus は、焦点を絞った、多くの場合洞察力に富んだフォローアップの質問で応答します。時には、あなたが指定するとは思わなかったような質問も行われます。このプロセスは、要件を事前に明確にし、後で誤解や無駄な作業を防ぐのに役立ちます。

この方法でのプロンプトを試したことがない場合は、試してみてください。おそらく、新しい機能が、やり取りが少なくなり、真の意図に近づいていることに気づくでしょう。

## フェーズ 1: 基礎を築く

### 1. プロンプトを出す前に計画を立てる

ZenOpus を使用する前に、何を構築するかを定義します。このステップを省略することは、何を描くかを決めずに絵を描き始めるようなものです。簡単な計画セッション (紙とペン、音声メモ、ChatGPT など、機能するものなら何でも) を使用して、次の 4 つの質問に答えてください。

- この製品または機能は何ですか?
- 誰のためのものですか？
- なぜ彼らはそれを使うのでしょうか？
- ユーザーが取るべき重要なアクションは何ですか?

あなたは仕様ドキュメントを書いているわけではありません。方向性を決めているのです。思考が一致すればするほど、プロンプトはより鋭くなります。曖昧なアイデアは曖昧なアウトプットを生み出します。明確な思考は明確な結果をもたらします。

**プロンプトの例**

```text
Z 世代のフリーランサーを対象とした予算作成アプリの 1 ページ サイトを構築します。主な CTA は「Start Saving Smarter」である必要があります。大きな文字とパンチの効いた色を使用した、大胆で表現力豊かな美学に焦点を当てます。
```

### 2. ユーザージャーニーを視覚的にマッピングする

デザインとは個々の画面ではなく、画面間で何が起こるかが重要です。 ZenOpus の最良の出力は、明確で論理的なフローから得られます。ユーザーがページにアクセスしてから主要なアクションを完了するまでのプロセスをマッピングします。

トランジションで考える:

- ユーザーが最初に目にするものは何でしょうか?
- 何が信頼を築くのでしょうか？
- 何が彼らに行動する自信を与えているのでしょうか？
- その行動はどこにつながるのでしょうか？

<Info>
シンプルな 3 ステップのスケッチ (ヒーロー → 機能 → CTA) だけでも、プロンプトを 10 倍効果的にすることができます。
</Info>

> 「ブロックを積み上げているのではありません。行動を導いているのです。どのセクションにも存在理由があり、次のセクションにつながる理由があるはずです。」

### 3. 最初に適切なデザインを作成する

あなたの視覚言語は基礎であり、洗練された層ではありません。 ZenOpus は、希望する外観と操作性を早い段階で把握する必要があります。そうしないと、インターフェイスは機能しても完全に間違っていると感じる可能性があります。設計上の問題を後から修正しないでください。事前に決めておきましょう。

方向を選択します。例:

- 穏やかでエレガントな
- 大胆かつ破壊的
- プレミアムで洗練された

次に、バズワード、トーン記述子、UI パターンを使用して、そのスタイルをプロンプトに直接フィードします。 「スターター スタイル プロンプト」を作成し、一貫性を保つためにすべてのセクションでそれを再利用することもできます。

> 「良いデザインへの道を促すのではなく、そこから促すのです。」

**プロンプトの例**

```text
ウェルネスにインスピレーションを得た、落ち着いたデザインを使用します。柔らかなグラデーション、落ち着いたアースカラー、丸い角、たっぷりとしたパッド。フォントは「インター」です。全体的なトーンは優しくて安心感を与えるはずです。
```

## フェーズ 2: システムで考える

### 4. ページではなくコンポーネントごとにプロンプトを表示する

ZenOpus は、ページ全体を一度に構築するのではなく、モジュール化された部分で UI を構築する場合に最適に機能します。ランディング ページ全体を生成するように依頼するのは、レシピをブレンダーにダンプするようなものです。何かを得ることができますが、それは使用できません。

代わりに、レゴ ブロックで建物を作るようにプロンプトに取り組みます。各ブロックには、ヒーロー セクション、機能グリッド、お客様の声のスライダー、価格表など、明確な目的と構造が 1 つある必要があります。 1 つを構築し、レビューして改良し、次に進みます。

コンポーネントごとにプロンプトを表示すると、明確さ、制御性、柔軟性が得られます。何かが正しくない場合は、ページ全体を再プロンプトする代わりに、そのブロックだけを修正できます。これは、複数のページにわたって設計思考を拡張するのにも役立ちます。最初から書き直すことなく、コンポーネントを再利用して適応させることができます。

> 「全ページのプロンプトは煩わしさを感じさせますが、セクションベースのプロンプトは信号を与えます。」

**プロンプトの例**

```text
グラスモーフィズム効果を使用してフローティング メニュー バーを作成します。ホーム、検索、音楽、お気に入り、追加、プロフィール、設定アイコンが含まれます。穏やかなフローティング アニメーションとスムーズなホバー インタラクションを追加します。
```

### 5. 実際のコンテンツを使用したデザイン

ZenOpus は、「lorem ipsum」や「feature 1 / feature 2」などのプレースホルダー コンテンツではうまく機能しません。モデルは構造と意図に反応するようにトレーニングされており、意図を示す最も早い方法は実際の言葉を使用することです。

最終的なコピーが完成していない場合でも、メッセージを反映したコピーを使用してください。瞑想アプリのページを作成している場合は、ユーザーが実際に読むものを書きます。捏造しないでください。

これは、ZenOpus がより適切なレイアウトと間隔を生成するのに役立つだけでなく、より賢明な設計上の決定を下すのにも役立ちます。実際の見出しには 1 行ではなく 2 行が必要になる場合があります。 CTA は、名詞ではなく動詞の方がうまく機能する可能性があります。プレースホルダー テキストはこれらの問題を隠します。実際のコンテンツはそれらを早期に明らかにします。

> 「デザインは制約を好みます。実際のコンテンツは適切な制約を生み出します。」

**プロンプトの例**

```text
「冷静にデザインする」という見出しのヒーローセクション。サブテキスト: 「ZenOpus を使用してストレスを構造に変換します。」 CTA: 「Free の構築を開始します。」十分な垂直方向のスペースを確保した、コピー中心のレイアウトを使用します。
```

### 6. アトミックに話す: ボタン、カード、モーダル

ZenOpus はアトムで考えます。 UI 言語が小さく、より具体的であればあるほど、パフォーマンスは向上します。 「サインアップのあるセクション」を求める代わりに、「電子メールの入力フィールドと丸い CTA ボタンを備えたフォームを追加してください」と言います。これらのアトミックな命令はネイティブ パターンとして解釈されます。

システムのように考えてください。カード、バッジ、トグル、チップ、フォーム フィールド、ドロップダウンなどについて説明します。「送信後に成功トーストが表示されるモーダル」と言える場合は、「ユーザー インターフェイス」とは言わないでください。

このアトミックボキャブラリーを使用すると、段階的に複雑さを重ねることもできます。まずはカードから始めましょう。次に、バッジを追加します。次に、ホバー状態を追加します。各層は最後に自然に構築されます。これにより、プロンプトを書き換えることなく、より詳細な制御とより忠実な出力が得られます。

> 「部品が小さければ小さいほど、反応はよりスマートになります。」

**プロンプトの例**

```text
ユーザーのプロフィール写真、名前、フォロー ボタンを含むカードを作成します。認証済みユーザーのバッジを追加し、バッジの上にマウスを置くとツールチップを表示します。
```

### 7.流行語を使って美学を取り入れる

視覚スタイルは、ZenOpus のプロンプトの最も重要な部分の 1 つですが、最も誤解されています。レイアウトを説明するだけでは十分ではありません。ツールをバイブでガイドする必要があります。バズワードはそれを実現する最も早い方法です。

ZenOpus は、「ミニマル」、「表現力豊か」、「映画的」、「遊び心がある」、「プレミアム」、「開発者中心」などの用語を理解します。これらは単なるものではなく、タイポグラフィ、間隔、影、境界線の半径、カラー パレットに影響を与えるプロンプト表示可能なパラメータです。

これらの単語はプロンプトの早い段階で使用してください。さらに良いのは、すべてのセクションにそれらを含めることです。全体的に一貫したデザインが得られ、恐ろしい「デフォルト UI」の外観が回避されます。

また、流行語を組み合わせてトーンを進化させることもできます。ページは、主人公では「大胆で破壊的」に始まり、価格設定セクションでは「穏やかで安心感を与える」ものになる可能性があります。 1 つのスタイルに固執するのではなく、意図を持ってデザインしています。

> 「デザインは単なる構造ではありません。それはトーンです。流行語がそれを定義します。」

**プロンプトの例**

```text
プレミアムで映画のようなランディング ページのヒーローをデザインします。レイヤードの深さ、半透明のサーフェス、ソフトなモーション ブラー、見出しと背景の間の劇的なコントラストを使用します。
```

## フェーズ 3: 正確に構築する

### 8. レイアウトにプロンプトパターンを使用する

構造化された反復可能なパターンを使用すると、プロンプトを作成するのが簡単になり、より効果的になります。これらをレイアウトのレシピと考えてください。毎回レイアウトを一から説明するのではなく、一貫したリズムで説明します。

適切なレイアウト プロンプトでは、セクションが視覚的な部分に分割され、その順序が明確になり、スタイルの設定方法が定義されます。ほとんどのパターンは「ヘッダー → コンテンツ → アクション」の構造に従いますが、設計内容に応じてそのフローをカスタマイズできます。

個人用プロンプト ライブラリの構築を開始します。たとえば、機能グリッド、お客様の声のスライダー、または価格表の移動先構造を作成します。これらのパターンを再利用してリミックスし、さまざまなページに合わせてわずかな変更を加えます。

これは堅いということではありません。それは効率的であることです。 ZenOpus は、構造化され、範囲が定められ、予測可能なプロンプト言語に最もよく反応します。プロンプト パターンを使用すると、作業が速くなり、画面間での出力の一貫性が高まります。

**プロンプトの例**

```text
見出しを中央に配置し、その後に 3 枚のカードを横に並べた特集セクションを作成します。各カードの上部にはアイコン、見出し、短い説明が含まれています。カードにはソフト シャドウがあり、ホバーするとカードが持ち上がる必要があります。
```

### 9. URL 経由でビジュアルを追加する

レイアウトをリアルにしたいですか? URL を使用して、製品デモ、Midjourney で生成されたクリップ、またはチュートリアル ビデオをドロップインします。 ZenOpus は、明確なプロンプトが表示された場合にビデオの埋め込みをサポートします。

配置 (例: ヒーローの下またはフィーチャー カード内)、スタイル (例: 丸い角、自動再生、ミュート)、およびコンテキスト (なぜそこにあるのか) を求めるプロンプトが表示されます。

**プロンプトの例**

```text
Midjourney の製品デモビデオを埋め込みます。この URL を使用してください: https://cdn.midjourney.com/video/cb84f296-92a0-4a37-a0e3-1c9c95299488/0.mp4。ソフト シャドウ付きの全幅カードの機能セクションの下に配置します。
```

### 10. [編集] ボタンを使用したレイヤーコンテキスト

[編集] ボタンは ZenOpus の最も強力な機能の 1 つですが、それは適切に使用した場合に限られます。何かを変更する必要がある場合、プロンプト全体を書き直すのではなく、編集機能を使用して特定のレイヤーまたは要素に焦点を当てます。これにより、すでに動作しているものを妨げることなく、迅速に反復処理を行うことができます。

編集はデザインをオーバーライドするようなものだと考えてください。それらは代替品ではなく、調整品です。たとえば、CTA ボタンを選択してコピーのみを変更したり、セクション全体のタイポグラフィに影響を与えずにカード レイアウトを微調整したりできます。

これにより、プロジェクトがクリーンでモジュール化された状態に保たれます。急激な膨張を避け、より速く動作し、すでに問題のなかったものを壊すリスクを減らします。編集すると、毎回最初からやり直す必要がないため、より自信を持って実験できるようになります。

編集するときは、言語を正確にしてください。何を変えたいのか、何を変えるべきなのかを正確に言いましょう。 「これを改善する」などの一般的な表現ではなく、「置き換える」、「更新する」、または「調整する」などの指示を使用します。

**プロンプトの例 (編集内)**

```text
CTA ボタンのテキストを「Get Started」に変更し、パディングを水平 24 ピクセルに増やします。既存の背景色とフォントを維持します。
```

## フェーズ 4: 反復して出荷する

### 11. ZenOpus Cloud を念頭に置いて構築する

レイアウトの設計は、製品構築の一部にすぎません。 ZenOpus プロジェクトに見栄えだけでなく、実際にどのように機能するかを考える必要があります。組み込みのクラウドは、その出発点として最適です。

クラウドは、ユーザー認証、データベース、ストレージ、さらにはエッジ機能をすべて最新のフロントエンドとうまく組み合わせる方法で処理します。また、ZenOpus を使用すると、プロンプトから直接、これらの現実世界の動作に一致するように UI を形成できます。

クラウドを念頭に置いて設計する場合は、次のことを予想することから始めます。

- **認証ロジック** — ユーザーがログインしている場合とログアウトしている場合に UI に何を表示する必要がありますか?
- **動的コンテンツ** — ユーザーデータ、投稿、項目、または指標をテーブルから取得していますか?
- **状態** — データが空である場合、読み込み中である場合、または失敗した場合はどうなりますか?

この初期の考え方は、より賢明なセクションを促すのに役立ち、その後の開発をよりスムーズにします。

**プロンプトの例**

```text
ユーザーがクラウド経由でログインしている場合は、右上にプロフィール画像と名前が表示されます。そうでない場合は、「ログイン」ボタンを表示し、認証画面に誘導します。
```

> ヒント: 設計中に動作するバックエンドは必要ありませんが、あたかもすでに存在しているかのように UI を形成することで、レイアウトを将来にわたって保証することができます。

### 12. バージョン管理はあなたの友達です

ZenOpus は変更を自動保存しますが、何も考えずにすぐに行動すべきという意味ではありません。何を変更したのか、なぜ変更したのかを追跡していないと、ただ騒音を立てているだけになり、すぐにコントロールを失ってしまいます。

適切なバージョン管理とは、ファイルに名前を付けることではありません。それは反復して考えることです。意味のある変更は一度に 1 つずつ加えてください。ヒーロー コピーを更新します。フィーチャ グリッドを追加します。レイアウトを調整します。結果を確認してください。次に、前に進みます。

これが、実際の設計システムが意図を持ってレイヤーごとに構築される方法です。

ZenOpus はバックグラウンドでバージョンを管理しますが、独自のシステムを採用してください。

- マイルストーンで考える (例: レイアウトのロック、コンテンツの追加、ロジックの接続)
- プロンプトにメモを使用します (例: 「CTA テキストを変更し、カードの間隔を調整しました」)
- 大きな変更を加える前にプレビューし、危険を冒す場合は複製してください

> 「自動保存は自動的に整理されるという意味ではありません。意図的に構築します。意識して繰り返します。」

**すぐに習慣になる**

```text
大きな変更を加える前に、現在のバージョンを複製してラベルを付けます。低コストで大きなセーフティネット。
```
