このページでは、**ZenOpus アプリ用の安全なコードを作成する方法**と、開発中によくあるセキュリティ上の間違いを回避する方法に焦点を当てています。

ZenOpus のセキュリティ モデル、セキュリティ スキャン、および調査結果の詳細については、[セキュリティの概要](/jp/features/security)、[プロジェクト セキュリティ ビュー](/jp/features/security-view)、および [Workspace セキュリティ センター](/jp/features/security-center) を参照してください。

## ZenOpus アプリの構造

ZenOpus アプリは、責任を明確に分離して構築されています。

- **フロントエンド**はユーザーのブラウザで実行され、常に公開されており、決して信頼してはなりません
- **エッジ機能**はサーバー側で実行され、検証、認証、ビジネス ロジックを処理します。
- **データベース**は、データ アクセスを強制するために行レベル セキュリティ (RLS) を備えた PostgreSQL を使用します

この分離は、ZenOpus のセキュリティ モデルの基礎です。以下のセクションでは、この構造に従い、各層でよくあるセキュリティ上の間違いとその回避方法について説明します。

## フロントエンドセキュリティの基本

すべてのフロントエンド コードはユーザーのブラウザで実行され、検査、変更、またはバイパスできます。フロントエンド コードはセキュリティに関する決定を行うべきではありません。

### フロントエンドコードにシークレットを保存しないでください

フロントエンド コードに保存されているシークレットはユーザーに表示されるため、侵害されていると見なす必要があります。

```typescript
// ❌ Wrong: secrets exposed to anyone
const API_KEY = "sk-1234567890abcdef";
const STRIPE_SECRET = "sk_live_...";
```

**代わりに**: ZenOpus にシークレットを安全に追加するよう依頼してください。シークレットはバックエンド インフラストラクチャに保存され、Edge Functions からのみアクセスされます。

**プロンプトの例**

```text
フロントエンド コードに公開せずに、Stripe 支払い用の API キーを安全に追加します
```

```text
公開されたシークレット、API キー、または機密情報のフロントエンド コードを確認します。
```

### セキュリティのためにフロントエンドの検証に依存しない

フロントエンド検証によりユーザー エクスペリエンスは向上しますが、セキュリティは保証されません。

```typescript
// ❌ Wrong: client-side validation can be bypassed
const validateUser = (userData) => userData.email.includes("@");
```

**代わりに**: チェックをバイパスできないエッジ機能のすべてのデータを検証してサニタイズします。

**クライアント入力は常に信頼できないものとして扱います。** データを処理または保存する前に、**エッジ機能**で形式を検証し、ビジネス ルールを適用し、入力をサニタイズします。

**プロンプトの例**

```text
セキュリティを向上させるために、検証ロジックを React コンポーネントから Edge Function に移動します。
```

```text
バックエンドのエッジ機能に移行する必要があるクライアント側の検証を特定する
```

## エッジ機能によるバックエンドセキュリティ

エッジ機能は、パブリック フロントエンドとプライベート システムの間に安全な境界を形成します。これらは、ユーザーが機密ロジックにアクセスしたり変更したりできない、分離されたサーバー側環境で実行されます。

セキュリティ、データ整合性、または外部サービスに影響を与えるロジックは、フロントエンド コンポーネントではなく、Edge Functions で実行する必要があります。ここでの検証と認可を一元化することで、すべてのリクエストが一貫して評価されるようになり、重要なロジックがブラウザーに到達するのを防ぎます。

エッジ関数は以下を処理する必要があります。

- **認証と認可**  
リクエストを行っているユーザーと、そのユーザーにアクションの実行が許可されているかどうかを確認します。
- **データの検証とサニタイズ**  
すべての入力を信頼できないものとして扱い、ビジネス ルールを一貫して適用します。
- **Business ロジックとワークフロー**  
登録、支払い、承認、状態遷移などの複数段階のプロセスを処理します。
- **外部サービスとの統合**  
サードパーティ API を安全に呼び出し、認証情報がブラウザに漏洩しないようにします。
- **機密データの処理**  
個人データや財務データを安全に処理し、意図せずに公開することを避け、重要なセキュリティ イベントを記録します。

このロジックをサーバー側に維持することで、重要な動作がブラウザーに到達せず、すべてのリクエストが一貫して評価されることが保証されます。

**プロンプトの例**

```text
適切な検証とセキュリティチェックを備えたユーザー登録用のエッジ機能を作成する
```

```text
支払い処理ロジックをフロントエンドから安全なエッジ機能に移動します。
```

```text
タイプとサイズを検証してファイルをアップロードするための Edge 関数を構築する
```

## RLS によるデータベースのセキュリティ

行レベルのセキュリティ (RLS) は、データベース内の個々の行を読み取りまたは変更できるユーザーを制御します。これは保護の最終層として機能し、フロントエンドまたはバックエンドのロジックに障害が発生した場合でもアクセス ルールを適用します。

ZenOpus は基本的な RLS ポリシーを自動的に設定しますが、開発の早い段階でそれらを確認して調整する必要があります。 RLS は、実際のデータが存在する前に変更するのがはるかに簡単で、ポリシーは複雑なビジネス ロジックではなくアクセス制御に重点を置いている場合に最も効果的です。

### 一般的な RLS パターン

ほとんどの ZenOpus アプリは、次のような少数のアクセス パターンに当てはまります。

- **個人データ**: ユーザーは自分のデータのみにアクセスできます
- **チームベースのアクセス**: チームメンバーはチームデータへのアクセスを共有します
- **所有権のある公開コンテンツ**: 誰でも読み取り可能、所有者のみが変更可能
- **組織ベースのアクセス**: ユーザーは組織内のデータにアクセスします

可能な限り、これらのパターンに基づいてテーブルとポリシーを設計してください。

### 公開前の RLS チェック

ライブを開始する前に、次のことを確認してください。

- RLS はすべての機密テーブルで有効になっています
- ユーザーは他のユーザーの個人データにアクセスできません
- 共有データと公開データは意図したとおりに動作します
- 新しいテーブルにはポリシーがありません

[プロジェクト セキュリティ ビュー](/jp/features/security-view) で RLS 関連の調査結果を確認します。

**プロンプトの例**

```text
RLS ポリシーを確認して、ユーザーが自分のデータのみにアクセスでき、共有データが適切に保護されていることを確認します。
```

```text
RLS ポリシーを設定テーブルに追加して、ユーザーが自分の設定のみにアクセスできるようにします
```

```text
チームとプロジェクトに RLS を設定して、チーム メンバーに自分のチームのプロジェクトのみが表示されるようにする
```

## 認証セキュリティの基本

すべての認証の決定は、ブラウザーではなくサーバー側で行われる必要があります。クライアント側の認証チェックはユーザーによって検査、変更、またはバイパスされる可能性があるため、アクセス制御に依存しないでください。

認証は、セッションとトークンを安全かつ一貫して検証できる **エッジ機能** で強制する必要があります。

```typescript
// ❌ Wrong: client-side authentication checks
const isAuthenticated = localStorage.getItem("authToken") !== null;
if (isAuthenticated) showAdminPanel();
```

```typescript
// ✅ Correct: server-side validation in an Edge Function
const { user } = await supabase.auth.getUser();
if (!user) {
  return new Response("Unauthorized", { status: 401 });
}
```

**認証のベスト プラクティス**

- Edge Functions での認証とセッション トークンを検証する
- サーバー側の役割と権限を確認する
- 組み込みのセッション管理を使用して安全なトークンを処理する
- セッションの有効期限が切れたときにユーザーをログインにリダイレクトしますが、常に最初にサーバーで検証します
- フロントエンド認証状態は UI を制御するためにのみ使用し、アクセスを強制することはありません

## Workspace セキュリティ

公開してはいけないアプリケーションの場合:

- [プロジェクトの可視性](/jp/features/project-visibility) を **Workspace** に設定します
- アプリが一般に[公開](/jp/features/publish)されていないことを確認してください
- 内部ツールであっても認証を必要とする
- 誰がアクセス権を持っているかを定期的に監査する

Workspace 全体の監視には、[セキュリティ センター](/jp/features/security-center) を使用します。

## 公開前のセキュリティチェックリスト

アプリを公開する前に、次のことを確認してください。

- フロントエンドコードにシークレットは存在しません
- すべての検証と重要なロジックは Edge Function で実行されます
- 行レベルのセキュリティ (RLS) ポリシーが構成およびテストされている
- 認証はサーバー側で強制されます
- 内部アプリにはworkspaceの可視性があります
- [セキュリティ ビュー](/jp/features/security-view) のすべての重要な検出結果に対処します
- 外部 API 呼び出しはサーバー側で行われます

<Tip>
**セキュリティは継続的なプロセスです**。

機能の追加、認証の変更、データ アクセスの変更、または新しい外部統合の導入を行うたびに、これらのプラクティスを確認してください。

エッジ機能、行レベルのセキュリティ ポリシー、認証フローを定期的に確認すると、時間の経過とともにリグレッションを防ぐことができます。
</Tip>
