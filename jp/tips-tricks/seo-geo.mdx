## ZenOpus における SEO と GEO の仕組み

ZenOpus を使用すると、多くの柔軟性が得られます。**検索エンジン最適化 (SEO)** と **生成エンジン最適化 (GEO)** は、ページ、コンテンツ、機能を作成するときに ZenOpus エージェントが生成するコードから生まれます。

エージェントはプロンプトに基づいてタイトル、説明、タグなどの要素を追加する場合がありますが、これは体系的ではなく、**完全または正確な実装を保証するものではありません。**

このため、**SEO と GEO をコードのように扱う**、つまり意図的にレビュー、テスト、改良する必要があります。このガイドでは、何を尋ねるべきか、そしてそれを確認する方法を正確に説明します。

### 内蔵速度ツール

ZenOpus には、**Google Lighthouse を利用した組み込みの Speed ツール**も含まれています。公開されたサイトのパフォーマンス、アクセシビリティ、ベスト プラクティス、SEO チェックの簡単なスナップショットが提供されます。

![ZenOpus スピード ツール](/images/zenopus/tips-tricks/seo-geo/speed-tool.png)

これはサイトの健全性を最初に確認するのに最適ですが、Lighthouse ではクロール可能性、構造化データ、メタデータの精度、ソーシャル プレビューなどの重要な要素は検証されないため、このガイドはそれらのギャップを埋めます。

### カスタムドメイン

**カスタム ドメイン**の使用は、SEO と GEO にとって最も重要な手順の 1 つです。ブランド ドメインは、検索エンジンがサイトを理解し、バックリンクの価値を長期にわたって維持し、すべてのトラフィックが単一の正規 URL に確実に統合されるようにします。

ZenOpus を使用すると、次のことが可能になります。

- 独自のドメインまたはサブドメインを接続する
- **プライマリ ドメイン**を 1 つ選択します (他のすべてのドメインはそこにリダイレクトされます)

カスタム ドメインを設定したら、[**Google Search Console で検証**](#googleサーチコンソール（gsc）の設定)すると、Google が正しいドメインをクロールしてインデックスに登録できるようになります。セットアップ手順の詳細については、[**カスタム ドメイン**](/jp/features/custom-domain) を参照してください。

## ZenOpus のアーキテクチャを理解する

ZenOpus は、React と Vite を使用して最新の Web アプリケーションを構築します。これは、アプリケーションが **クライアントサイド レンダリング (CSR)** **シングルページ アプリケーション (SPA)** であることを意味します。ページごとに異なる HTML ファイルを提供する代わりに、ブラウザーは小さな初期 HTML シェルをロードし、JavaScript を使用してルーティングとレンダリングを処理します。

WordPress や Webflow などの **従来のプラットフォーム** は、URL ごとに完全に組み立てられた HTML ページをブラウザーに送信します。

**ZenOpus** は、デバイスが各ビューを構築するために使用するコードと命令を送信します。すべての「ページ」は実際には、同じアプリケーションの異なる状態にすぎません。

訪問者にとって、これは高速なナビゲーション、瞬時の遷移、シームレスなエクスペリエンスを意味します。

### これがSEOとGEOにとって何を意味するか

**検索エンジン、ソーシャル プラットフォーム、AI クローラー**の場合、アーキテクチャによってコンテンツの発見方法が変わります。

**Google および従来の検索エンジン:**

Google は CSR サイトをインデックスに登録できますが、これは 2 つの段階で行われます。

1. **最初の HTML をクロール**し、ページ構造を収集します
1. **後で戻って JavaScript (JS) をレンダリングし**、完全なコンテンツをキャプチャします

この 2 段階のプロセスは確実に機能しますが、次の点に注意してください。

- インデックス作成には **少し長くなります** (数時間ではなく数日)
- 新しいページが後で結果に表示される場合があります
- **ランキングには影響しません**。インデックス作成速度のみに影響します。

**ソーシャル メディア プラットフォームと AI システム:**

- **Facebook、X/Twitter、LinkedIn** などのプラットフォームはコンテンツのレンダリングを待機しないため、最初の HTML ページ構造のみが表示されます。
- すべてのページが同じ基本タイトルとプレビュー画像を共有している場合、ソーシャル リンクには一般的な情報または不正確な情報が表示される可能性があります。
- 多くの **AI システム**は動的にレンダリングされたコンテンツを確実に認識できないため、ページを見逃したり、部分的なコンテンツしか表示されない可能性があります。

**最新の検索エンジンは CSR を適切に処理します。** CSR はインデックス作成やランキングをブロックせず、実装とコンテンツの品質が最も重要です。

このガイドのベスト プラクティスに従って、**ZenOpus サイトが検索エンジン、ソーシャル プラットフォーム、AI システムに対して最適化され、効果的に競争できるようにします**。

## 基礎: サイトをクロール可能にする

これらの手順により、検索エンジンが ZenOpus アプリのすべての重要なページにアクセスし、検出し、インデックスを作成できるようになります。

### XMLサイトマップ

サイトマップは、検索エンジンがすべてのページを検出するのに役立ちます。これは、クローラーがすべてのルートを簡単に見つけることができない CSR サイトでは特に重要です。 ZenOpus エージェントは、要求に応じてすべてのパブリック ルートを含む `sitemap.xml` を生成できます。

<Accordion title="sitemap.xml を実装して検証する方法">
**プロンプトの例**

> /sitemap.xml に XML サイトマップを作成し、すべてのパブリック ルートをリストします。 lastmod の日付と優先順位を含めます: ホームページ 1.0、メイン ページ 0.8、ブログ投稿 0.6。

**確認してください**

- プロジェクトを公開してサイトマップ ファイルを公開します
- `https://yourdomain.com/sitemap.xml` でアクセスできることを確認してください (XML が返されるはずです)
- 含まれているすべての主要なルートを確認します (ホーム、メイン ページ、ブログ、製品)
- コンテンツが変更されるたびに `lastmod` を更新します
- `priority` を使用して重要性を示します（0.0～1.0）
- Google Search ConsoleとBingウェブマスターツールに送信する
- ページの追加/削除時に再生成して再送信します (自動ではありません)。たとえば、次のようにプロンプトを出します。
    > sitemap.xml を更新して /new-page を含め、/old-page を削除してから、Google Search Console に再送信します。

**sitemap.xml の例**

```text
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://yourdomain.com/</loc>
    <lastmod>2025-01-15</lastmod>
    <priority>1.0</priority>
  </url>
  <url>
    <loc>https://yourdomain.com/about</loc>
    <lastmod>2025-01-15</lastmod>
    <priority>0.8</priority>
  </url>
</urlset>
```
</Accordion>

### ロボット.txt

`robots.txt` は、サイトのどの領域をクロールできるかを検索エンジンに伝えます。 Google がページを正しくレンダリングするために必要な主要なリソースをブロックしないようにすることが重要です。 ZenOpus エージェントは、要求に応じて `robots.txt` ファイルを作成できます。

<Accordion title="robots.txt の実装と検証方法">
**プロンプトの例**

> すべてのクローラーを許可し、サイトマップ: [https://yourdomain.com/sitemap.xml](https://yourdomain.com/sitemap.xml) を参照し、AI 可視化のために GPTBot と PerplexityBot を許可する robots.txt を /public/robots.txt に作成します。

**確認してください**

- `https://yourdomain.com/robots.txt` でアクセスできることを確認してください。
- CSS、JavaScript、`/assets/` フォルダーを決してブロックしないでください
- サイトマップを指す `Sitemap:` 行が含まれています
- 戦略に基づいて AI ボット アクセス (GPTBot、PerplexityBot) を制御します

**robots.txt の例**

```text
ユーザーエージェント: *
許可: /
サイトマップ: https://yourdomain.com/sitemap.xml

ユーザーエージェント: GPTBot
許可: /
```
</Accordion>

### 正規タグ

Canonical タグは、どの URL がページの「公式」バージョンであるかを検索エンジンに知らせ、重複コンテンツの問題を防ぐのに役立ちます。 ZenOpus エージェントは、要求に応じて正規タグを生成できます。

<Accordion title="正規タグを実装および検証する方法">
**プロンプトの例**

> 独自の URL を指すすべてのページに正規タグを追加します。末尾にスラッシュを付けない [https://yourdomain.com](https://yourdomain.com) 形式を使用します。

**確認してください**

- 各ページをご確認ください。これをブラウザのコンソールに貼り付けます。
    ```text
    console.log('Canonical:', document.querySelector('link[rel="canonical"]')?.href);
    ```
- ページごとに **正確に 1 つ** の正規タグが存在することを確認します。
- 優先ドメインと URL バリアント (HTTPS、末尾のスラッシュ) が一致していることを確認してください。
- 意図的でない限り (フィルタリングされた製品ビューなど)、異なるページが同じ正規を参照することは避けてください。
</Accordion>

### クリーンな URL とルーティング

検索エンジンは URL に依存してページの内容を理解するため、クリーンで説明的な URL はクロール可能性とランキングの両方を向上させるのに役立ちます。 ZenOpus エージェントは通常、React Router を通じてクリーンなルートを自動的に生成します。

<Accordion title="クリーンな URL を実装して検証する方法">
**プロンプトの例**

> すべてのルートを確認し、URL がクリーンでわかりやすいものであることを確認してください。単語の区切りにはハイフンを使用し、クエリ パラメーターを避けてください。

**確認してください**

- 各ルートに移動し、ブラウザ コンソールでエラーを確認します。
- URL は安定していて、説明的であり、ランダムな ID は含まれていない必要があります。
- サイト全体で一貫した形式 (末尾のスラッシュの有無、`www` の有無)
- すべてのルートは JavaScript エラーなしでロードされます
</Accordion>

### 内部リンク

内部リンクはページを接続するため、ユーザーと検索エンジンの両方がサイトを簡単に移動できるようになります。検索エンジンは、これらのリンクを利用してコンテンツを検出し、トピックの関係を理解し​​、サイト全体にページの権限を配布します。

ZenOpus エージェントは、要求に応じてナビゲーション/フッター リンクを作成し、コンテキスト リンクを提案または作成できます。

<Accordion title="内部リンクの実装と検証方法">
**プロンプトの例**

> 関連ページに 3 ～ 5 個のコンテキスト内部リンクを追加します。 「ここをクリック」の代わりに説明的なアンカー テキストを使用します。

**確認してください**

- リンクがクロール可能であることを確認します。この数値は、ナビゲーションとコンテンツに基づいて予想される数値とほぼ一致するはずです。
    ```text
    console.log('リンク:', document.querySelectorAll('a[href]').length);
    ```
- `onClick` ハンドラーではなく、実際の `<a href>` タグを使用する
- 重要なページには、それらを指す複数の内部リンクがあります
- 最も重要なページへのリンクをフッターに含めて、サイト全体に表示されるようにします。
- アンカー テキストはキーワードを含む説明的なものです (「ここをクリック」ではありません)
    - 適切なアンカー テキスト: `organic dog food benefits`
    - 不正なアンカー テキスト: `click here`、`read more`
- ホームページから 3 回のクリックでアクセスできる詳細なページ
</Accordion>

## オンページ SEO: 検索エンジンがコンテンツを理解できるようにする

これらの実践により、各ページの内容が明確になり、検索エンジンがコンテンツを解釈して正確にランク付けしやすくなります。

### ページタイトル

ページ タイトルは、ページ上の重要なランキング要素です。これらは検索結果やブラウザのタブに表示され、クリックスルー率に大きな影響を与えます。 ZenOpus エージェントはページの目的に基づいてタイトル タグを生成できます。

<Accordion title="ページタイトルを実装および検証する方法">
**プロンプトの例**

> すべてのルートのページ タイトルを更新します。各ページの目的に合った明確で説明的なタイトルを使用し、60 文字以内に収めてください。

**確認してください**

- 各ページのタイトルの内容と長さを確認してください。これをブラウザのコンソールに貼り付けます。
    ```text
    console.log('タイトル:', document.title, '| 長さ:', document.title.length);
    ```
- 各ルートには固有の説明的なタイトルが付いています (どこでも汎用的な `Home` ではありません)。
- 切り捨てを避けるために 60 文字未満にしてください
- 主なキーワードとブランド名を含む
</Accordion>

### メタディスクリプション

メタディスクリプションは、検索結果のページタイトルのすぐ下に表示されます。これらは直接的なランキング要因ではありませんが、クリックスルー率に強い影響を与えます。 ZenOpus エージェントは、要求に応じてメタ説明を追加できます。

<Accordion title="メタディスクリプションを実装および検証する方法">
**プロンプトの例**

> すべてのキーページに一意のメタディスクリプションを記述します。それぞれを明確で役立つものにし、140 ～ 160 文字以内にしてください。

**確認してください**

- 各ページの説明と長さを確認してください。これをブラウザのコンソールに貼り付けます。
    ```text
    console.log('Desc:', document.querySelector('meta[name="description"]')?.content, '| Length:', document.querySelector('meta[name="description"]')?.content.length);
    ```
- 各ページには固有の説明があります (**140 ～ 160 文字**)
    - 悪い: `Welcome to our website` (一般的、繰り返し)
    - 良い: `Premium organic dog food delivered fresh to your door. Free shipping over $50.` (158 文字)
- メリット、差別化、ソフトな行動喚起を強調します
- ページ間で重複しない
</Accordion>

### 見出し構造（H1→H3）

明確な見出し構造 (H1 から H3) は、検索エンジンがコンテンツがどのように構成されているか、どのセクションが最も重要であるかを理解するのに役立ちます。 ZenOpus エージェントは、要求に応じて適切な H1 タグと論理階層を生成できます。

<Accordion title="見出し構造を実装および検証する方法">
**プロンプトの例**

> 各ページの見出し構造を確認してください。上部に 1 つの H1 があり、H2 が主要なセクション、H3 がネストされたコンテンツである必要があります。見出しレベルをスキップしないでください。

**確認してください**

- H1 タグが 1 つだけであることを確認します。これをブラウザのコンソールに貼り付けます。1 が返されるはずです。
    ```text
    console.log('H1s:', document.querySelectorAll('h1').length);
    ```
- 見出しの内容を表示します。これをブラウザのコンソールに貼り付けます。
    ```text
    console.log('H1:', document.querySelector('h1')?.textContent);
    ```
- H1 には主なキーワードとページの目的が含まれます
- 見出しは論理的に流れます (H1 → H2 → H3、レベルをスキップしない)
- 見出しはスタイル設定のみに使用されるわけではありません
</Accordion>

### セマンティックHTML

セマンティック HTML では、意味のあるタグを使用してコンテンツを説明し、検索エンジンとスクリーン リーダーの両方がページの構造を理解できるようにします。 ZenOpus エージェントは通常、`<main>`、`<nav>`、`<section>` などのセマンティック要素を追加します。

<Accordion title="セマンティック HTML を実装して検証する方法">
**プロンプトの例**

> HTML 構造を確認し、必要に応じてセマンティック タグを使用します。メインコンテンツを `<main>` に、ナビゲーションを `<nav>` に、セクションを `<section>` に、フッター コンテンツを `<footer>` に配置します。

**確認してください**

- これをブラウザ コンソールに貼り付けます。該当する場合は `true` が返されます。

```text
console.log('Has <main>:', !!document.querySelector('main'));
console.log('Has <nav>:', !!document.querySelector('nav'));
console.log('Has <footer>:', !!document.querySelector('footer'));
console.log('Has <section>:', !!document.querySelector('section'));
```

- 各ページで次のことを確認してください。
    - 単一の `<main>` が主要なコンテンツをラップします
    - `<nav>` 内のナビゲーション
    - `<footer>` のフッター コンテンツ
    - `<section>` および `<article>` タグの関連コンテンツ
</Accordion>

### 画像の最適化

画像を最適化すると、アクセシビリティが向上し、話題の関連性が強化され、画像検索結果にコンテンツが表示されやすくなります。効率的な形式と適切なサイズの画像を使用すると、ページ速度も向上します。これは重要なランキング要素です。 ZenOpus エージェントは、必要に応じて `alt` テキストを追加し、遅延読み込みを有効にすることができます。

<Accordion title="画像の最適化を実装および検証する方法">
**プロンプトの例**

> すべての画像を確認し、明確で説明的な代替テキストを追加します。各画像に幅と高さの属性があり、高速に読み込めるように適切に圧縮されていることを確認してください。

**確認してください**

- すべての画像に代替テキストがあることを確認します。 これをブラウザのコンソールに貼り付けます - 0 が返されるはずです:
    ```text
    console.log('Missing alt:', document.querySelectorAll('img:not([alt])').length);
    ```
- `alt` テキストの品質を評価します - 関連するキーワードを使用して説明的である必要があります
    - 悪い: `alt="image"`、`alt="photo", alt=""`
    - 良い: `alt="Golden retriever eating organic grain-free dog food from a bowl"`
- 適切なファイル形式を使用し、ファイル サイズを小さく保ちます。
    - 写真やイラストの WebP
    - ロゴ、アイコン、シンプルなグラフィック用の SVG
    - 画像は 200KB 未満に圧縮
- 幅と高さの属性が存在します (レイアウトのずれを防止します)
</Accordion>

## オフページ SEO: バックリンクで権威を築く

検索エンジンは、コンテンツの信頼性と信頼性を判断するために、他の評判の良いサイトからのリンクに大きく依存しています。競争力のあるキーワードの場合、ページ上の最適化よりも強力なバックリンクが重要になる場合があります。

バックリンクは、ZenOpus サイトに役立ちます。

- 狙ったキーワードで上位表示
- インデックスをより速く作成する
- 話題の権威を築く
- 大規模なサイトや古いサイトと競合する

<Accordion title="高品質のバックリンクを構築する方法">
1. **リンク価値のあるコンテンツを作成する**

- 詳細なガイド、チュートリアル、リサーチ
- データ、統計、または独自の洞察
- インタラクティブなツール、計算機、ROI 推定ツール

## リッチリザルト: 検索で目立つようになりました

これらの機能強化により、ページがリッチリザルトの対象となり、ソーシャル プラットフォームや SERP で高品質のプレビューを生成できるようになります。

### 構造化データ（JSON-LDスキーマ）

JSON-LD スキーマを通じて実装された構造化データは、検索エンジンにコンテンツに関する追加のコンテキストを提供し、星による評価、製品の詳細、FAQ ドロップダウンなどのリッチな結果を可能にします。 ZenOpus エージェントは、要求に応じてスキーマ マークアップを生成できます。

頻繁に使用されるスキーマ タイプには次のものがあります。

- `Product`: 価格、在庫状況、レビュー、評価
- `Article`: 著者、発行日、注目の画像
- `FAQPage`: 質問と回答
- `LocalBusiness`: 住所、電話番号、営業時間などのビジネスの詳細

<Accordion title="構造化データを実装および検証する方法">
**プロンプトの例**

> `Product` スキーマを、名前、説明、価格、在庫状況、画像、ブランド、評価の詳細を含む製品ページに追加します。

**確認してください**

- 主要なページ (ホーム、製品、ブログ投稿、FAQ など) の存在を確認します。 これをブラウザ コンソールに貼り付けます。意図的に複数のスキーマ タイプを追加しない限り、1 が返されます。

```text
console.log('スキーマ:', document.querySelectorAll('script[type="application/ld+json"]').length);
```

- [Google リッチリザルト テスト](https://search.google.com/test/rich-results) で検証する
- エラーまたは必須フィールドが欠落している場合は修正してください
- スキーマ データと表示されているページ コンテンツの同期を維持する
</Accordion>

### ソーシャルメディアの最適化: Open Graph と Twitter Cards

Open Graph (OG) タグ、Twitter カード、および OG 画像は、ソーシャル メディアで共有されたときにページがどのように表示されるかを制御します。高品質のプレビュー画像により、共有パフォーマンスが大幅に向上し、ブランドへの信頼を構築できます。

ZenOpus エージェントにはデフォルトで基本的な OG および Twitter メタデータが含まれていますが、**ページごとに画像、タイトル、説明をカスタマイズする必要があります。**

ほとんどのソーシャル プラットフォーム (Facebook、Twitter/X、LinkedIn) は JavaScript を実行しません。つまり、最初の HTML で Open Graph と Twitter Card メタデータを明示的に構成しない限り、サイトでは汎用または不完全なリンク プレビューが表示される可能性があります。

<Accordion title="Open Graph、Twitter Cards、OG イメージの実装と検証方法">
**プロンプトの例**

> 独自の Open Graph および Twitter メタデータをすべての重要なページに追加します。ルートごとにカスタムのタイトル、説明、URL、ソーシャル イメージを設定します。ページごとに 1200×630 JPG 画像を 1 つ使用し、それに応じて `og:image` と `twitter:image` を更新します。

**確認してください**

- URL を共有する前に、URL がどのように表示されるかを確認してください。
    - [Facebook 共有デバッガー](https://developers.facebook.com/tools/debug/)
    - [X (Twitter) カード検証ツール](https://cards-dev.twitter.com/validator)
    - [LinkedIn ポストインスペクター](https://www.linkedin.com/post-inspector/)
- 重要な各ページには、HTML 内に固有の `og:title`、`og:description`、`og:image`、`og:url` があります。
- タグには `twitter:card` と `twitter:image` が含まれます
- プレビューには正しいタイトル、説明、画像が表示されます
- 画像は1200×630px、1MB以下
- ブランドと視覚的に一貫性がある
- 各ページに単一の汎用画像を使用することは避けてください。

**Open Graph と Twitter Card メタデータの例**

```text
<meta property="og:title" content="Organic Dog Food Delivery | PetFood Co" />
<meta property="og:description" content="Premium grain-free dog food delivered fresh to your door." />
<meta property="og:image" content="https://yourdomain.com/og-image.jpg" />
<meta property="og:url" content="https://yourdomain.com/products" />
<meta name="twitter:card" content="summary_large_image" />
<meta name="twitter:image" content="https://yourdomain.com/og-image.jpg" />
```
</Accordion>

## パフォーマンスとモバイル: 読み込み速度とユーザー エクスペリエンスの向上

これらの最適化により、現実世界の速度、モバイルの使いやすさ、およびコンテンツを効果的にレンダリングする Google の能力が向上します。

### コア Web バイタルとパフォーマンス

ページ速度はランキング要素として確認されています。高速なサイトほど上位にランクされ、コンバージョン率も高くなります。 ZenOpus エージェントは、Support 高速レンダリングに対する遅延読み込みやコードの縮小などの最新の最適化パターンを使用します。

- ZenOpus の **組み込み Speed ツール** (Google Lighthouse を利用) を使用して、サイトのパフォーマンス、アクセシビリティ、ベスト プラクティス、SEO チェックを分析します。
- **Google Search Console を使用して Core Web Vitals** を検証します。これは、ページの読み込みパフォーマンス、インタラクティブ性、視覚的な安定性に関する実際のユーザー エクスペリエンスを測定する一連の指標です。

<Accordion title="ウェブの重要な要素とパフォーマンスを最適化して検証する方法">
**プロンプトの例**

> 画像の圧縮、幅/高さ属性の追加、必須ではないスクリプトの延期、主要なアセットのプリロードにより、ホームページのパフォーマンスを向上させます。 Lighthouse でパフォーマンス スコア 90 以上を目標にします。

**確認してください**

- ZenOpus で **Speed** ツールを実行する（Google Lighthouse 監査）
- 目標スコア:
    - パフォーマンス: 90+
    - アクセシビリティ: 90+
    - ベストプラクティス: 90+
    - SEO: 100
- パフォーマンススコアが低いページを修正する
- GSC で、**Core Web Vitals ターゲット:** を確認して修正します。

|メトリック |ターゲット |何を測定するのか |修正方法 |
| --- | --- | --- | --- |
| LCP (最大コンテンツフルペイント) | 2.5秒未満 |読み込み性能、メインコンテンツが表示されるまでの時間 |ヒーロー画像の圧縮、WebP の使用、重要なリソースのプリロード、サーバーの応答時間の短縮 |
| INP (次のペイントへのインタラクション) | 200ミリ秒未満 | Web ページがユーザーの操作にどれくらい早く応答するか | JavaScript の実行時間を短縮し、重要ではないスクリプトを延期し、長いタスクを分割します。
| CLS (累積レイアウト シフト) | 0.1未満 |ロード中の視覚的な安定性 |すべての画像に幅と高さを追加し、動的コンテンツ用のスペースを確保し、読み込み後に折りたたむ部分の上にコンテンツを挿入しないようにします。
</Accordion>

### モバイルの最適化

検索の 60% 以上がモバイルで行われています。 Google はモバイル ファースト インデックスを使用しています。つまり、モバイル バージョンがランキングを決定します。 ZenOpus エージェントは、`viewport` タグを使用して応答性の高いレイアウトを生成します。 `viewport` タグは、モバイル対応のスケーリングに必要です。

<Accordion title="モバイルのパフォーマンスを最適化して検証する方法">
**プロンプトの例**

> モバイル エクスペリエンスを確認します。タップ ターゲット サイズを最小 48x48 ピクセルに増やし、インタラクティブな要素間にスペースを追加し、テキストが最小 16 ピクセルであることを確認し、小さな画面での水平スクロールを修正します。

**確認してください**

- **Chrome DevTools** と **実際のデバイス** を使用して、いくつかのデバイス サイズ (iPhone、Android、タブレット サイズなど) で主要なページをテストします。
- すべての主要なフローがモバイル上でスムーズに動作することを確認する
- どのページでも水平スクロールはありません
- ズームせずにテキストを読むことができます (最小 16 ピクセルのフォント サイズ)
- ボタンとリンクは少なくとも 48×48 ピクセルで、間隔があり、簡単にタップできます
- フォームはタッチ デバイスで適切に動作します (大きな入力、明確なラベル)
- ナビゲーションにアクセス可能 (ハンバーガー メニュー機能)
- 画像とビデオが画面幅内に収まる
- **`viewport` **タグが存在することを確認してください。** これをブラウザのコンソールに貼り付けると、次のように表示されます。
    ```text
    document.querySelector('meta[name="viewport"]')?.outerHTML
    ```
    ```text
    <meta name="viewport" content="width=device-width, initial-scale=1">
    ```
</Accordion>

## GEO: コンテンツを AI/LLM フレンドリーにする

Generative Engine Optimization (GEO) は、ChatGPT、Perplexity、Gemini、Claude などの AI システムがコンテンツを検出して引用するのに役立ちます。生成エンジンは、検索エンジンとは異なる方法で情報を抽出します。多くは JavaScript を実行しないため、静的な HTML とスキーマに大きく依存します。

GEO のベスト プラクティスのほとんどは、優れた SEO の延長です。前のセクションとここでの追加事項に従えば、AI の可視性を適切に得ることができます。

### AI のセマンティック HTML とスキーマ

AI クローラーは JavaScript を実行せずに HTML を直接読み取ります。明確な構造とスキーマは、引用のための事実を抽出するのに役立ちます。 ZenOpus エージェントはセマンティック構造を提供し、要求に応じてスキーマを追加できます。

<Accordion title="AI のセマンティック HTML とスキーマを実装および検証する方法">
**プロンプトの例**

> ホームページをチェックして、重要な情報が明確な見出しが付いたプレーンな HTML で記述されていることを確認してください。名前、説明、URL、ロゴ、ソーシャル リンクを含む `Organization` スキーマを追加します。

**確認してください**

- 重要な事実を静的 HTML に含めます (たとえば、何をしているか、誰にサービスを提供しているか、価格設定など)
- 明確な見出し (H2、H3) を使用して情報を構造化する
- 包括的なスキーマの追加: `Organization`、`Product`、`FAQPage`、`Article`
- AI が解析して引用できる、明確で事実に基づいた言葉で書く
- サイト、スキーマ、外部プロファイル全体で事実の一貫性を保ちます。
</Accordion>

### AI ボットへのアクセス (robots.txt)

トレーニングやクエリへの回答のために、どの AI システムがコンテンツにアクセスできるかを制御します。 ZenOpus エージェントは、要求されたときに必要なボットを許可するように `robots.txt` を構成できます。

<Accordion title="robots.txt で AI ボット アクセスを実装して検証する方法">
**プロンプトの例**

> `robots.txt` を更新して GPTBot、PerplexityBot、Claude-Web を許可し、Google-Extended と CCBot をブロックします。すべての標準の検索エンジン クローラーを許可したままにします。

**確認してください**

- 可視性戦略を決定する: AI の引用を許可するか、AI トレーニングをブロックする
- 戦略に基づいてボットを明示的に許可またはブロックする
    - 一般的なボット: GPTBot (ChatGPT)、PerplexityBot、Claude-Web、Google-Extended (トレーニング)、CCBot
- 変更後に再チェックして、目的のボットが誤ってブロックされていないことを確認してください

**robots.txt の例**

```text
ユーザーエージェント: *
許可: /

ユーザーエージェント: GPTBot
許可: /

ユーザーエージェント: Google-Extended
許可しない: /
```
</Accordion>

### LLM で引用可能なコンテンツ パターン

AI システムは、明確で構造化されており、引用しやすいコンテンツを好みます。この形式で情報を提供すると、正確に引用される可能性が高くなります。 ZenOpus エージェントは、要求に応じて FAQ セクションと構造化コンテンツを生成できます。

<Accordion title="LLM で引用可能なコンテンツ パターンを実装および検証する方法">
**プロンプトの例**

> FAQ の回答を、短く、直接的で、AI が引用しやすいように更新します。主な回答から始めて、具体的な事実を含め、曖昧なマーケティング表現を削除します。

**確認してください**

- よくある質問に対する明確な回答を FAQ/Q&A 形式で記述します
- 引用しやすい定義パターンを使用します:「X は [明確な定義]」
- 曖昧なマーケティングコピーを避ける
- 簡単に解析できるようにリストと箇条書きを作成する
- 可能な場合は統計情報とソースを含めます

**例**

- 悪い: `Amazing transformative solutions`
- 良い: `USDA-certified organic dog food delivered weekly. Plans start at $45/month with free shipping over $50.`
</Accordion>

### 静的 LLM に適した概要ページ

専用の概要ページにより、AI システムが会社の重要な情報を理解し、引用することがはるかに簡単になります。 ZenOpus エージェントは、要求に応じて、重要な情報をクロール可能な形式で含む専用の概要ページを作成できます。

<Accordion title="LLM に適した静的な概要ページを実装して検証する方法">
**プロンプトの例**

> パブリック フォルダーの下に静的な `llm.html` ページを作成し、会社の事業内容、提供内容、違い、価格設定の基本、よくある質問、連絡方法を明確に説明します。 `Organization` および `FAQPage` スキーマを追加し、単純な H2/H3 見出しでページを編成します。この新しい URL を sitemap.xml に含めます

**確認してください**

- ページは、`/llm.html` や `/about-ai.html` などの明確な URL で作成されます。
- このページは `sitemap.xml` に含まれています。
- コンテンツは明確な H2/H3 見出しで構成されています。
- このページには、会社概要、製品/サービス、差別化要因、価格設定モデル、連絡先の詳細、FAQ などの重要な情報が含まれています。
- `Organization` および `FAQPage` スキーマが存在するため、AI システムは構造化された事実を抽出できます。
- コンテンツは事実に基づいており、簡潔で、引用可能です (マーケティング的な宣伝文句は避けてください)。
- 提供内容の変更に応じて、情報は常に最新の状態に保たれます。
</Accordion>

## モニタリングとメンテナンス: SEO と GEO を健全に保つ

効果的な SEO と GEO には継続的なモニタリングが必要です。 Google Search Console と定期的なチェックインを使用して、サイトを健全に保ち、インデックスに登録され、長期にわたって良好なパフォーマンスを維持します。

### Googleサーチコンソール（GSC）の設定

Google Search Console は、Google がサイトをどのようにクロール、インデックス付け、ランク付けするかを示します。これは、SEO のパフォーマンスを追跡し、技術的な問題を発見するための重要なツールです。

<Accordion title="監視とメンテナンスのために GSC を設定して使用する方法">
**カスタム ドメイン**を使用している場合は、Google Search Console で**カスタム ドメインを申請して確認**してください。

**サイトの所有権を確認** (方法を 1 つ選択してください)

サービスを提供する前に、Google がウェブサイトの所有者であることを確認できるように、ドメインを認証します。 Google はこの検証を使用して、インデックス作成、サイトマップのステータス、ライブ ドメインの検索パフォーマンスを追跡します。

- **DNS TXT レコード** (推奨): ドメイン プロバイダーの DNS 設定に TXT レコードを追加します。
- **メタ タグの検証**: メタ タグを追加する prompt ZenOpus

> GSC 検証メタ タグ: &lt;meta name=‘google-site-verification’ content=‘YOUR_CODE’ /&gt; を &lt;head&gt; に追加します。

- **HTML ファイルのアップロード**: Google の検証ファイルをアップロードし、サイトのルート (通常は `/public`) に配置するように ZenOpus に依頼します。

**サイトマップを送信**

1. GSC で **サイトマップ** に移動します
1. サイトマップの URL を入力してください: `https://yourdomain.com/sitemap.xml`
1. **送信**をクリックします

Google がこの情報を処理して報告するまでには **24～48 時間**かかる場合があります。

**URL 検査ツールを使用する**

これを使用して次のことを行います。

- Google が実際のコンテンツ (空白のページではない) を認識していることを確認する
- CSR レンダリングの問題を診断する
- 主要なリソース (JavaScript および CSS) がブロックされているかどうかを確認する

任意の URL の場合:

1. **URL 検査** バーに URL を入力します
1. **ライブ URL をテスト** をクリックします
1. **テスト済みページを表示** を開いて次のことを確認します。
    - Googlebot が認識するもののスクリーンショット
    - レンダリングされたHTML
    - コンソールエラー
    - ブロックされたリソース
1. 新しいページまたは更新されたページの [**インデックス作成のリクエスト**] をクリックします (レート制限あり)

**毎週監視するレポート**

- **カバレッジ/ページのインデックス作成:**
    - インデックス付きページと除外ページ
    - クロールエラーと除外理由
    - 時間の経過に伴う指数化の傾向
- **パフォーマンス (検索結果):**
    - インプレッション数、クリック数、CTR、平均掲載順位
    - 上位のクエリとページ
    - ページの可視性の向上または喪失
- **Web の重要な要素:**
    - 現実世界の LCP、INP、CLS
    - 遅くて問題のあるページを特定する
    - モバイルとデスクトップのパフォーマンスを比較する
- **モバイルでの使いやすさ:**
    - タップターゲット、ビューポート、コンテンツ幅に関する問題
    - モバイルランキングを維持するためにエラーを修正する
</Accordion>

### 推奨されるメンテナンススケジュール

SEO と GEO を健全に保つには、定期的なチェックインが必要です。この継続的なスケジュールを使用して、インデックス作成の問題、パフォーマンスの変化、コンテンツの更新に先んじて対処してください。

<Accordion title="メンテナンススケジュール: 毎週、毎月、四半期ごとに確認する内容">
**毎週**

- GSC カバレッジをチェックして、新しいエラーや新しくインデックスされたページがないか確認してください
- 検索パフォーマンスの傾向を一目で見る
- 新しく重要な URL に対して URL 検査を実行し、必要に応じてインデックス作成を要求します。

**毎月**

- ページが変更された場合は `sitemap.xml` を更新します
- パフォーマンスの低いタイトルと説明を更新します (CTR の低いページ)
- 組み込みの **Speed ツール (Lighthouse 監査)** を実行します。
- コンテンツの改善やリンク構築を小規模に行う

**四半期**

- より詳細な技術監査を行う: 正規化、`robots.txt`、スキーマ、内部リンク、モバイルのユーザビリティ
- キーワードのパフォーマンスと上位のランディング ページを確認する
- 古いコンテンツまたは衰退したコンテンツを更新する
</Accordion>

<Accordion title="プリレンダリングとは何ですか?いつ必要になるのですか?">
プリレンダリング (動的レンダリングとも呼ばれます) は、ボット専用の静的な HTML スナップショットを生成しますが、人間の訪問者には引き続き JavaScript を利用したアプリが表示されます。これにより、検索エンジンと AI クローラーがコンテンツを理解する速度が向上します。

プリレンダリングは特に次の場合に役立ちます。

- コンテンツの多いサイト
- 新しいページを頻繁に公開するサイト
- SEO に重点を置いた、または競争の激しいニッチ市場
- より高速なインデックス作成が必要なプロジェクト
- 多くのクローラは JavaScript を実行せず、完全にレンダリングされた HTML の恩恵を受けるため、AI / LLM の強力な可視性が得られます。

ZenOpus には、そのままではプリレンダリングが含まれていませんが、次のような外部サービスを使用して、ホスティングまたはプロキシ経由でプリレンダリングを追加できます。

- [Prerender.io](https://Prerender.io): 広く使用されている信頼性の高いオプション
- [DataJelly](https://datajelly.app/): 簡単なセットアップで同様の機能を実現
- [Rendertron](https://github.com/GoogleChrome/rendertron): オープンソースの代替手段 (自己ホスト型)
</Accordion>
<Accordion title="CSR SEO の主な制限は何ですか?また、それらを回避するにはどうすればよいですか?">
クライアントサイド レンダリング (CSR) は多くのサイトで適切に機能しますが、SEO、GEO、プレビューの動作を最適化するには、注意すべき点がいくつかあります。それらに対処する方法は次のとおりです。

- **インデックス作成は SSR よりわずかに遅い場合があります**  
→ サイトマップを送信し、優先ページの URL 検査ツールを使用します。
- **多くの AI クローラーと一部のボットは JavaScript を実行しません**  
→ 明確なセマンティック HTML とスキーマを使用し、LLM 概要ページの追加を検討してください。
- **ソーシャル プラットフォームはリンク プレビュー用の JavaScript を実行しません**  
→ Open Graph タグと Twitter/X Card タグを静的 HTML に直接追加します。
- **メタデータはルート間で自動的に更新されません**  
→ ZenOpus に `react-helmet-async` をインストールし、各ページに固有のタイトルと説明を設定するよう依頼します。
</Accordion>
