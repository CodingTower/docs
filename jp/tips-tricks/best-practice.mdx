このガイドは、初心者でも経験者でも、すべてのユーザーがすぐに慣れて、ZenOpus で構築するときによくある落とし穴を避けるのに役立ちます。

## 1. 基礎を確立する: ナレッジ ファイルを使用する

**重要な理由:** [ナレッジ](/jp/features/knowledge) ファイルはプロジェクトの頭脳です。これはすべてのプロンプトとともに送信され、AI が完全なコンテキストを理解するのに役立ちます。

**含める内容:**

- 製品のビジョン (PRD のように考えてください)
- ユーザージャーニーとペルソナ
- 主な特徴と機能
- デザインシステムとUIガイダンス
- 役割固有の動作 (管理者、ユーザー、投資家など)

<Tip>
Plan mode を使用してナレッジ ファイルを自動生成できます。

```text
すでに実装した機能に基づいて、T=0 でプロジェクトのナレッジを生成します。
```
</Tip>

## 2. ベストプラクティスを促す

**明確で冗長なプロンプト = より良い出力。** AI をエンジニアリング パートナーのように考えてください。AI は、ユーザーが指示したことだけを認識します。

**プロンプトのヒント:**

- 具体的にする: 正確なページ (例: `/dashboard`) と予想される動作について言及します。
- 自然言語を使用する
    ```text
    このコンポーネントには、投資家ロールを持つユーザーにはアクセスしてもらいたいのですが、管理者にはアクセスできないようにします。
    ```
- スクリーンショットの追加: バグや UX の問題を説明する場合に特に役立ちます。
- ガードレールを追加する: 触れてはいけないものを AI に指示します。例えば。
    ```text
    /shared/Layout.tsx は編集しないでください。
    ```
- プロンプト間で重要な指示を繰り返します。 AI のメモリには制限がある場合があります。
- 一度に 5 つのことを実装しようとすることは避けてください。作業をテスト可能な小さな塊に分割します。次に進む前に、各ブロックの間に Plan mode を使用して検証します。
    ```text
    **機能の内訳テンプレート:**
    新しいページを作成する
    UIレイアウトを追加する
    データを接続する
    ロジック + エッジケースを追加
    役割ごとのテスト
    ```
- アプリに複数のロール (管理者、投資家、スタートアップなど) がある場合は、プロンプトが *どのロールに適用されるかを常に定義してください。これは、共有ロジック/コンポーネントによって引き起こされるバグを回避するのに役立ちます。
    ```text
    投資家として会社のダッシュボードを表示したいのですが、編集できないはずです。この機能は投資家ロールのみに分離してください。
    ```

## 3. Plan mode を早期かつ頻繁に使用する

[**Plan mode**](/jp/features/plan-mode) = AI 副操縦士。準備が整うまでコードを編集することなく、デバッグ、ブレインストーミング、実装計画を行うのに役立ちます。

**Plan mode に切り替える場合:**

- 「修復を試みる」試行が 2 ～ 3 回失敗した後
- 複雑なロジックまたはデータベースの問題をデバッグする場合
- 新しい機能を計画するとき
    ```text
    X を実装する 3 つの方法を提案する
    ```

<Tip>
**ワークフローのヒント:**

ユーザーの中には、時間の 60 ～ 70% を Plan mode を使用することを好む人もいます。完全に満足した場合にのみ、「計画を実行する」をクリックしてください。
</Tip>

<Tip>
Plan mode の使用を忘れた場合でも、この形式により出力の一貫性が向上し、付随的な編集が防止されます。

```text
/settings ページで、[機能] を実装します。予期される動作は [XYZ] です。必要な場合を除き、コンポーネント A、レイアウト B、または共有ロジックには触れないでください。 Tailwind / Supabase / X のベスト プラクティスに従ってください。
```
</Tip>

<Tip>
不要なコードの実行を回避するには:

```text
調査はしますが、まだコードは記述しません。
```

```text
何も変更せずにこれを解決する 3 つの方法を提案します。
```

これにより、コントロールを自分の手で維持できます。
</Tip>

<Tip>
AI が「ループ」に入ったときは、次のシーケンスを使用して、壊れたコードにパッチを適用する無限のサイクルを回避します。

1. Plan mode に切り替える
1. エラーのスクリーンショットを貼り付けます
1. 言います:
    ```text
    他の機能を損なわずにこれを調査してください。必要に応じて、最後に動作していたバージョンに戻し、そこから修正します。
    ```
</Tip>

## 4. Supabase でよくある落とし穴を回避する

**注意:** [Supabase](/jp/integrations/supabase) は完全に元に戻りません。バージョンを元に戻すと、データベース スキーマが破損する可能性があります。

**ベストプラクティス:**

- フロントエンドが安定した **後** に Supabase に接続します
- 元に戻す必要がある場合は、AI に次のプロンプトを出します。
    ```text
    T=0 で SQL スキーマを検証し、重大な変更が発生していないことを確認してください。
    ```
- 公開する前にデータベースにリンクされた機能を必ずテストしてください

## 5. ビジュアル編集を使用して UI を簡単に修正する

[**ビジュアル編集ツール**](/jp/features/visual-edit) は無料で高速です。次のプロンプトの代わりに使用します。

- テキスト、色、フォント、レイアウトの微調整の変更
- 複数の小さな要素を一度に編集する
- 安全でクレジットのないコミット (元に戻すことも可能)

## 6. GitHub + バージョン管理を賢く利用する

- すべての編集はコミットです。 **ピン留め**を使用して安定バージョンをマークします。すべての機能が機能した後: **ピン留め**
- すべてのバグの後: **バージョンを視覚的に比較します。** 次のプロンプトを表示できます。
    ```text
    T–1 と T–0 のバージョンを比較します。何が変わったのでしょうか？何が壊れているのでしょうか?
    ```
- AI が多くの機能を破壊していると感じた場合は、安定したバージョンに戻ってください。
- **GitHub ブランチング**はご自身の責任で使用してください。プロジェクトの同期の問題を防ぐため、ZenOpus の `main` に戻す前にブランチを削除しないでください。

## 7. すべてがうまくいかないときは、リミックスしてください

多くのユーザーは、最初からやり直した方が 2 回目は時間がかからないことに気づいています。

- [Remix](/jp/introduction/faq#プロジェクトをコピー（リミックス）するにはどうすればよいですか) は、T=0 にプロジェクトのクリーン コピーを作成します。
- より適切なプロンプトとより明確な知識を使用して再構築する
- 古いプロジェクトは参照としてのみ使用してください

**使用例:**

- バグだらけのループにはまってしまう
- 履歴を保存した状態でクリーンを再起動したい
- Supabase を切断し、新しいパスを試す必要があります

<Tip>
リミックスするには、まず Supabase を切断する必要があります。
</Tip>

## 8. 辛抱強く、落ち着いて

あなたは一人ではありません。 AI はある瞬間には魔法のようになり、次の瞬間にはイライラすることがあります。多くの場合、ビルドの最後の 5% が最も遅くなりますが、最も重要です。

<Tip>
**黄金律:**

プロンプトには時間をかけてください。すべてを再確認してください。作業をテスト可能な小さなブロックに分割します。入力が正確であればあるほど、出力も向上します。
</Tip>

## 9. ドキュメントを使用して助けを求める

- [ドキュメント](/) には、ウォークスルー、テンプレート、SEO のヒント、統合などが含まれています。 doc AI アシスタントで直接質問できます。
- ピア Support の [Discord コミュニティ](https://discord.gg/zenopus-dev) に参加してください。

## 10. ボーナスヒント

- 長いプロンプトの場合は、ディクテーションを使用して音声メモ プロンプトを追加します (例: Mac では、マイクを使用して長いプロンプトをディクテーションします)。より良いインプットをより速く作成できるようになります。イライラしたり疲れたりしたときに特に役立ちます。
- 「`I am frustrated…`」プロンプト パターンを使用して AI の集中力を高めます
- 大規模な編集後は、必ず複数のロールとその動作を再確認してください (特に条件付きロジックの場合)。
- 素早いデバッグのために安定バージョンをフォールバックとして保存します
- 予期しない副作用が発生した場合、これは過度に汎用的なロジックによって引き起こされるバグを回避するのに役立ちます。
    ```text
    [ロール X] 専用のコンポーネントを作成し、範囲が明確でない限り共有コンポーネントを再利用しないでください。
    ```
