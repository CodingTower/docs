ZenOpus Cloud は、実稼働対応のアプリケーションを構築するための**最も簡単、最速、そして最も楽しい**方法となるように設計されています。

高品質の AI エージェント、ビジュアル エディター、プレビュー環境、および完全に管理されたインフラストラクチャを提供するため、オペレーティング システムではなく構築に集中できます。

ZenOpus は、**ユーザーがロックインされることがない**ように意図的に構築されています。いつでも、コードのエクスポート、データ、セルフホスト部分またはスタック全体の移行、または別のプロバイダーへの移動を行うことができます。必要に応じてではなく、自らの選択によって ZenOpus を使用していただきたいと考えています。

## 重要な原則

- **自分のコードはあなたが所有しています**。  
すべての ZenOpus プロジェクトは、[GitHub 統合](/jp/integrations/github) を使用していつでも継続的に GitHub に同期できます。リポジトリのクローンを作成したり、__​​TERM2__ の外部でリポジトリを変更したり、独自のインフラストラクチャにデプロイしたり、制限なく完全に自己ホストしたりすることができます。
- **あなたのデータはあなたが所有しています**。  
データベース、ストレージ、構成は移植可能です。スキーマとデータをエクスポートしたり、別のプロバイダーに移行したり、管理するインフラストラクチャ上ですべてを実行したりできます。 ZenOpus は独自のデータ ストアに依存しません。
- **ZenOpus はオープンソース テクノロジに基づいて構築されています**。  
すべてはオープン スタンダードとオープンソース テクノロジーで実行されます。アプリケーションは、主要なクラウド プロバイダー (AWS、GCP、Azure)、マネージド ホスティング プラットフォーム、またはセルフホステッド インフラストラクチャ上で実行できる標準的な Vite + React プロジェクトです。独自のフレームワークや隠れた依存関係はありません。
- **私たちは管理ではなく品質を通じてお客様の信頼を獲得します**。  
ZenOpus Cloud は、速度、シンプルさ、信頼性を考慮して最適化されています。要件が変わっても、摩擦なく移行できます。その自由は意図的なものです。

## ZenOpus Cloud で得られるもの

ZenOpus は多くのホスティングおよびインフラストラクチャ設定で動作するように設計されていますが、ZenOpus Cloud は、特にアクティブな開発および反復中にプラットフォームが最も価値を提供する場所です。

ZenOpus Cloud でホストすると、開発、プレビュー、実稼働が 1 つの統合環境で一緒に動作します。

あなたは構築に集中します。プラットフォームを担当しております。

**ZenOpus Cloud でアプリを実行すると、次のことが可能になります:**

- アプリケーションと緊密に統合された AI 支援開発
- ビジュアル編集およびリアルタイム プレビュー環境
- カスタム ドメイン、自動 SSL、グローバル配信を使用した本番ホスティング
- 自動展開と環境管理
- フルマネージドのバックエンド、データベース、インフラストラクチャ
- 組み込みの認証とデータ分離
- 統合された統合とシークレット管理
- ランタイム機能へのマネージド AI プロバイダー アクセス
- 自動セキュリティ スキャンとコンプライアンス対応インフラストラクチャ (SOC 2 Type 2 および ISO 27001 認定)
- 運用上のオーバーヘッドを最小限に抑える

## ZenOpus の使用を停止すると何が変わるか

ZenOpus の使用を完全に停止すると (エディター、エージェント、プレビュー、およびホスティング)、ZenOpus は開発およびデプロイメントのワークフローを管理しなくなります。

**あなたには以下の責任があります:**

- 開発環境の構成と保守およびワークフローのデバッグ
- CI/CD パイプラインとリリース プロセスの構築と管理
- 運用インフラストラクチャ、SSL 証明書、CDN 構成、およびスケーリングの運用
- データベース、ストレージ システム、バックアップ、アクセス制御の実行
- 認証の実装とデータ分離ポリシーの強制
- OAuth フロー、シークレット、外部 API 統合の管理
- AI プロバイダーのアカウント、資格情報、請求、レート制限の管理
- セキュリティ監査、監視、コンプライアンス体制の維持

これは、単にフルマネージド プラットフォームから移行するための運用上の境界にすぎません。

## ZenOpus がスタックにどのように適合するか

ZenOpus を使用して構築されたアプリケーションは、次の 3 つの独立した部分で構成されます。

|パート |デフォルト | | でも実行できます。
| --- | --- | --- |
|コード | ZenOpus で管理され、GitHub に同期されます。 Git ベースのワークフロー |
|フロントエンド | ZenOpus Cloud とカスタム ドメイン Support |マネージド プロバイダー (Netlify、Cloudflare Pages、S3 + CDN など)、または自己管理型インフラストラクチャ (コンテナー、仮想マシン、または Kubernetes) |
|バックエンドとデータ | ZenOpus Cloud |マネージド Supabase、セルフホスト型 Supabase、または同等の認証、ストレージ、エッジ サービスを備えたその他の PostgreSQL ベースのバックエンド |

ZenOpus Cloud はフロントエンドとバックエンドの両方をホストできますが、どちらかの部分をマネージド プラットフォームまたは独自のインフラストラクチャに独立して移動できます。

## ZenOpus プラットフォームの互換性

ZenOpus アプリケーションは、オープンソース テクノロジに基づいて構築された標準の **Vite + React プロジェクト**です。フロントエンドは、Support Node.js が構築するインフラストラクチャおよび静的ファイル ホスティングにデプロイできます。

- **アマゾン ウェブ サービス (AWS):**  
静的ホスティング + CDN (S3 + CloudFront)、コンテナ サービス (ECS)、マネージド Kubernetes (EKS)、アプリケーション ホスティング (Amplify)
- **Google クラウド プラットフォーム (GCP):**  
静的ホスティング + CDN (Cloud Storage + Cloud CDN)、コンテナ サービス (Cloud Run)、マネージド Kubernetes (GKE)
- **Microsoft Azure:**  
静的サイト ホスティング (Static Web Apps)、コンテナー サービス (Azure Container Apps)、マネージド Kubernetes (AKS)
- **セルフホスト型インフラストラクチャ:**  
コンテナー (Docker)、Kubernetes クラスター、Nginx または Apache を実行する仮想マシン
- **マネージド フロントエンド ホスティング プラットフォーム:**  
Netlify、Cloudflare ページ、Vercel

バックエンドが ZenOpus Cloud を使用している場合、フロントエンドを任意の場所でホストしている間、バックエンドは ZenOpus Cloud のままにすることができます。フロントエンド ホスティングの移行にはアーキテクチャの変更は必要ありません。

バックエンドを移動する必要がある場合は、Support して **マネージド Supabase** および **セルフホスト型 Supabase** に移行します。一部の ZenOpus Cloud 機能は、ZenOpus Cloud 以外では使用できません。

基盤となるデータベースは PostgreSQL ですが、ZenOpus アプリケーションは認証、ストレージ、リアルタイム、エッジ機能などの Supabase 固有のサービスに依存しています。プレーンな PostgreSQL または他のデータベース プロバイダーに移行するには、同等の認証、ストレージ、エッジ サービスを実装する必要があり、そのままではサポートされません。

## 一般的な展開とホスティングのセットアップ

ほとんどの人は、最終的に次のいずれかの構成になります。

### ZenOpus Cloud のみ (推奨)

- プレビューと本番環境は ZenOpus Cloud でホストされます
- バックエンドとフロントエンドは ZenOpus によって完全に管理されます
- カスタム ドメインを完全にサポート
- インフラストラクチャのセットアップやメンテナンスは不要

ほとんどのアプリケーションに最適です。

### ZenOpus + マネージド プラットフォーム (ハイブリッド)

- ZenOpus は開発、プレビュー、ツールに使用されます
- マネージド サービスにデプロイされた 1 つ以上の実稼働コンポーネント:
    - マネージド バックエンド (たとえば、マネージド Supabase、または独自の認証、ストレージ、バックエンド サービスと組み合わせたクラウドマネージド PostgreSQL)
    - マネージドフロントエンドホスティングプラットフォーム (Cloudflare Pages、Netlify、クラウドプロバイダーの静的ホスティングなど)
- ZenOpus と運用環境の間のブリッジとして GitHub が使用される

特定のバックエンドまたはコンプライアンス要件があるユーザーに最適です。

### 完全に自己管理されたインフラストラクチャ (高度な)

- GitHub経由でコードを同期
- バックエンド サービスとデータベースは、お客様が運用するインフラストラクチャ上で実行されます (たとえば、セルフホスト型の Supabase、または独自の認証、ストレージ、エッジ サービスと組み合わせたマネージド PostgreSQL)。
- コンテナ、VM、または内部プラットフォームを介してデプロイされたフロントエンド
- ZenOpus は引き続き開発に使用できます。または、開発を他のツールに完全に移行できます。

<Note>
ZenOpus プラットフォーム自体 (エディターと AI エージェント) はマネージド サービスであり、自己ホストしたり、顧客 VPC 内にデプロイしたりすることはできません。  

  

 ZenOpus を使用して構築したアプリケーションは、どこにでもホストできます。
</Note>

### セットアップの比較

|セットアップ |コード |フロントエンド |データ |バックエンド |
| --- | --- | --- | --- | --- |
| ZenOpus Cloud のみ (推奨) | ZenOpus で管理 (GitHub 同期はオプション) |カスタム ドメイン Support を使用して ZenOpus によって完全に管理されます。 ZenOpus によって完全に管理 | ZenOpus による完全管理 |
| ZenOpus + マネージド プラットフォーム (ハイブリッド) | ZenOpus と GitHub リポジトリ間の双方向同期 |選択したプラットフォーム (Netlify、Cloudflare Pages、S3 + CDN など) でホスト |選択したプロバイダー (通常は管理対象の Supabase) によってホストされます。選択したプロバイダー (マネージド Supabase または同等のバックエンド サービス) によってホストされます。
|完全に自己管理されたインフラストラクチャ (上級) | Git リポジトリ (ZenOpus 開発中はオプション) |インフラストラクチャ (コンテナー、仮想マシン、または Kubernetes) でホスト |インフラストラクチャ上でホストされます (通常は、同等のサービスを備えたセルフホスト型の Supabase または PostgreSQL ベースのデータベース)。インフラストラクチャ上でホスト (セルフホスト型の Supabase または同等の認証、ストレージ、リアルタイム、エッジ サービスを運用) |

## 推奨される導入パス

事前にすべてを決める必要はありません。 ZenOpus を使用すると、シンプルに始めて、要件の変化に応じて段階的に移行できます。

推奨されるパス:

1. **ZenOpus Cloud** から開始します。これは、運用オーバーヘッドを最小限に抑えて本番環境に移行する最速のパスです。
1. **GitHub に同期**。これにより、コードの所有権が保証され、外部コラボレーションが可能になります。
1. **仮想的な制約ではなく、実際の制約に達した場合にのみコンポーネントを移動します**。

ほとんどのアプリケーションでは、ZenOpus Cloud でのホスティングが最も効率的で信頼性の高いオプションです。

ZenOpus Cloud では満たせない特定の要件がある場合にのみ、スタックの一部を移動することを検討してください。

- **コンプライアンスまたはデータ常駐** の義務は特定のインフラストラクチャ プロバイダーに関連付けられています
- **ネットワーク構成** は ZenOpus ではサポートされていません (VPC ピアリング、プライベート エンドポイントなど)
- **組織のポリシー** では、自己管理または監査されたホスティング環境が必要です

これらは特殊なケースであり、デフォルトのパスではありません。よくわからない場合は、ZenOpus Cloud から始めてください。後で再構築しなくてもいつでも移動できます。

実際の例と展開手順については、[ZenOpus Cloud 外部での展開とホスティング](/jp/tips-tricks/external-deployment-hosting) を参照してください。
