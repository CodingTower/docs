このガイドでは、ZenOpus Cloud の外部に ZenOpus アプリケーションをデプロイおよびホストするための **実践的な例と手順** を提供します。

これは、[ZenOpus Cloud による展開、ホスティング、および所有権のオプション](/jp/tips-tricks/deployment-hosting-ownership) で説明されている概念に基づいて構築されており、最も一般的な移行パスに焦点を当てています。これらのシナリオはいずれも、独立して任意の順序で採用できます。

このガイドでは、**ホスティング**はアプリケーションが実行される場所を指し、**デプロイメント**はコードがどのように構築され、その環境に配信されるかを指します。

すべてを ZenOpus Cloud で実行する場合は、このガイドは必要ありません。

## 移行する前に

このガイドは、コンプライアンスの制約、データ常駐のニーズ、組織のインフラストラクチャ ポリシーなど、アプリケーションの一部を ZenOpus Cloud の外部で実行する必要がある特定の要件を持つチームを対象としています。

ZenOpus Cloud は、次のような統合開発環境と運用環境をすでに提供しています。

- カスタム ドメインと自動 SSL を使用した本番ホスティング
- 自動化された展開と環境管理
- 行レベルのセキュリティによるマネージド認証
- データベースとファイルストレージ
- あらゆる変更に対する自動プレビュー環境
- プレビュー環境および開発環境でデバッグできる AI エージェント
- ランタイム機能へのマネージド AI プロバイダー アクセス
- 組み込みの OAuth 構成とトークンの更新
- セキュリティ スキャンとコンプライアンス対応インフラストラクチャ (SOC 2 Type 2、ISO 27001)

ほとんどのチームは移行する必要がありません。 ZenOpus Cloud から開始して、後で実際の制約に達した場合にコンポーネントを移動できます。

## 前提条件: プロジェクトを GitHub に接続する

このガイドのすべてのシナリオでは、[プロジェクトを GitHub に接続する](/jp/integrations/github) 必要があります。

GitHub が接続されている場合:

- ZenOpus プロジェクトはリポジトリと継続的に同期されます
- 外部プラットフォームは GitHub から直接デプロイします
- ZenOpus は引き続き開発、プレビュー、ツールを個別に管理します

ここからこのガイドでは、プロジェクトが GitHub に接続されていることを前提としています。

## マネージド プラットフォーム上で運用フロントエンドをホストする

これは、ZenOpus Cloud の外に移動する場合の最も一般的な最初の手順です。

開発とプレビューに ZenOpus を使用し続けながら、運用フロントエンドをマネージド ホスティング プラットフォームにデプロイします。バックエンドとデータは ZenOpus Cloud に残すことも、別の場所で実行することもできます。

### あなたが責任を負っていること

運用フロントエンドが ZenOpus Cloud の外部で実行される場合、次の責任があります。

- フロントエンド展開パイプラインとロールバック
- 運用環境変数
- CDN の動作とキャッシュ
- フロントエンドの可用性と稼働時間
- 実稼働ログとデプロイメント履歴
- 本番ブランチまたはリリースのプレビュー環境

ZenOpus は、制御していない運用インフラストラクチャを監視またはデバッグできません。

### 一般的なアプローチ

- **Git ベースのホスティング プラットフォーム**  
これらのプラットフォームは GitHub リポジトリに直接接続し、プッシュするたびに自動的にビルドしてデプロイします。
    - ネットリファイ
    - クラウドフレアページ
    - ヴェルセル
    - AWS Amplify ホスティング
    - Azure 静的 Web アプリ
    - Google Firebase ホスティング
- **オブジェクト ストレージ + CDN ホスティング**  
これらのプラットフォームは CDN の背後で静的ファイルをホストしますが、`dist/` 出力を生成してアップロードするにはビルド パイプラインが必要です。
    - **AWS:** S3 + CloudFront
    - **Google Cloud:** クラウド ストレージ + クラウド CDN
    - **Azure:** Azure Storage (静的 Web サイト) + Azure CDN または Front Door

### Git ベースのホスティング プラットフォームへのデプロイ

このアプローチは、GitHub リポジトリから自動的に構築およびデプロイされるプラットフォームに適用されます。

<Accordion title="例: Git ベースのホスティング プラットフォームへのデプロイ">
<Steps>
<Step title="リポジトリに接続する">
GitHub リポジトリをホスティング プラットフォームに接続します。
</Step>
<Step title="ビルド設定を構成する">
ほとんどのプラットフォームは以下を自動検出します。

- **ビルド コマンド**: `npm run build`
- **出力ディレクトリ**: `dist`
- **ノードのバージョン**: 22
</Step>
<Step title="環境変数を構成する">
バックエンドのないプロジェクトには環境変数は必要ありません。

ZenOpus Cloud を使用するプロジェクトには、プロジェクトの `.env` ファイルの値が必要です。

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
VITE_SUPABASE_PROJECT_ID
```

これらは、ZenOpus のコード エディターの `.env` ファイル、または同期された GitHub リポジトリで見つけることができます。
</Step>
<Step title="SPA ルーティングの構成">
直接 URL ナビゲーションが `404` を返した場合は、すべてのルートが `/index.html` を提供するようにフォールバック リライトを構成します。この方法はプラットフォームによって異なります (たとえば、Netlify/Cloudflare の `_redirects` ファイル、Azure の `staticwebapp.config.json`、Amplify のルールの書き換え、Vercel の `vercel.json`、Firebase の `firebase.json`)。
</Step>
<Step title="OAuth リダイレクト URL を更新する">
アプリが Google サインインまたは他の OAuth プロバイダーを使用している場合は、認証プロバイダーの許可されたリダイレクト URL に新しい運用ドメインを追加します。
</Step>
<Step title="デプロイ">
GitHub へのプッシュごとに、新しい運用環境のデプロイメントがトリガーされます。
</Step>
</Steps>

**結果:** 公開アクセス可能な運用フロントエンドが HTTPS 経由で提供され、ZenOpus Cloud バックエンドに接続されます。
</Accordion>

### CI/CD を使用したオブジェクト ストレージ + CDN へのデプロイ

CDN ベースのホスティングでは、アプリを構築して `dist/` 出力をアップロードするための CI/CD パイプラインが必要です。ビルド手順はプロバイダーを問わず同じです。導入はプロバイダーごとに異なります。各プラットフォームの公式ドキュメントへのリンクを参照してください。

<Accordion title="例: CI/CD を使用したオブジェクト ストレージ + CDN へのデプロイ">
認証には、すべての主要なクラウド プロバイダー Support **OpenID Connect (OIDC)** と GitHub Actions を使用します。これにより、長期有効な資格情報をシークレットとして保存する必要がなくなります。これが推奨されるアプローチです。

- [AWS: OIDC で IAM ロールを使用する](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Google Cloud: Workload Identity Federation](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)
- [Azure: フェデレーション ID 資格情報](https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation)

<Steps>
<Step title="GitHub シークレットを追加する">
GitHub リポジトリ設定に必要なシークレットを追加します (**設定 → シークレットと変数 → アクション**)。

|秘密 |値 |
| --- | --- |
| VITE_SUPABASE_URL | Supabase プロジェクトの URL (プロジェクトの .env ファイルから) |
| VITE_SUPABASE_PUBLISHABLE_KEY | Supabase 公開可能キー (プロジェクトの .env ファイルから) |
</Step>
<Step title="ワークフローファイルを作成する">
たとえば、リポジトリに `.github/workflows/deploy.yml` を作成します。

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build application
        run: npm run build
        env:
          VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
          VITE_SUPABASE_PUBLISHABLE_KEY: ${{ secrets.VITE_SUPABASE_PUBLISHABLE_KEY }}

      # Deploy step is provider-specific. See documentation links below.
      # The build output is in the dist/ directory.
```
</Step>
<Step title="プロバイダー固有の展開手順を追加する">
ビルドステップの後に、プラットフォームのデプロイメントステップを追加します。現在のベスト プラクティスについては、公式ドキュメントを参照してください。

|プラットフォーム |ドキュメント |
| --- | --- |
| AWS S3 + CloudFront | Amazon S3 、AWS 向け GitHub アクションへのデプロイ |
| Google クラウド ストレージ |静的ウェブサイトのホスティング、GitHub Actions for Google Cloud |
| Azure Blob ストレージ | Azure Storage での静的 Web サイトのホスティング、Azure 用 GitHub Actions |
</Step>
<Step title="SPA ルーティングの構成">
すべての CDN ベースのホスティングでは、ファイルと一致しないルートに対して **200** ステータス コードを含む `index.html` を返すように構成する必要があります。これは通常、次のように構成されます。

- **200 を返すカスタム エラー応答** (CloudFront)
- **ロードバランサ上の URL 書き換えルール** (Cloud CDN)
- **URL 書き換えルール** (Front Door)

<Warning>
ストレージ レベルのエラー ページ設定は 404 ステータス コードを返すため、これに依存しないでください。
</Warning>
</Step>
</Steps>
</Accordion>

## 実稼働フロントエンドを独自のインフラストラクチャでホストする

フロントエンドのホスティング、ネットワーキング、またはランタイム環境を完全に制御する必要がある場合は、このアプローチを使用します。これは **セルフホスティング** と呼ばれることもあります。

フロントエンドは GitHub リポジトリから構築され、管理するインフラストラクチャにデプロイされます。バックエンドとデータは ZenOpus Cloud に残すことも、別の場所で実行することもできます。

### あなたが責任を負っていること

運用フロントエンドが管理するインフラストラクチャ上で実行される場合、次の責任があります。

- ビルドとデプロイメントの自動化
- SSL/TLS設定
- CDN とリバース プロキシの構成
- 監視、ロギング、稼働時間
- インフラストラクチャの更新とセキュリティ
- ブランチまたはリリースのプレビュー環境

ZenOpus は、制御していない運用インフラストラクチャを監視またはデバッグできません。

### 一般的なアプローチ

- **コンテナベースの展開**  
例: Kubernetes (EKS、GKE、AKS)、ECS、Nomad、または内部コンテナ プラットフォーム経由でデプロイされた Docker
- **Web サーバーの背後にある仮想マシン**  
例: Nginx または Apache を実行する Linux VM、構成管理または内部ツールを通じて管理
- **社内 PaaS プラットフォーム**  
例: 社内導入プラットフォームまたはプライベート クラウド PaaS ソリューション

### ビルド要件

ZenOpus プロジェクトは標準の Vite アプリケーションであり、静的フロントエンドとして構築されます。

- **ビルドコマンド:**`npm run build`
- **出力ディレクトリ:**`dist/`
- **ノード バージョン:** 22 を推奨

`VITE_` というプレフィックスが付いた環境変数は、実行時ではなく **ビルド時**に埋め込まれます。

ZenOpus Cloud をバックエンドとして使用する場合は、`npm run build` を実行する前にこれらを設定する必要があります。

- __コード0__
- __コード0__

これらの値は、プロジェクトの `.env` ファイルで確認できます。

### コンテナベースのデプロイメント (Docker)

<Note>
これらの Docker 構成は、ZenOpus エージェントを使用して直接生成できます。以下の [ZenOpus を使用した Docker デプロイメントの生成](#zenopus-を使用した-docker-デプロイメントの生成) を参照してください。
</Note>

<AccordionGroup>
<Accordion title="例: 手動の Docker デプロイメント">
これは、Kubernetes、ECS、Cloud Run、またはその他のコンテナ オーケストレーション プラットフォームにデプロイするための最も一般的なアプローチです。フロントエンドを単一の VM 上のスタンドアロン コンテナーとして実行する場合にも機能します。

<Steps>
<Step title="GitHub リポジトリのクローンを作成します">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="Dockerfileを作成する">
ZenOpus プロジェクトは、標準のマルチステージ Docker ビルドを使用してコンテナ化できます。最初のステージでは依存関係をインストールして `npm run build` を実行し、2 番目のステージでは nginx で静的出力を提供します。

適応できる Dockerfile の例を次に示します。

```docker
# Build stage
FROM node:22-alpine AS builder

WORKDIR /app

# Copy package files and install dependencies
COPY package*.json ./
RUN npm ci

# Copy source code
COPY . .

# Environment variables are embedded at build time
ARG VITE_SUPABASE_URL
ARG VITE_SUPABASE_PUBLISHABLE_KEY
ENV VITE_SUPABASE_URL=$VITE_SUPABASE_URL
ENV VITE_SUPABASE_PUBLISHABLE_KEY=$VITE_SUPABASE_PUBLISHABLE_KEY

# Build the application
RUN npm run build

# Production stage
FROM nginx:alpine

# Copy built assets to nginx
COPY --from=builder /app/dist /usr/share/nginx/html

# Copy nginx configuration for SPA routing
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```
</Step>
<Step title="nginx設定ファイルを作成する">
ZenOpus プロジェクトはクライアント側ルーティング (BrowserRouter を備えた React Router) を使用するため、Web サーバーはすべてのルートに対して `index.html` を返す必要があります。 `nginx.conf` ファイルを作成します。

```nginx
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    # Cache static assets
    location /assets/ {
    	expires 1y;
    	add_header Cache-Control "public, immutable";
	}

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```
</Step>
<Step title="コンテナイメージをビルドする">
環境変数をビルド引数として設定します。

```bash
docker build \
  --build-arg VITE_SUPABASE_URL=<your-url> \
  --build-arg VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
  -t my-zenopus-app .
```
</Step>
<Step title="コンテナー レジストリにプッシュする">
たとえば:

```bash
docker tag my-zenopus-app your-registry/my-zenopus-app:latest
docker push your-registry/my-zenopus-app:latest
```
</Step>
<Step title="オーケストレーション プラットフォームを使用してデプロイする">
Kubernetes、ECS、Cloud Run、Nomad、または内部プラットフォームを使用してコンテナをデプロイします。ランタイム環境変数は必要ありません。それらはすでにビルドに埋め込まれています。
</Step>
</Steps>

**結果:** 実行中のコンテナが、ポート 80 で HTTP 経由でフロントエンドにサービスを提供します。

<Warning>
環境変数はビルド時に埋め込まれます。それらを変更するには、コンテナー イメージを再構築する必要があります。
</Warning>
</Accordion>
<Accordion title="例: 自動化された Docker デプロイメント (CI/CD)">
このワークフローでは、Docker イメージを構築し、それをコンテナー レジストリにプッシュして、Kubernetes、ECS、またはその他のオーケストレーション プラットフォームにデプロイします。これにより、上記で説明した手動のコンテナベースのデプロイメント手順が自動化されます。

<Note>
おそらく、セキュリティ要件、既存のパイプライン、組織のポリシーに合わせて調整する必要があります。
</Note>

認証には、主要なコンテナ レジストリ Support **OpenID Connect (OIDC)** と GitHub Actions を使用します。これにより、長期有効な資格情報をシークレットとして保存する必要がなくなります。これが推奨されるアプローチです。

- [AWS ECR: OIDC で IAM ロールを使用する](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html)
- [Google Artifact Registry: Workload Identity Federation](https://cloud.google.com/iam/docs/workload-identity-federation-with-deployment-pipelines)
- [Azure Container Registry: フェデレーション ID 資格情報](https://learn.microsoft.com/en-us/entra/workload-id/workload-identity-federation)
- [GitHub Container Registry: GITHUB_TOKEN を使用](https://docs.github.com/en/packages/working-with-a-github-packages-registry/working-with-the-container-registry) (シークレットは必要ありません)

**前提条件:**

- コンテナー レジストリ (例: GitHub Container Registry、AWS ECR、Google Artifact Registry、Azure Container Registry、Docker Hub)
- 上記のコンテナベースの手動デプロイメントセクションの `Dockerfile` および `nginx.conf` ファイル
- コンテナをデプロイするためのオーケストレーション プラットフォーム (Kubernetes、ECS など)

<Steps>
<Step title="GitHub シークレットを追加する">
GitHub リポジトリ設定に必要なシークレットを追加します (**設定 → シークレットと変数 → アクション**)。

|秘密 |値 |
| --- | --- |
| VITE_SUPABASE_URL | Supabase プロジェクトの URL (プロジェクトの .env ファイルから) |
| VITE_SUPABASE_PUBLISHABLE_KEY | Supabase 公開可能キー (プロジェクトの .env ファイルから) |
</Step>
<Step title="ワークフローファイルを作成する">
たとえば、リポジトリに `.github/workflows/deploy-container.yml` を作成します。

```yaml
name: Build and Push Container

on:
  push:
    branches: [main]

env:
  IMAGE_NAME: my-zenopus-app
  REGISTRY: ghcr.io/your-org  # Replace with your registry URL

jobs:
  build-and-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # TODO: Add registry authentication step here.
      # This workflow will not run until you configure authentication.
      # See "Registry-specific authentication" table below.

      - name: Build and push Docker image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          build-args: |
            VITE_SUPABASE_URL=${{ secrets.VITE_SUPABASE_URL }}
            VITE_SUPABASE_PUBLISHABLE_KEY=${{ secrets.VITE_SUPABASE_PUBLISHABLE_KEY }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```
</Step>

| AWS ECR | Amazon ECR ログインアクション |
| Google アーティファクト レジストリ | Google 認証アクション |
| Azure コンテナー レジストリ | Azure ログイン アクション |
|ドッカーハブ | Docker ログイン アクション |
</Step>

### VMまたは静的サーバーの展開

<Accordion title="例: 手動 VM または静的サーバーの展開">
このアプローチは、Linux VM、ベアメタル サーバー、または Nginx や Apache などの Web サーバーを実行する任意のマシンにデプロイするためのものです。

<Steps>
<Step title="GitHub リポジトリのクローンを作成します">
```bash
git clone <your-github-repo-url>
cd <project-name>
```
</Step>
<Step title="依存関係をインストールする">
```bash
npm ci
```
</Step>
<Step title="環境変数を使用してアプリケーションをビルドする">
ビルドを実行する前に環境変数を設定します。

```bash
VITE_SUPABASE_URL=<your-url> \
VITE_SUPABASE_PUBLISHABLE_KEY=<your-key> \
npm run build
```

ビルド出力は `dist/` ディレクトリにあります。
</Step>
<Step title="ビルド出力をサーバーにアップロードします">
たとえば、`scp` を使用すると、次のようになります。

```bash
scp -r dist/* user@your-server:/var/www/html/
```

または `rsync` を使用します。

```bash
rsync -avz dist/ user@your-server:/var/www/html/
```
</Step>
<Step title="Webサーバーを設定する">
ZenOpus プロジェクトはクライアント側ルーティング (BrowserRouter を使用した React Router) を使用するため、Web サーバーはすべてのルートに対して `index.html` を返す必要があります。

**nginx 設定の例:**

```nginx
server {
    listen 80;
    server_name your-domain.com;
    root /var/www/html;
    index index.html;

    # Cache static assets
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Handle client-side routing - required for ZenOpus projects
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

**Apache 構成の例 (.htaccess):**

```apache
RewriteEngine On
RewriteBase /
RewriteRule ^index\.html$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /index.html [L]
```
</Step>
<Step title="TLS を構成する (推奨)">
[Certbot](https://certbot.eff.org/) を使用して無料の Let’s Encrypt 証明書を追加するか、既存の SSL 証明書を構成します。
</Step>
</Steps>

**結果:** アプリはサーバーから HTTP(S) 経由で提供されます。

<Warning>
環境変数はビルド出力に組み込まれます。サーバー上でランタイム構成は必要ありません。環境変数を変更するには、アプリケーションを再構築し、`dist/` 出力を再アップロードします。
</Warning>

## マネージド プロバイダー上のホスト バックエンドとデータ (Supabase の例)

通常、このオプションは、完全な運用責任を負わずに、データベースへの直接アクセス、高度なデータベース機能、またはインフラストラクチャ所有権のより明確な分離が必要な場合に選択されます。

バックエンド サービスとデータベースをマネージド バックエンド プロバイダーに移動します。最も直接的な移行パスは、ZenOpus Cloud のデフォルトのバックエンド アーキテクチャとほぼ一致する **マネージド Supabase** です。運用フロントエンドは ZenOpus Cloud またはその他の場所で実行できます。

移行後:

- 運用フロントエンドは新しいバックエンドを指す必要があります
- 開発中に ZenOpus エディターとプレビュー環境を引き続き使用できます。

ZenOpus アプリケーションは Supabase 互換サービス (認証、ストレージ、リアルタイム、エッジ機能、行レベルのセキュリティ) に依存しているため、このガイドでは **Supabase** を参照移行パスとして使用します。

他のバックエンド プラットフォームへの移行は可能ですが、プロバイダーによっては同等の認証、ストレージ、バックエンド サービスの実装が必要になる場合があります。

以下の詳細な手順では、プロジェクトを ZenOpus Cloud からマネージド Supabase インスタンスに移行する方法を説明します。

### あなたが責任を負っていること

バックエンドが ZenOpus Cloud の外部で実行される場合、ZenOpus が以前に管理していた以下のようなバックエンド機能に対して責任を負います。

- データベースの可用性、スケーリング、バックアップ
- バックエンドの監視とインシデント対応
- 行レベルのセキュリティ構成とメンテナンス
- 認証プロバイダーの構成
- OAuth 認証情報、リダイレクト URL、およびシークレットのローテーション
- バックエンドの環境変数と構成
- 設定ミスや漏洩した機密情報のセキュリティ スキャン
- バックエンドインフラストラクチャのコンプライアンス態勢

ZenOpus は、制御していないバックエンド インフラストラクチャを監視またはデバッグできません。マネージド OAuth 構成と自動トークン更新は、バックエンドが ZenOpus Cloud で実行されている場合にのみ使用できます。

### 何がどのように移行するのか

|アプリのコンポーネント |移行方法 |メモ |
| --- | --- | --- |
|データベーススキーマ | SQL 移行による自動 |テーブル、列、インデックス、RLS ポリシー、関数、トリガーが含まれます。
|ストレージバケット | SQL 移行による自動 |アクセス ポリシーを含む |
|認証プロバイダー |マニュアル |新しいホスティング環境で認証 (Google OAuth、GitHub など) を再構成する |
|環境変数とシークレット |マニュアル |新しいホスティング環境で外部サービス (Stripe など) の API キー、トークン、または認証情報を再構成します。
|データ (テーブルの内容) |マニュアル | CSV としてエクスポート/インポート |
|ストレージファイル |マニュアル |手動でダウンロード/アップロードする |
|ユーザーアカウント |手動、部分的 |データベースからユーザー データをエクスポートすることはできますが、ユーザー パスワードをエクスポートすることはできないため、パスワード リセット フローをトリガーする必要があります。実際のユーザーをオンボーディングする前に移行を計画することをお勧めします。 |

## バックエンドとデータを独自のインフラストラクチャ上でホストする (Supabase の例)

このオプションは、厳密なコンプライアンス、データ常駐、またはインフラストラクチャ制御要件を対象としています。

バックエンドとデータベースは、運用するインフラストラクチャ上で実行します。最も直接的なセルフホステッド パスは **セルフホステッド Supabase** で、ZenOpus アプリケーションが依存する認証、ストレージ、リアルタイム、およびエッジ機能のサービスを提供します。

運用フロントエンドは ZenOpus Cloud またはその他の場所に残すことができます。 ZenOpus は引き続き開発に使用できます。あるいは、開発を他のツールに完全に移行することもできます。

同等の認証、ストレージ、リアルタイム、エッジ サービスを実装しない限り、スタンドアロンの PostgreSQL データベースだけを実行するだけでは十分ではありません。

### あなたが責任を負っていること

運用しているインフラストラクチャ上でバックエンドが実行される場合、以前に管理していた次のようなバックエンド機能 ZenOpus に対して責任を負います。

- PostgreSQL の操作、バックアップ、災害復旧
- 認証、ストレージ、リアルタイムサービスの可用性と信頼性
- 行レベルのセキュリティ設計と適用
- セキュリティパッチの適用とバージョンアップの管理
- エッジ機能の導入と実行
- パフォーマンスのチューニングとスケーリング
- モニタリング、アラート、インシデント対応
- インフラストラクチャのコンプライアンス認証

ZenOpus は、セルフホスト型インフラストラクチャのいかなる部分も監視、操作、またはデバッグしません。また、セルフホスト型実稼働環境用の実稼働プレビューは生成されません。

**Supabase を自己ホストする場合は、通常:**

- [Docker を使用した Supabase の公式セルフホスティング ガイド](https://supabase.com/docs/guides/self-hosting/docker) を使用して、独自のインフラストラクチャに Supabase をデプロイします。
    <Note>
    これらの Docker 構成は、ZenOpus エージェントを使用して直接生成できます。以下の [ZenOpus を使用した Docker デプロイメントの生成](#zenopus-を使用した-docker-デプロイメントの生成) を参照してください。
    </Note>
- PostgreSQL、認証、ストレージ、および必要なサービスを構成します。
- ZenOpus プロジェクト (`supabase/migrations/`) からセルフホスト インスタンスに移行ファイルを適用します。
- 自己ホスト型 Supabase を指すようにアプリケーション環境変数を更新します。
    ```text
    VITE_SUPABASE_URL=https://your-self-hosted-domain.com
    VITE_SUPABASE_PUBLISHABLE_KEY=あなたのアノンキー
    ```

インフラストラクチャのセットアップと運用に関する詳細なガイダンスについては、[Supabase の公式セルフホスティング ドキュメント](https://supabase.com/docs/guides/self-hosting) に従ってください。

## ZenOpus を使用した Docker デプロイメントの生成

コンテナ化されたデプロイメントを希望する場合は、prompt ZenOpus を使用して、プロジェクトの Docker および Docker Compose 構成を生成できます。エージェントは、生成されたファイルの詳細、サービス ポートを提供し、チャットでコマンドを実行します。以下にサポートされている 3 つのパターンを示します。

### フロントエンドのみ

React アプリのみを Nginx によって提供される静的サイトとしてパッケージ化します。フロントエンドのみを移動したい場合にこれを使用します。

たとえば:

```text
Nginx を使用したフロントエンドのみのデプロイメント用にこのプロジェクトを Docker 化する
```

### バックエンドのみ (セルフホスト型 Supabase)

フロントエンドをバンドルせずに、完全な Supabase スタックをローカルで実行します。フロントエンドを個別に開発する場合、または別のホストから提供する場合にこれを使用します。

たとえば:

```text
セルフホスト型 Supabase バックエンドのみ (フロントエンド コンテナーなし) 用の Docker Compose セットアップを作成する
```

### フルスタック (フロントエンド + セルフホスト型 Supabase)

フロントエンドとセルフホスト型 Supabase スタックを単一の Docker Compose セットアップにバンドルします。これは、完全に自己完結型の展開に使用します。

たとえば:

```text
Docker Compose にバンドルされた自己ホスト型 Supabase バックエンドを使用してこのプロジェクトを Docker 化します
```

### セルフホスト型 Supabase には手動構成が必要です

エージェントは、使用前に置き換える必要があるプレースホルダー シークレットを生成します。

1. JWT シークレットを生成します: `openssl rand -base64 32`
1. JWT シークレットを使用して API キーを生成します。 [API キーの生成については、Supabase セルフホスティングのドキュメント](https://supabase.com/docs/guides/self-hosting/docker#generate-api-keys) に従ってください。
1. `docker-compose.yml` 内の次の値を置き換えます。
    - __コード0__
    - __コード0__
    - __コード0__
    - __コード0__

### 制限事項

- エージェントは Docker ビルドを実行またはテストできません。ローカルで確認する必要があります。
- エージェントは実際のシークレットを生成できません。プレースホルダーは手動で交換する必要があります。
