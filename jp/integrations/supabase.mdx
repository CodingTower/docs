ZenOpus の **ネイティブ** [Supabase](https://supabase.com/) **統合** により、単一の使いやすいチャット インターフェイスを通じてフロントエンド UI とバックエンド データベースの両方を管理できます。言い換えれば、ZenOpus を離れることなく、アプリの画面をデザインし、クラウド PostgreSQL データベースをセットアップすることができます。この統合されたアプローチにより、誰でも強力なアプリ開発にアクセスできるようになります。技術者以外のユーザーは ZenOpus のガイダンスに頼ることができ、経験豊富な開発者は必要に応じて高度な Supabase 機能を利用できます。

## 概要

<iframe src="https://www.youtube.com/embed/-sSOyO0FiPE?si=WljVA-E0YU4LObMY" title="YouTubeビデオプレーヤー" width="100%" height="315" frameBorder="0" allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture; web-share" referrerPolicy="strict-origin-when-cross-origin" allowFullScreen />

Supabase は Firebase に代わるオープンソースであり、リアルタイム機能、ユーザー認証、ファイル ストレージ、サーバーレス機能を備えたホスト型 PostgreSQL データベースを提供します。 Supabase を ZenOpus アプリに接続すると、ボイラープレート コードを記述したり、手動でサーバーを構成したりすることなく、すぐに本番環境に対応したバックエンドを取得できます。 Supabase の直感的な Web ダッシュボードにより、データとユーザーの管理が簡単になります。また、その堅牢な SQL 基盤により、PostgreSQL データベースの能力とスケーラビリティを最大限に維持できるようになります。

### Supabase の統合によって利用可能になる主な機能:

- **データベース (PostgreSQL)** – 完全な SQL Support を使用してアプリ データを保存およびクエリします。 ZenOpus は、プロンプトに基づいて必要なテーブルとスキーマを自動的に生成できます。
- **ユーザー認証** – ユーザーのサインアップ、ログイン、アクセス制御を安全に管理します。 ZenOpus は、簡単なプロンプトを使用して、事前に構築された認証フロー (電子メール/パスワードなど) をアプリに追加できます。
- **ファイル ストレージ** – Supabase Storage 経由で画像やその他のファイルをアップロードして提供します。ユーザーのプロフィール写真、アップロード、またはアプリで処理する必要がある静的メディアに最適です。
- **リアルタイム更新** – Supabase はライブ データの変更をアプリにストリーミングできます。これにより、すべてのユーザーに対して即座に更新されるライブ チャット、アクティビティ フィード、共同ダッシュボードなどの機能が有効になります。
- **エッジ機能 (サーバーレス)** – Supabase のインフラストラクチャ上でカスタム バックエンド ロジック (JavaScript/TypeScript で) を実行します。 ZenOpus は、電子メールの送信、支払いの処理、外部 API との統合などのタスクのためにこれらの関数を作成してデプロイします。

### ZenOpus の Supabase 統合を使用する理由は何ですか?

ZenOpus を使用すると、フロントエンドの設計とバックエンドのセットアップに別々のツールを使いこなす必要がなくなります。 ZenOpus の AI と会話するだけで、UI を構築でき、*さらに*、基礎となるデータベースとサーバー機能が自動的に作成されます。これは、開発が迅速化され、統合の悩みが軽減されることを意味します。たとえば、prompt ZenOpus で「ユーザー フィードバック フォームを追加し、応答をデータベースに保存する」と指定した場合、ZenOpus はフォーム UI を生成し、** フィードバックを保存するための Supabase テーブルを一度に設定します。このシームレスなエンドツーエンドの生成は ZenOpus の独自の強みであり、初心者が複雑なアプリを構築できるようにし、パワー ユーザーがより迅速に作業を進めることができるようにします。

## ユーザー認証の追加

おそらく最初に必要なものの 1 つはユーザー認証です (ユーザーがアプリにサインアップしてログインできるようにするため)。 ZenOpus の Supabase 統合により、チャット経由でのセットアップが簡単になります。

### メールアドレスとパスワード（基本設定）

プロジェクトが Supabase に接続されると、ZenOpus が認証ページを生成できるようになります。たとえば、**「ログインの追加」** と入力するだけで済みます。これにより通常、アプリ内に基本的なログイン ページ (および関連するサインアップ フロー) が作成され、Supabase の認証システムに接続されます。

ZenOpus でログイン UI を追加した後、テスト用のユーザーを作成する方法がいくつかあります。

<Note>
プロンプトの例:

```text
ログインの追加
```
</Note>

- **アプリのサインアップ フォーム経由:** アプリに新しく追加されたインターフェイスを使用してユーザーを登録します (これにより、Supabase と通信してアカウントが作成されます)。
- **Supabase ダッシュボード経由:** Supabase プロジェクトのダッシュボードに移動し、**認証 > ユーザー** に移動し、新しいユーザー (電子メールとパスワード) を手動で追加します。これは、テスト アカウントをすばやく設定するのに便利です。

<Tip>
開発の便宜を図るため、テスト中は Supabase での電子メール確認を無効にすることをお勧めします。こうすることで、ユーザーは電子メールを確認することなくすぐにログインできます。これを行うには、Supabase ダッシュボードを開き、**認証 > 設定 (またはプロバイダー) > 電子メール** に移動し、**「電子メールの確認」要件を無効にします**。セキュリティのために、後から実稼働環境で確認を再度有効にすることができます。
</Tip>

ZenOpus は、認証済みユーザーを認識するように ZenOpus アプリをすでに構成しています (たとえば、必要に応じて、特定のページをログイン ユーザーのみに制限します)。 ZenOpus のエディターまたはプロンプトを使用して、ログイン/サインアップ ページの外観と操作性をさらにカスタマイズできます。 Supabase Integration 2.0 では、サインアップ フォームにフィールドを追加したり、ブランドに合わせてフォームをスタイル設定したりすることもできます。

### ソーシャルログイン (例: Google)

<iframe src="https://www.youtube.com/embed/W6-_0-Eekio?si=8WkxiJn0oe4wfSbe" title="YouTubeビデオプレーヤー" width="100%" height="315" frameBorder="0" allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture; web-share" referrerPolicy="strict-origin-when-cross-origin" allowFullScreen />

Supabase は、Google、GitHub、Twitter などの OAuth ログインをサポートしています。これらを ZenOpus アプリに統合することもできます。

1. **Supabase でプロバイダーを有効にする:** Supabase ダッシュボードで、**認証 > プロバイダー** に移動します。外部ログインプロバイダーのリストが表示されます。 **Google** に切り替えて (たとえば)、指示に従って Google から必要な OAuth クライアント ID とシークレットを入力します。 *(Supabase では、Google の開発者コンソールからこれらの認証情報を取得する方法について説明します。)* 設定を保存します。これで、Supabase プロジェクトは Google ログインの処理方法を認識しました。
1. **ZenOpus アプリの UI を更新します。** 次に、prompt ZenOpus してソーシャル ログイン オプションを追加できます。例: **「ログイン ページに『Google でサインイン』ボタンを追加します。」** ZenOpus は認証ページを変更し、Google サインイン ボタンと、Supabase 経由で OAuth フローを開始するために必要なコードを追加します。

これが完了すると、アプリのユーザーは [Google でサインイン] をクリックし、認証のために Google にリダイレクトされ、ログイン ユーザーとしてアプリに戻ることができるようになります。他のプロバイダー (GitHub、Facebook など) も同様の方法で有効にすることができます。忘れずに Supabase でそれぞれを構成し、それに応じて ZenOpus UI を調整してください。

## Supabase を使用したデータ管理

認証を超えて、ほとんどのアプリの中核はデータの読み取りと書き込みです。 ZenOpus と Supabase を使用すると、チャット プロンプトを離れることなくデータベース テーブルを作成し、それらを UI に接続できます。

### ZenOpus によるデータベース テーブルの作成

Supabase に接続した後、データを永続的に保存する必要がある場合は、ZenOpus を指示すると、データベースのセットアップが調整されます。通常、プロセスは次のようになります。

<Steps>
<Step title="必要な機能とデータについて説明する">
ZenOpus のチャットで、データ ストレージに関連する機能をリクエストします。たとえば、次のように言うことができます: **「フィードバック フォームを追加し、回答をデータベース テーブルに保存します。」** どのようなデータを保存するかをできるだけ明確にしてください (例: メッセージと評価を含むユーザー フィードバック、または名前と価格などを含む在庫アイテム)。 ZenOpus は、この情報を使用して、必要なデータベース構造を決定します。
</Step>
<Step title="生成された SQL スニペットを確認する">
ZenOpus の AI がアプリのアップデートを提案します。リクエストにはデータの保存が含まれるため、ZenOpus は SQL スキーマ スニペ​​ット、つまり、Supabase に必要なテーブルまたは列を作成するためのコマンドも生成します。たとえば、「`id`*、*`message`*、および *`rating`* という列を持つテーブル *`feedback`* を作成します」* とその SQL コードを提供します。このスニペットはチャットまたはサイドバーに表示され、コピーできます。
</Step>
<Step title="SupabaseでSQLを実行する">
Supabase ダッシュボードを開き、**SQL エディタ**に移動します。 ZenOpus から SQL スニペットを貼り付けて実行すると、データベースに新しいテーブルが作成されます (または既存のテーブルが変更されます)。 Supabase **テーブル エディタ** で、指定した列を持つテーブルが存在することを確認できます。
</Step>
<Step title="確認して ZenOpus による統合を完了させます">
ZenOpus に戻り、SQL を実行したことを確認します (チャットで確認を求められる場合もありますが、単純に ZenOpus に「完了」と伝えるか、表示される確認をクリックすることもできます)。 ZenOpus は統合を完了します。アプリの UI を更新して、フォーム入力、リスト、またはその他のコンポーネントを新しい Supabase テーブルにバインドします。例を続けると、アプリのフィードバック フォームが接続されました。ユーザーがフィードバックを送信すると、フィードバックは `feedback` テーブルに保存され、それらのエントリを取得して表示することもできます。
</Step>
</Steps>

それです\!データベース スキーマを手動で設計したり、バックエンド コードを記述したりする必要はありません。ZenOpus と、Supabase が説明に基づいてそれを処理しました。新しいデータ機能 (ブログ投稿、コメント、製品など) に対してこのプロセスを繰り返すことができます。

たとえば、ユーザーがアプリに記事を投稿できるようにしたい場合は、次のようにプロンプトを表示できます。**「ユーザーがタイトルとコンテンツを含む投稿を作成し、データベースに保存できるようにします。」** ZenOpus は `posts` テーブル (タイトル、コンテンツ、作成者、タイムスタンプなどのフィールドを含む) を生成し、それを Supabase に追加する SQL を提供し、フロントエンド フォームとページをそのテーブルに接続します。

<Note>
Supabase のダッシュボードは、データを管理するための豊富なインターフェイスを提供します。スプレッドシートのような UI でテーブルの行を表示および編集したり、テーブル間の関係を定義したり、CSV や Excel からデータをインポートしたりすることもできます。内部ではすべて PostgreSQL です。つまり、必要に応じて複雑なクエリを実行したり、SQL 機能を使用したりできます。 Supabase は、SQL に詳しくない場合でもクエリを作成できるように、SQL エディターに **AI SQL Assistant** を提供しています。これは、高度な分析やトラブルシューティングに便利です。
</Note>

### ファイルストレージ (画像とメディア)

ZenOpus アプリでファイルのアップロード (ユーザー プロフィール写真、添付ファイル、メディアなど) を処理する必要がある場合は、Supabase 統合が対応します。 Supabase には、ファイル (画像、ビデオ、PDF など) をデータベースと一緒に便利にホストするための **ストレージ** サービスが含まれています。

ZenOpus アプリに **アップロード** コンポーネントまたは画像アップロード機能を追加すると、ZenOpus はバックグラウンドで Supabase Storage を利用します。アップロードされたファイルは、Supabase プロジェクト内のストレージ バケットに保存され、後でファイルを表示またはダウンロードするために使用する URL または参照を取得します。

デフォルトでは、Supabase の無料利用枠では、それぞれ最大 **50MB** のファイルをアップロードできます。ほとんどの画像や短いビデオにはこれで十分です。アプリでより大きなファイルを処理する必要がある場合、Supabase の有料プランでは Support より大きなアップロードが可能です (非常に大きなファイルの **再開可能なアップロード** を含む)。必要に応じて、ファイルをフォルダー (バケット) に整理し、Supabase ダッシュボードを介してアクセス許可を管理できます。

<Note>
アプリにユーザー プロフィールがあり、ユーザーにプロフィール写真を追加してもらいたいとします。 prompt ZenOpus: *「アカウント設定ページにプロフィール写真のアップロードを追加する」* ZenOpus は、画像をアップロードするための UI を作成します。 Supabase 統合のおかげで、ユーザーがファイルをアップロードすると、そのファイルは Supabase ストレージ (バケット、例: `public/avatar-images`) に保存され、ZenOpus がその画像 URL の取得を処理して、アプリにプロフィール写真を表示します。これらすべては、ストレージ処理コードを作成しなくても行われます。
</Note>

## シークレットの保存 (API キーと構成)

多くのアプリケーションでは、決済用の Stripe や AI 機能用の OpenAI など、サードパーティ サービスに接続するために秘密キーまたは API 認証情報が必要です。 Supabase が接続されている場合、ZenOpus はこれらのシークレットを安全に管理および使用する方法を提供します。

ZenOpus は、機能にシークレットが必要な場合に自動的に検出し、必要な値の入力を求める UI を表示します。

これらのシークレットは、プロジェクトの Supabase の Edge Function シークレット マネージャーに安全に保存されます。これらは暗号化され、バックエンドで安全に保管されます。 Edge Functions (次のセクションを参照) を展開すると、これらのシークレットにアクセスして外部サービスに接続できます。

<Tip>
たとえば、Stripe を支払いに統合する場合は、Stripe 秘密キーをシークレットとして保存します。 ZenOpus が支払い処理関数を Supabase にデプロイすると、関数が Stripe で認証できるように、シークレット ストアからそのキーが自動的に組み込まれます。こうすることで、シークレットをアプリにハードコーディングする必要がなく、シークレットが公に公開されることを避けられます。
</Tip>

## エッジ機能を使用したバックエンド ロジック

<iframe src="https://www.youtube.com/embed/p_hKIDlxXp0" title="YouTubeビデオプレーヤー" width="100%" height="315" frameBorder="0" allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture; web-share" referrerPolicy="strict-origin-when-cross-origin" allowFullScreen />

アプリでは、基本的なデータ CRUD (作成、読み取り、更新、削除) を超えたカスタム バックエンド ロジックが必要になる場合があります。 **Supabase Edge Functions** は、イベントまたはリクエストによってトリガーされたバックエンドでコードを実行できるサーバーレス関数 (AWS Lambda に似ています) です。 ZenOpus の統合は、必要なバックエンドの動作を平易な言語で定義できることを意味し、ZenOpus は必要なエッジ関数コードを作成して Supabase にデプロイします。

ZenOpus のエッジ関数の一般的な使用例は次のとおりです。

- **AI サービスの使用:** 例: OpenAI または Anthropic API を使用して一部の入力を処理します (上記のように、API キーはシークレットとして保存されます)。
- **電子メールまたは通知の送信:** 例:ユーザーがサインアップするときに、Resend などの電子メール API を介してウェルカム電子メールを送信します。
- **支払いの処理:** 例: Stripe の API を使用してチェックアウト セッションを作成したり注文を履行したりする。
- **スケジュールされたタスク:** 例:クリーンアップまたはサマリー ジョブを 1 時間/日ごとに実行します (Supabase Edge 機能はスケジュールに基づいてトリガーできます)。
- **複雑な計算またはサードパーティの統合:** クライアント側で実行したくないコードはすべて、Edge Function で実行できます。

バックエンド関数を追加するには、ZenOpus チャットで必要なものを説明するだけです。例: *「ユーザーがフィードバック フォームを送信すると、OpenAI を使用してテキストを分析し、センチメント スコアを保存します。」* ZenOpus は、このロジックのコードを Supabase Edge 関数として生成し (この場合、OpenAI API を呼び出します)、それを Supabase プロジェクトにデプロイします。また、適切なタイミング (フォーム送信時など) でこの関数を呼び出し、応答を処理するように ZenOpus アプリも更新されます。

Edge 関数は、Supabase ダッシュボードの **関数** で見つけて監視できます。各関数には、最近の実行と出力またはエラーを示すログがあります。 ZenOpus の Supabase Integration 2.0 は、何か問題が発生したときにこれらのログを自動的に読み取ることで、このエクスペリエンスを向上させます。関数でエラーが発生した場合、ZenOpus はトラブルシューティングに役立つエラー メッセージをチャットに表示します。もちろん、詳細を確認したり、安心するために、いつでも Supabase のログをご自身で確認することができます。

<Warning>
**ライブを開始する前に:** Supabase のデフォルトのセキュリティ ルールは開発には寛容ですが、運用環境でデータを保護するには **行レベル セキュリティ (RLS)** ポリシーを設定する必要があります。 RLS を使用すると、データベース テーブルの各行を読み書きできるユーザーを定義できます (たとえば、ユーザーが自分のデータのみにアクセスできるようにするなど)。 ZenOpus は、プロンプトを表示すると、基本的な RLS ポリシーの生成を支援します (たとえば、「ユーザーが自分のフィードバックのみを編集できるようにセキュリティ ポリシーを適用する」)。ただし、これらのポリシーは、Supabase ダッシュボードの **[認証] > [ポリシー]** で必ず確認してテストしてください。実際のユーザーをアプリに招待する前に、適切なセキュリティ設定を行うことが重要です。
</Warning>

### よくある質問

<AccordionGroup>
<Accordion title="Supabase の統合は実際に何をするのですか?">
これにより、ZenOpus アプリにフルマネージドのバックエンドが提供されます。これがなければ、ZenOpus で UI を構築することはできますが、すぐにデータを保存したりユーザーを管理したりする場所がなくなります。 Supabase に接続すると、ZenOpus はユーザー アカウントの作成 (認証)、データベースへのデータの保存と取得、ファイルのアップロード、サーバー側のコードの実行などをすべて自動的に行うことができます。基本的に、Supabase はアプリの背後にあるデータベースとサーバーを提供し、ZenOpus がプロンプトを通じてアプリを駆動します。
</Accordion>
<Accordion title="ZenOpus と Supabase には別のアカウントが必要ですか?">
はい。 ZenOpus と Supabase は 2 つの別個のプラットフォームです。 ZenOpus アカウントに加えて、(データベースをホストするための) Supabase のアカウントが必要になります。良いニュースは、どちらにも無料枠があるため、費用をかけずに開始できることです。後で使用量や機能を増やすためにアップグレードする場合は、各サービスの請求を個別に処理することになることに注意してください。
</Accordion>
<Accordion title="ZenOpus を Supabase に接続するにはどうすればよいですか?">
1. ZenOpus エディターで、**統合** セクションに移動します。
1. [**Supabase に接続**] をクリックし、認証手順に従います。
1. 必要に応じて、ZenOpus 内に新しい Supabase プロジェクトを作成します。
1. ZenOpus は必要なデータベース スキーマを自動的に生成し、プロジェクトに接続します。
</Accordion>
<Accordion title="Supabase に接続した ZenOpus アプリを外部自動化ツールと統合できますか?">
絶対に。 ZenOpus + Supabase を使用すると、データは Supabase のデータベースに存在し、Supabase はテーブル (およびクライアント ライブラリ) 用に自動生成された RESTful API も提供します。これは、Zapier、[Make.com](http://Make.com) などのツール、またはその他のサービスを使用して、HTTP リクエストを介してアプリのバックエンド データを操作できることを意味します。たとえば、Zapier はレコードを取得したり、アプリの Supabase テーブルにレコードを追加したりできます。さらに、Supabase Edge Functions (ZenOpus が作成に役立ちます) を使用してカスタム API エンドポイントを作成し、より複雑なワークフローをトリガーすることができます。つまり、サードパーティの自動化サービスとの統合は可能です。これには、Supabase の API キーまたは Webhook を使用した少しの構成が必要になるだけです。
</Accordion>
<Accordion title="アプリが成長した場合、Supabase はどの程度拡張可能ですか?">
Supabase は **PostgreSQL** 上に構築されており、大量のデータと高トラフィックを処理できます。無料のデータベースは、そのまま使用しても、適切なワークロード (数百万行、複数の接続) を処理できます。ニーズが増大するにつれて、Supabase プランをアップグレードして、より多くのストレージ、スループット、機能を得ることができます。多くの実稼働アプリは完全に Supabase 上で実行されるため、安心してご利用いただけます。無料利用枠の使用制限 (Supabase のサイトで説明されています) に注意し、制限に近づいた場合はスケールアップする計画を立ててください。
</Accordion>
<Accordion title="チャットやライブフィードなどのリアルタイム機能をアプリに追加するにはどうすればよいですか?">
Supabase には、データベースにリアルタイム サブスクリプションが組み込まれています。これは、アプリが特定のテーブルの変更 (挿入、更新、削除) をリッスンし、即座に反応できることを意味します。これを活用するには、通常どおり ZenOpus アプリ機能 (`messages` テーブルにメッセージを書き込むチャット ルームなど) を設計します。 ZenOpus は Supabase のリアルタイム機能を認識しているため、そのテーブルの変更をサブスクライブするようにフロントエンドをセットアップできます。実際には、チャット メッセージなどのテーブルを作成した後、prompt ZenOpus して **「チャットのリアルタイム更新を有効にする」**ことができます。これにより、内部で Supabase のリアルタイム API が使用されます。ユーザーは更新しなくても、新しいメッセージがライブで表示されるようになります。これは、ライブ更新が役立つあらゆるシナリオ (コメント、通知、ダッシュボードなど) で機能します。
</Accordion>
<Accordion title="Supabase ストレージでのファイルのアップロードに制限はありますか?">
無料枠では、Supabase は個々のファイルのアップロード サイズを **50MB** に制限します。これは、画像、音声、短いビデオなどのほとんどのユースケースをカバーします。より大きなファイル (長いビデオや大規模なデータセットなど) を処理する必要がある場合は、有料プランにアップグレードすると、より大きなアップロードが可能になり、再開も可能になります。また、プロジェクトごとの全体的なストレージ割り当て (保存できる合計 GB 数など) があり、プランが高くなるほど増加することにも注意してください。
</Accordion>
<Accordion title="Supabase で認証を構成するにはどうすればよいですか?">
ZenOpus は認証を自動的にセットアップしますが、次の操作が必要になる場合があります。

- **[Supabase ダッシュボード] > [認証]** に移動します。
- **電子メール サインイン/サインアップ**を有効にします。
- ローカルテストを容易にするために電子メール確認を無効にします。
</Accordion>
<Accordion title="1 つの Supabase データベースを複数の ZenOpus プロジェクトに使用できますか?">
はい、できます。 ZenOpus で複数のフロントエンド アプリケーションを構築し、すべてが同じ Supabase プロジェクトに接続する (つまり、同じデータベースと認証を共有する) 場合があります。これは高度ですが、可能です。たとえば、メイン アプリと管理ダッシュボードを、1 つの共通データベースを使用して別個の ZenOpus プロジェクトとして使用できます。各プロジェクトで Supabase を接続する場合は、同じ Supabase プロジェクトを選択するだけです。これらのアプリはすべて同じデータの読み取り/書き込みを行うため、それに応じて設計する必要があることに注意してください。
</Accordion>
<Accordion title="ライブアプリに影響を与えずにデータベースへの変更をテストしたい場合はどうすればよいですか?">
現時点では、各 ZenOpus プロジェクトは 1 つの Supabase プロジェクトに接続されており、ZenOpus には組み込みのステージング モードがありません。安全に実験できるプレイグラウンドが必要な場合、Supabase は **ブランチング** と呼ばれる機能を提供します。これにより、変更をテストするためにデータベースの一時コピー (git ブランチなど) を作成できます。テストのために、別の ZenOpus プロジェクトをデータベースのブランチまたは複製に接続できます。一般に、本格的なプロジェクトの場合、ライブ アプリでスキーマを変更する場合は注意が必要です。おそらく、バックアップを作成するか分岐を使用し、準備ができたら変更をマージします。 ZenOpus の統合は進化しているため、将来の更新では、よりシームレスなステージング ワークフローが導入される可能性があります。
</Accordion>
<Accordion title="Supabase または ZenOpus はカスタム SQL またはデータベース ロジックの作成に役立ちますか?">
はい。 Supabase の Web インターフェイスには、自然言語から SQL クエリを生成できる **AI SQL Assistant** が含まれています。したがって、複雑なクエリが必要な場合、または SQL の作成に慣れていない場合は、Supabase SQL エディターでそのツールを試すことができます。 ZenOpus 側では、通常、SQL を手書きする必要はまったくありません。スキーマの作成とクエリのほとんどは AI が処理します。ただし、上級ユーザーで何かカスタムをしたい場合は、ZenOpus の GitHub 統合をいつでも使用して、コードを検査または編集したり、必要に応じてデータベース上で生の SQL を実行したりできます。
</Accordion>
<Accordion title="ZenOpus アプリで支払いを処理するにはどうすればよいですか?">
支払いは、Stripe などの統合を介して処理され、Supabase Edge Functions と組み合わせて使用できます。たとえば、prompt ZenOpus: 「チェックアウト ボタンを追加し、Stripe で支払いを処理する」とします。次に、ZenOpus は、Stripe の API と通信する Edge 関数を作成し (シークレットとして保存されている Stripe 秘密キーを使用)、おそらくトランザクションの詳細を Supabase データベースに保存します。 UI はチェックアウト/支払いボタンを含むように更新されます。基本的に、Supabase は支払いを実装するための環境 (エッジ機能、データベース) を提供し、ZenOpus がコードのスキャフォールディングを行います。詳細については、ZenOpus のドキュメントにある Stripe と Payments の統合ガイドを参照してください。このガイドでは、特に Stripe の設定について説明しています。
</Accordion>
</AccordionGroup>
