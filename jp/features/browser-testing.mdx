ブラウザー テストでは、ZenOpus が仮想環境で実行されている実際のブラウザーでアプリと対話できるようになります。エージェントは、コードのみに依存するのではなく、ボタンをクリックし、フォームに入力し、ページを移動し、スクリーンショットを使用して実際のユーザーの行動を検証できます。

ブラウザーのテストを使用して、ユーザー フローを検証し、視覚的または動作の問題をデバッグし、さまざまな画面サイズで変更が期待どおりに機能することを確認します。

## ブラウザテストでできること

ブラウザーのテストを実行すると、エージェントは次のことができます。

- アプリのスクリーンショットをキャプチャする
- ボタンとリンクをクリックします
- 入力してフォームを送信する
- ページ間を移動する
- コンソールログとネットワークリクエストを読み取る
- アプリの操作中にランタイムエラーを検出する
- モバイル、タブレット、デスクトップのレイアウトなど、さまざまな画面サイズをテストします。デフォルトでは、プレビューと同じサイズが使用されます。

## ブラウザテストの仕組み

ZenOpus は、最初に高速な検証方法を試した後、ブラウザー テストを選択的に使用します。

ブラウザ テストは、次の場合に実行されることがあります。

- 何かが動作することを確認することを明示的に要求します。一般的なトリガー フレーズは次のとおりです。  
「動作することを確認する」、「動作することを確認する」、「これをテストする」、「動作するかどうかを確認する」
- ログやコード検査ではビジュアルまたは UI の問題を特定できない
- より大きな機能変更があり、ZenOpus がフローのテストを提案し、確認します。

<Tip>
同じプロンプト内で ZenOpus に *大きな変更を加えてテスト* するよう依頼することは避けてください。

ブラウザーのテストが行き詰まって停止すると、そのステップ中に行われた作業がすべて失われる可能性があります。

より安全なフローは、最初にビルドしてから、フォローアップ プロンプトでテストすることです。
</Tip>

### どこで実行されるか

ブラウザは安全な仮想環境でリモートで実行されます。これは個人のブラウザではなく、ローカル セッションを引き継ぎません。

- ブラウザ テストは常に、現在作業しているプロジェクトのプレビューに対して実行され、表示しているアプリのバージョンを正確にテストします。
- アプリが認証を使用する場合、ブラウザーのテストは、プレビューで現在ログインしているのと同じアプリ ユーザーを使用して実行されます。クリックしたりトリガーしたりしてはいけないアクションや領域がある場合は、明示的に指定してください。
- 一部の ZenOpus Cloud アプリは [テスト環境とライブ環境](/jp/features/environments) 機能を使用します。この機能を有効にすると、**テスト** 環境でプロジェクト プレビューに対してブラウザ テストが実行されます。

### ブラウザのテスト実行中に何が起こるか

専用の **詳細ビュー** でテスト プロセスを追跡できます。エージェントは次のことを行います。

- ボタンのクリックやページの移動など、実行中のステップを表示します。
- アプリのスクリーンショットをキャプチャします
- URLとインタラクションの結果を表示します
- 何がうまくいったのか、何がうまくいかなかったのかをまとめます

ブラウザ テストは、エージェントが実際のブラウザと対話して結果を確認するため、通常のアクションよりも時間がかかります。

## 制限事項

- キャンバスベースのツールや描画ツールと対話することはできません
- ファイルのアップロードまたはダウンロードができない
- 右クリック操作は実行できません
- ドラッグ アンド ドロップ操作を実行することもありますが、クリックよりも信頼性が低くなります。
- クリップボード操作やテキスト選択を実行する可能性がありますが、これらは信頼できません。
- 微妙な視覚的デザインの詳細や色の違いを評価するには信頼性がありません
- テキストラベルのボタンと比較して、アイコンのみのボタンの操作が難しい場合があります

## よくある質問

<AccordionGroup>
<Accordion title="ブラウザのテストは自動的に実行されますか?">
はい、場合によっては。 ZenOpus は、検証が必要な場合にブラウザ テストを自動的に提案または使用することがありますが、デフォルトでは保守的であり、最初に軽量な方法を試します。
</Accordion>
<Accordion title="ZenOpus がブラウザ テストを使用していることを確認するにはどうすればよいですか?">
いつでもブラウザのテストを明示的に要求できます。 一般的なトリガー フレーズには、「動作することを確認してください」、「動作していることを確認してください」、「これをテストしてください」、「動作しているかどうかを確認してください」などがあります。

たとえば:

```text
ブラウザーのテストを使用して、このフローを検証します。
```
</Accordion>
<Accordion title="ブラウザのテストはどの環境で実行されますか?">
ブラウザ テストは常に、現在作業しているプロジェクトのプレビューに対して実行されます。表示しているアプリのバージョンを正確にテストします。

プロジェクトで **テスト環境とライブ環境の分離** を使用している場合、ブラウザ テストはテスト環境のプレビューに対して実行されます。
</Accordion>
<Accordion title="ブラウザのテストではアプリの権限が使用されますか?">
はい。アプリが認証を使用する場合、エージェントは現在ログインしているのと同じアプリ ユーザーとしてログインし、ブラウザのテストは同じアプリ ユーザーを使用して実行されます。

何かをクリックしたりトリガーしたりしてはいけない場合は、明示的に示してください。
</Accordion>
<Accordion title="これは私の本当のブラウザですか?">
いいえ。ブラウザのテストでは、リモートの仮想環境で実行されている実際のブラウザを使用します。ローカルのブラウザやセッションを引き継ぐことはありません。
</Accordion>
<Accordion title="ブラウザーのテストで同じアクションが失敗し続ける場合はどうなりますか?">
エージェントが同じ要素の操作に繰り返し失敗する場合は、ブラウザの操作制限（おそらく）、または非表示要素、壊れた状態、予期しないレイアウトなどのアプリの根本的な問題が原因である可能性があります。

次のことを試してください。

1. 詳細ビューでテスト実行のスクリーンショットとログを確認します。
1. ZenOpus に要素を操作しやすくするよう依頼します (たとえば、表示テキストやアクセシビリティ ラベルを追加するなど)。
1. 問題が解決しない場合は、その要素との対話を避けるようにエージェントに明示的に指示してください
</Accordion>
</AccordionGroup>
