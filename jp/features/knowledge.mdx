知識を利用すると、ZenOpus エージェントに永続的な指示とコンテキストを提供できます。すべての会話で同じ説明を繰り返す代わりに、それらを一度定義すると、ZenOpus は編集を生成するときにそれらを考慮します。

知識は 2 つのレベルで定義されます。

- **Workspace ナレッジ** は、コーディング標準、優先ライブラリ、命名規則など、Workspace 内のすべてのプロジェクトに適用される共有ルールを定義します。
- **プロジェクトの知識** により、アプリケーションの目的、データベース スキーマ、アーキテクチャの決定、ドメイン用語など、単一のプロジェクトに固有のコンテキストが追加されます。

どちらも、ZenOpus エージェントに永続的な指示を提供するプレーン テキスト フィールドです。

## Workspace の知識

Workspace ナレッジは、共有ルールを一度定義して、Workspace 内のすべてのプロジェクトに適用できるテキスト フィールドです。プロジェクトの知識により、特定のプロジェクトに固有のコンテキストを追加できます。

**複数のプロジェクト間で一貫性が必要なルールや規約**に使用するのが最適です。これらのガイドラインを一度定義すると、すべてのプロジェクトの知識フィールドで同じ指示を繰り返す必要がなくなります。

Workspace の知識は、複数の人がプロジェクトを構築しているチーム環境で特に役立ちます。 Workspace 管理者は、コーディング標準、テスト要件、アーキテクチャ ルールなどのガードレールを定義できるため、すべてのプロジェクトが同じ規則に自動的に従うようになります。

**Workspace の所有者と管理者** のみが Workspace のナレッジを管理できます。 Workspace ナレッジを管理するには、次のいずれかに進みます。

- **設定 → ナレッジ**
- **プロジェクト設定 → ナレッジ**

Workspace ナレッジは最大 **10,000 文字**をサポートします。

### 何を含めるか

Workspace の知識は次の場合に最適です。

- コーディングスタイルの規約
- 命名規則
- 推奨されるライブラリまたはフレームワーク
- 共有されたアーキテクチャ パターン
- テスト要件
- コードの品質またはリンティングのルール
- 言語または書式設定の設定
- ブランドの声、UIコピー、またはデザインガイドライン
- ZenOpus が避けるべきこと

### 例

```text
コーディング標準
- TypeScript 厳密モードを常に有効にします。
- `any` は決して使用しないでください。 `unknown` を使用して型を絞り込みます。
- 名前付きエクスポートを優先します。デフォルトのエクスポートは使用しないでください。
- `let` よりも `const` を優先します。 `var` は決して使用しないでください。

命名規則
- 変数と関数にはキャメルケースを使用します。
- コンポーネントと型には PascalCase を使用します。
- ファイル名にはkebab-caseを使用してください。

スタイリング
- スタイル設定には Tailwind CSS を使用します。
- インラインスタイルは使用しないでください。
- CSS モジュールは使用しないでください。

図書館
- 可能な場合は、shadcn/ui コンポーネントを使用してください。
- サーバーの状態には React Query を使用します。
- クライアント状態には Zustand を使用します。

建築
- API 呼び出しをサービス層経由でルーティングします。
- React コンポーネントから `fetch` を直接呼び出さないでください。

テスト
- 新しいユーティリティ関数とフックの単体テストを作成します。
- 変更を加えた後、既存のテストを実行します。
- 完了とマークする前に、ブラウザで新しい機能を確認してください。

コードの品質
- 大幅な変更を行った後、リンターを実行します。
- 未使用のインポートとデッドコードを削除します。

ローカリゼーション
- コードのコメントや変数名は英語で記述してください。
- 日付形式 DD/MM/YYYY を使用します。
- ヨーロッパの数値形式 (小数点区切り文字としてカンマ) を使用します。

ブランドボイス
- ユーザー向けのすべてのコピーでは、フレンドリーでプロフェッショナルなトーンを使用します。
- 見出しとボタンには大文字と小文字を使用します (たとえば、「新しいプロジェクトの作成」ではなく、「新しいプロジェクトの作成」)。
- 「Lorem ipsum」のようなプレースホルダー テキストを使用せず、常に現実的なコピーを作成してください。

一般規則
- `console.log` ステートメントを追加しないでください。
- 非推奨の React パターンは使用しないでください。
- コードのコメントは英語で書きます。
```

### 重要な注意事項

- **変更はすぐに適用されます**  
アクティブな会話中に Workspace の知識を更新すると、ZenOpus は更新された指示をフォローアップ メッセージに使用します。
- **Workspace ごとに 1 つの Workspace 知識**  
各 Workspace には、すべてのプロジェクトで共有される単一の Workspace ナレッジがあります。同じ Workspace 内のプロジェクトのサブセットに対して、異なる Workspace レベルの命令を定義することはできません。
- **長い会話における__Workspaceの知識**  
Workspace の知識は、プロジェクト コードやその他のコンテキスト ソースとともにプロジェクトの知識とともに常に含まれます。ただし、多くのコンテキストを伴う非常に長い会話では、指示が常に一貫して従うとは限りません。

## プロジェクトの知識

プロジェクト ナレッジは、単一プロジェクトの永続的な指示とコンテキストを保存するテキスト フィールドです。これを使用して、アプリケーションの目的、アーキテクチャ、ドメイン用語など、その特定のプロジェクトにのみ適用される情報を提供します。

プロジェクトを編集する権限を持つ人は誰でも、プロジェクトの知識を更新できます。プロジェクトのナレッジを管理するには、**プロジェクト設定 → ナレッジ** に移動します。

プロジェクトのナレッジは最大 **10,000 文字**までサポートされます。

### 何を含めるか

プロジェクトに関する優れた知識には通常、次のものが含まれます。

- アプリケーションの機能
- ユーザーペルソナまたは対象ユーザー
- データベーススキーマまたはキーテーブル
- アーキテクチャに関する決定
- ドメイン用語
- プロジェクト固有の制約
- デザインガイドライン（色、タイポグラフィ、レイアウト）
- API ドキュメントや内部ツールなどの重要なリファレンスへのリンク
- セキュリティまたはコンプライアンスの要件

### 例

```text
プロジェクト概要
これは、レストランのマネージャーが複数の場所にわたる食品在庫を追跡するための B2B SaaS アプリケーションです。

ユーザー
主なユーザーは、在庫レベルと食材の使用状況を迅速に把握する必要があるレストランのマネージャーです。
二次ユーザーは、在庫変更の記録を担当するスタッフです。

主要なデータベーステーブル
- inventory_items (id、名前、カテゴリ、数量、単位、location_id)
- 場所 (ID、名前、workspace_id)
- トランザクション (id、item_id、quantity_change、type、created_at)

設計ガイドライン
- スタイル設定には Tailwind CSS を使用します。
- 既存のカラー パレットと間隔スケールに従います。
- ページ全体で一貫した間隔とレイアウト パターンを使用します。
- 利用可能な場合は shadcn/ui コンポーネントを優先します。

アーキテクチャのルール
- 通貨値をセント単位で整数として保存します。
- すべての突然変異に対して楽観的な更新を使用します。
- 再利用可能なコンポーネントを /components に配置します。

ドメイン用語
- 「在庫品目」とは、追跡される原材料または製品を指します。
- 「トランザクション」は在庫数量の変化を表します。

外部参照
- API ドキュメント: https://docs.example.com/api
```

**Workspace の知識**を使用して、コーディング標準、優先ライブラリ、アーキテクチャ上の規則、および複数のプロジェクトに適用する必要があるその他のルールを定義します。

次に、**プロジェクトの知識**を使用して、アプリケーションの目的、データベース スキーマ、ドメイン用語、統合など、各プロジェクトに固有のコンテキストを追加します。

明確な指示がより良い結果を生み出します。

良い例:

```text
TypeScript 厳密モードを常に有効にします。決して使用しないでください。
```

あまり役に立たない例:

```text
きれいなコードを書きましょう。
```

</Accordion>
<Accordion title="オンボーディングドキュメントのように書く">
チームに加わった新しい開発者にプロジェクトについて説明するところを想像してください。 ZenOpus がそれらを変更しようとしないように、重要なアーキテクチャ上の決定を含めます。
</Accordion>
<Accordion title="指示は簡潔にしてください">
長い段落よりも短い指示の方が効果的です。箇条書きリストと直接的なルールを好みます。
</Accordion>
<Accordion title="知識を定期的に復習する">
スタックやアーキテクチャが変更された場合は、ZenOpus が古いパターンに従わないように知識を更新してください。

すべてを文書化する必要はありません。いくつかの明確なルールや短いプロジェクトの説明でも、編集の品質を大幅に向上させることができます。
</Accordion>
</AccordionGroup>

## よくある質問

<AccordionGroup>
<Accordion title="知識の文字数制限は何ですか?">
プロジェクトの知識と Workspace の知識 Support は両方とも 10,000 文字まで。
</Accordion>
<Accordion title="Workspace ナレッジ命令のセットを複数持つことはできますか?">
いいえ。各 Workspace には、すべてのプロジェクトで共有される Workspace ナレッジが 1 つあります。
</Accordion>
<Accordion title="プロジェクトのリポジトリに指示ファイルを追加できますか?">
はい。 `AGENTS.md` や `CLAUDE.md` などの命令ファイルも、ZenOpus エージェントにガイダンスを提供できます。

リポジトリで命令ファイルを直接管理している技術ユーザーの場合、セッションの長さに関係なく、ルートレベルの `AGENTS.md` ファイルが常に ZenOpus エージェントによって読み取られます。
</Accordion>
<Accordion title="ZenOpus はどのコンテキスト ソースを読み取りますか?">
メッセージを送信すると、ZenOpus は **プロジェクトの知識**、**Workspace の知識**、**プロジェクト コード**を読み取り、編集を生成する前にプロジェクトがどのように機能するかを理解します。

また、接続されているサービスや、`AGENTS.md` や `CLAUDE.md` など、プロジェクトの GitHub リポジトリ内の指示ファイルからの統合ナレッジも調べます。
</Accordion>
<Accordion title="プロジェクトと Workspace の知識が競合した場合はどうなりますか?">
ZenOpus が応答を生成するときに、プロジェクトの知識と Workspace の知識が一緒に含まれます。

命令が競合する場合、ZenOpus は **プロジェクトの知識** で定義された命令を優先することをお勧めします。これらの命令は現在のプロジェクトに特に適用されるためです。

混乱を避けるために、共有ルールは Workspace ナレッジに、プロジェクト固有の詳細はプロジェクト ナレッジに保管してください。
</Accordion>
</AccordionGroup>
