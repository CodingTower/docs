<Info>
ZenOpus デザイン システムは **エンタープライズ プラン** で利用でき、React コンポーネントとして実装されたデザイン システムとネイティブに動作します。
</Info>

**デザイン システム** を使用すると、コンポーネント ライブラリ、スタイル ガイドライン、およびインストール手順を一度定義し、Workspace 内のすべてのプロジェクトで再利用できます。プロジェクトの開始時にファイルをコピーするテンプレートとは異なり、デザイン システムは、ZenOpus が各プロジェクトの生成中に読み取って適用する継続的な指示を提供します。

ZenOpus では、デザイン システムは **専用プロジェクト**として作成されます。このプロジェクトは、設計システムの信頼できる情報源として機能し、Workspace 内の他のプロジェクトに接続できます。

### 得られるもの

- UI コンポーネントとスタイルの信頼できる一元的な情報源
- 接続されているすべてのプロジェクトへの更新の自動伝播
- 生成時の明確で強制可能な設計とコードのガイドライン

## デザインシステムの仕組み

ZenOpus のデザイン システムは、コンポーネント、スタイル、セットアップ手順がプロジェクト間でどのように適用されるかを定義します。デザイン システムは通常の ZenOpus プロジェクトであり、他のプロジェクトと同様に開いて編集できます。

### デザインシステムの構成要素

ZenOpus の設計システムは、次の 3 つのコア部分を組み合わせています。

1. **コンポーネント**: React コンポーネント ライブラリ (パブリックまたはプライベート npm パッケージ)
1. **ガイドライン**: スタイル ルール、使用パターン、およびコンポーネントの仕様
1. **インストール**: セットアップ手順と必要な構成

これらをすでに用意している場合は、ZenOpus で **デザイン システム プロジェクト**を作成し、既存のマテリアルを参照できます。ドキュメントが散在しているか不完全な場合は、ZenOpus 内で直接設計システムを作成および改善できます。

### デザイン システム プロジェクトと .loveable フォルダー

デザイン システム プロジェクトは、特別な `.zenopus` フォルダーを所有します。このフォルダーには、接続されたプロジェクトでコードを生成するときに ZenOpus が使用する次のような命令が含まれています。

- `system.md`: 接続されたプロジェクトに対して常にロードされるコアのインストール手順と高レベルのガイドライン
- `rules/`: ZenOpus がプロジェクト生成中に参照できるコンポーネント仕様、ガイドライン、パターンを含む個別のマークダウン ファイル。

### デザインシステムをプロジェクトに適用する方法

プロジェクトがデザイン システムに接続されると、ZenOpus はデザイン システムの `.zenopus` フォルダーの内容を読み取り、プロジェクト生成中にそれらのルールを適用します。

デザイン システム プロジェクトに加えられた更新は、次回 ZenOpus がコードを生成するときに、接続されているすべてのプロジェクトによって取得されます。

### 重要な注意事項と制限事項

- `.zenopus` フォルダーはデザイン システム プロジェクトによって所有されており、接続されているプロジェクトから編集することはできません。
- デザイン システムの更新は、次世代でそれに接続されているすべてのプロジェクトに自動的に適用されます。
- 設計システムはまだバージョン管理されていません。ワークフローでこれが必要な場合は、アカウント チームにお問い合わせください。
- MCP サーバーまたはパッケージに必要なものがすべて含まれている場合は、インストールのみの設計システムを作成できます。
- デザイン システムは React コンポーネント ライブラリとネイティブに連携します。
- プロジェクト設定からあらゆるプロジェクトをデザイン システムに変えることができます。プロジェクトがデザイン システムとしてマークされると、Workspace 内の他のプロジェクトが接続できるようになります。
- 設計システムまたは関連ドキュメントが VPC 内でホストされている場合は、アカウント チームに連絡してガイダンスを求めてください。

## デザインシステムを作成する

<Steps>
<Step title="アクセスの準備 (プライベート パッケージのみ)">
コンポーネント ライブラリまたはパッケージがプライベートである場合は、**Workspace 設定 → ビルド シークレット** に移動し、必要なビルド シークレット (npm トークンなど) を追加します。
</Step>
<Step title="デザインシステムプロジェクトを作成する">
[zenopus.dev](https://zenopus.dev) プロンプト ボックスの `+` ドロップダウンから、**デザイン システム** に移動し、**デザイン システムの作成** を選択します。

![Dsクリエイトデザインシステム](/images/zenopus/features/design-systems/ds-create-design-system.png)

これにより、新しいデザイン システム プロジェクトが作成されます。 ZenOpus はプロジェクトに名前を割り当てます。この名前はプロジェクト設定から変更できます。

デザイン システムに意味のある名前を付けます。この名前は、後でデザイン システムをプロジェクトに接続するときに使用されます。
</Step>
<Step title="ZenOpus とチャットして設計システムを構成する">
デザイン システムは主に 2 つの方法でセットアップできます。

**オプション A: 既存の React ライブラリを自動フェッチ**

デザイン システムが React コンポーネントとして実装されている場合は、チャットでリポジトリまたはドキュメントの URL を共有します。

- public の場合、ZenOpus が自動的に取得します。
- プライベートの場合、ZenOpus は構成したシークレットを使用してそれを取得します。

コンポーネントをインポートすることに加えて、ZenOpus **それらのコンポーネントの正しい使用方法**を伝える必要があります。このガイダンスは次のものから得られます。

- 既存のドキュメント サイトまたはリポジトリ
- インポートされた PDF、Markdown ファイル、またはスクリーンショット
- 設計システムのドキュメントを公開する MCP サーバー
- チャットで提供される直接の指示

<Info>
既存のライブラリを取得する場合は、ガイド付きの質問を無視して URL の入力に進みます。
</Info>

**オプション B: 構造化された命令を使用して設計システムを構築する**

ドキュメントが散在しているか不完全な場合は、ZenOpus を明示的にガイドして設計システムを構築してください。このアプローチにより、次のことが可能になります。

- PDF、Markdown ファイル、スクリーンショット、その他のアセットをインポートする
- コンポーネント、パターン、トークン、スタイルのガイドラインを定義する
- インストールとセットアップの手順を書く
- 接続されているプロジェクト間ですべてを動的に更新します

**推奨されるプロンプト構造**

```text
【上位目標】

【インストール手順】

[システムをセットアップするためのその他のコンテキスト、たとえば技術スタックや制約]

【コンポーネントライブラリリクエスト】

[詳細なコンテキストについては、ドキュメント、サイト、PDF、MCP へのリンク。 ZenOpus は Web サイトをクロールできるため、ここに幅広いコンテキストを自由に含めることができます。]
```

最初の実行後、ZenOpus は機能するデザイン システム プロジェクトを生成します。例: [https://design-system-demo.zenopus.app/](https://design-system-demo.zenopus.app/)
</Step>
</Steps>

## デザインシステムをプロジェクトに接続する

各プロジェクトの設定では、どのデザイン システムが接続されているか、または接続可能であるかを確認できます。

複数のデザイン システムを 1 つのプロジェクトに接続できます。接続された設計システムの順序によって、プロジェクト生成中にエージェントがそれらの設計システムに優先順位を付ける方法が決まります。プロジェクト設定で、接続されたデザイン システムの順序を変更できます。

接続されているデザイン システムは、プロジェクト設定からいつでも削除できます。削除された設計システムは、そのプロジェクトの将来の世代には適用されなくなります。

### デザインシステムを新しいプロジェクトに接続する

[zenopus.dev](https://zenopus.dev) プロンプト ボックスの `+` ドロップダウンから、**デザイン システム** に移動し、使用可能なデザイン システムのリストからデザイン システムを選択し、プロンプトを入力します。 ZenOpus は、選択したデザイン システムを使用して新しいプロジェクトを生成します。

![DSセレクトデザインシステム](/images/zenopus/features/design-systems/ds-select-design-system.png)

### デザインシステムを既存のプロジェクトに接続する

デザイン システムを既存のプロジェクトに接続するには、次の手順に従います。

1. **プロジェクト設定 → デザイン システム** に移動します。
1. **利用可能**でデザイン システムを選択し、**追加**をクリックします。
1. **[保存]** をクリックします。

## デザインシステムを改良する

デザインシステムは進化するものです。初期セットアップ後、`.zenopus` フォルダーの内容を繰り返し処理します。

1. **構造を確認してください**  
`.zenopus` フォルダーを開いて、ZenOpus が設計システムをどのように解釈したかを確認します。これらのファイルは、接続されているすべてのプロジェクトに適用されます。
1. **不一致を修正**  
コンポーネントが正しくレンダリングされない場合は、ZenOpus に修正を依頼し、同時に `.zenopus` フォルダーを更新します。
1. **詳細を追加**  
必要に応じて、追加のコンポーネントのドキュメント、トークン定義、または使用パターンをリクエストします。

## .loveable フォルダー構造

通常、ZenOpus はこれを自動的に生成しますが、構造を理解するとデバッグに役立ちます。予期しない動作に気付いた場合は、この構造が一般的にに従っていることを確認してください。

```text
.愛らしい/
§── system.md (最大 ~500 行)
│ §── インストール手順（10行）
│ │ - プライベートパッケージのセットアップ
│ │ - 環境変数
│ │ - サンプル .npmrc
│ §── アセットリンク（フォント、アイコンなど）
│ §── ディレクトリガイド（20行）
│ │ - どのようなファイルが存在するか
│ │ - いつ参照するか
│ └── 大まかなガイドライン（残り行）
│ - コードパターン
│ - 設計原則
│ - ディシジョンツリー
└── ルール/
    §── コンポーネント/
    │ §── button.md
    │ §── input.md
    │ └── modal.md
    §── パターン/
    │ §──forms.md
    │ └── ナビゲーション.md
    ━── スタイリング/
        §── color.md
        └── タイポグラフィ.md
```

## MCP サーバーを使用して外部設計システムのドキュメントを取得する

MCP サーバーを使用すると、npm パッケージとは別に保存されている設計システムのドキュメントを取得できます。詳細については、「パーソナル コネクタを使用したツールとの統合 (MCP サーバー)」(/jp/integrations/mcp-servers) を参照してください。

## テンプレートとデザインシステムのどちらかを選択してください

[テンプレート](/jp/features/business/design-templates) は、開始点として使用される完全なプロジェクトのコピーです。

次の場合に **テンプレート** を使用します。

- 複雑なセットアップが必要です (複数のパッケージ、カスタム構成)
- フレームワークのバリアント (React、Vue、Vite) が必要です
- 再生成すべきではない重い定型文があります

次の場合に **デザイン システム** を使用します。

- 継続的な指導と一貫性が必要な場合
- パターンとコンポーネントは時間の経過とともに進化します
- 更新は自動的に反映されるはずです

**推奨されるアプローチ**:

- 初期プロジェクトの足場には **テンプレート** を使用する
- 進行中の UI とデザイン ルールには **デザイン システム** を使用する

## トラブルシューティング

<AccordionGroup>
<Accordion title="React バージョンの不一致">
デザイン システムが React 19 を使用しているが、プロジェクトのデフォルトが React 18 である場合、またはその逆の場合は、エージェントに明示的に伝えてください。たとえば:

```text
私のデザイン システムには React 19 が必要です。それに応じてプロジェクトの依存関係を更新してください。
```
</Accordion>
<Accordion title="競合する依存関係">
デザイン システムが `shadcn/ui` または Tailwind CSS を使用していない場合は、競合を避けるためにそれらを削除するようにエージェントに依頼してください。たとえば:

```text
このプロジェクトから shadcn/ui と Tailwind CSS をアンインストールしてください。私のデザイン システムは代わりに [あなたのスタイリング ソリューション] を使用します。
```
</Accordion>
<Accordion title="コンポーネントが見つかりません">
デザイン システム プロジェクトが `.zenopus/rules/` フォルダーに正しくエクスポートされ、コンポーネントが文書化されていることを確認します。エージェントは、これらのルール ファイルに基づいてコンポーネントの使用方法を理解します。
</Accordion>
<Accordion title="スタイルが適用されない">
デザイン システムの CSS またはテーマ トークンが、`index.css` や `App.tsx` などのプロジェクトのメイン エントリ ファイルにインポートされていることを確認します。
</Accordion>
</AccordionGroup>

## よくある質問

<AccordionGroup>
<Accordion title="デザイン システムは単なる ZenOpus プロジェクトですか?">
はい、デザイン システムは、デザイン システムとしてマークされている通常の ZenOpus プロジェクトです。これは他のプロジェクトと同様に開いて編集することができ、他のプロジェクトがそれに接続して生成中にそのルールを使用することができます。
</Accordion>
<Accordion title="複数のデザイン システムを 1 つのプロジェクトに接続できますか?">
はい、プロジェクトには複数の接続された設計システムを含めることができます。接続された設計システムの順序によって、プロジェクト生成中にエージェントがそれらの設計システムに優先順位を付ける方法が決まります。
</Accordion>
<Accordion title="どのようなプロジェクトでもデザイン システムに変えることができますか?">
はい、プロジェクト設定からどのプロジェクトもデザイン システムとしてマークできます。マークを付けると、Workspace 内の他のプロジェクトが接続できるようになります。
</Accordion>
<Accordion title="自分のプロジェクトにどの設計システムが接続されているかを確認するにはどうすればよいですか?">
接続されているデザイン システムを表示するには、**プロジェクト設定 → デザイン システム** に移動します。

このタブには、現在プロジェクトにリンクされているすべてのデザイン システムが表示されます。ここから、次のことができます。

- Workspace から追加の設計システムを接続します
- 優先順位を並べ替える
- 現在のプロジェクトを Workspace 内の他のプロジェクトのデザイン システムとして使用するかどうかを選択します。
</Accordion>
<Accordion title="2 つの設計システムが重複するルールを定義するとどうなるでしょうか?">
複数の設計システムが接続されている場合、ZenOpus はそれらを順番に適用します。リストの前の方にある設計システムが、リストの後の方にあるものより優先されます。

プロジェクト設定で接続されたデザイン システムの順序を変更して、生成中の優先順位を変更できます。
</Accordion>
<Accordion title="プロジェクトからデザインシステムを削除できますか?">
はい、接続されたデザイン システムはプロジェクト設定からいつでも削除できます。削除すると、そのプロジェクトの将来の世代には適用されなくなります。
</Accordion>
<Accordion title="接続されているプロジェクトからデザイン システムを編集できますか?">
いいえ、`.zenopus` フォルダーはデザイン システム プロジェクトによって所有されており、接続されているプロジェクトから編集することはできません。デザイン システムを変更するには、デザイン システム プロジェクト自体を開いて編集します。
</Accordion>
<Accordion title="設計システムはバージョン管理されていますか?">
まだです。現在、設計システムには Support 組み込みのバージョニングがありません。ワークフローにとってバージョン管理が重要な場合は、アカウント チームにお問い合わせください。
</Accordion>
<Accordion title="システム Support はどのようなテクノロジーで設計されていますか?">
ZenOpus デザイン システム Support **すぐに使える React ベースのコンポーネント ライブラリ**。

デザイン システムが Vue、Angular、Svelte などの他のフレームワークで構築されている場合は、CSM または FDE に連絡して互換性オプションについて話し合ってください。
</Accordion>
<Accordion title="デザインシステムはテンプレートを置き換えますか?">
いいえ、デザイン システムはテンプレートを置き換えるものではありません。テンプレートとデザイン システムは異なる目的を果たし、一緒に使用されることがよくあります。テンプレートは初期プロジェクトの足場作りに最適ですが、デザイン システムはプロジェクト全体で継続的な UI とデザイン ガイダンスを提供します。
</Accordion>
</AccordionGroup>
