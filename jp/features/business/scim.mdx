**SCIM (クロスドメイン ID 管理システム)** は、**エンタープライズ プラン** でのみ利用可能です。これにより、アイデンティティ プロバイダー (IdP) を通じて自動化されたユーザー プロビジョニングとライフサイクル管理が可能になります。

SCIM を使用すると、IdP から Workspace アクセスを一元管理できます。ユーザーは割り当てられると自動的に追加され、割り当てが解除されると削除され、グループ メンバーシップに基づいて役割が割り当てられるため、手動でユーザーを管理しなくても Workspace の同期が維持されます。

## 前提条件

SCIM プロビジョニングを設定する前に、以下が必要です。

- **IdP 管理者アクセス** (Okta、Microsoft Entra ID、または使用しているその他の SCIM 2.0 プロバイダー)
- **ZenOpus Workspace 所有者または管理者の役割**
- **アクティブな SSO プロバイダーが構成されている** (OIDC または SAML)。詳細については、[**シングル サインオン (SSO) のセットアップ**](/jp/features/business/sso) を参照してください。
    - **SSO をまだお持ちでない場合:**  
    通常は、SSO 認証と SCIM プロビジョニングの両方を処理する単一のアプリケーションをアイデンティティ プロバイダーに作成します。
    - **すでに SSO が設定されている場合:**  
    オプションで、SCIM プロビジョニング専用の別のアプリケーションを IdP に作成できます。これにより、既存の SSO 設定を変更せずに維持できます。 ZenOpus は、どの IdP アプリケーションが SCIM を処理するかに関係なく、ユーザー認証に現在の SSO プロバイダーを引き続き使用します。
    <Note>
    一部のアイデンティティ プロバイダーでは、SSO 認証が OIDC を使用する場合でも、SAML アプリケーションで SCIM プロビジョニングを構成する必要があります。この場合、SCIM が有効になっている SAML アプリケーションはプロビジョニングのみに使用され、ZenOpus で SSO プロバイダーとして構成する必要はありません。
    </Note>

## ZenOpus での SCIM の仕組み

このセクションでは、ZenOpus がアイデンティティ プロバイダーからの SCIM イベントを処理する方法について説明します。

### ユーザープロビジョニング

IdP がユーザーを作成するか、ZenOpus アプリケーションに割り当てる場合:

1. IdP は SCIM リクエストを ZenOpus に送信します。
1. ZenOpus は、ユーザーの電子メール ドメインが Workspace に対して検証されていることを確認します。
1. ユーザーは参加への招待メールを受け取ります。
1. ユーザーが招待を受け入れてアカウントを作成すると、SCIM 構成に基づいて適切なロールを持つユーザーが Workspace に追加されます。

### ユーザーのプロビジョニング解除

IdP がユーザーを削除または非アクティブ化すると、次のようになります。

1. IdP は非アクティブ化リクエストを ZenOpus に送信します。
1. ユーザーは Workspace から削除されます。
1. ユーザーは Workspace にログインできなくなります。

<Note>
**Workspace 所有者は SCIM 経由でプロビジョニング解除できません。** これにより、Workspace 管理の偶発的なロックアウトが防止されます。
</Note>

### グループプッシュとロールの更新

IdP でグループベースのプロビジョニングが有効になっている場合:

1. グループ メンバーシップの変更は ZenOpus にプッシュされます。
1. マップされたグループに追加されたユーザーは、対応するロールを受け取ります。
1. すべてのマップされたグループから削除されたユーザーは、Workspace から削除されます。

### サポートされている SCIM 操作

ZenOpus は SCIM 2.0 仕様を実装し、次の操作をサポートします。

|リソース |サポートされている操作 |
| --- | --- |
|ユーザー |作成、読み取り、更新、削除、リスト |
|グループ |作成、読み取り、更新、削除、リスト、メンバーのプッシュ |

## SCIM プロビジョニングをセットアップする

SCIM プロビジョニングを設定するには、ZenOpus と ID プロバイダーの両方で構成が必要です。 ZenOpus から開始して SCIM プロビジョニングを有効にし、必要な認証情報を生成してから、IdP でのセットアップを完了します。

### ステップ 1: ZenOpus で SCIM を構成する

まず、ZenOpus で SCIM プロビジョニングを有効にし、アイデンティティ プロバイダーに必要な値をコピーします。

<Steps>
<Step title="ID 設定を開き、SCIM を有効にします">
**設定 → Workspace → ID → SCIM プロビジョニング** に移動し、**SCIM プロビジョニング** を有効にします。
</Step>
<Step title="SCIM 構成値をコピーする">
SCIM プロビジョニングが有効な場合、ZenOpus は次の値を生成して表示します。

- **API キー:** SCIM リクエストの認証に使用される安全な API トークン (1 回のみ表示)
- **ベース URL:** IdP がユーザーを同期するために使用するエンドポイント。

<Warning>
**API キーをすぐに保存します**。完全な API キーは 1 回だけ表示されます。キーを紛失した場合は、キーをローテーションして新しいキーを生成する必要があります。
</Warning>

これらの値は、アイデンティティ プロバイダーで SCIM を構成するときに必要になります。
</Step>
<Step title="デフォルトの役割を構成する">
**デフォルトのロール**は、どのグループ マッピングにも一致しない、新しくプロビジョニングされたユーザーに割り当てられます。利用可能な役割は次のとおりです。

- **ビューア**: 読み取り専用アクセス
- **エディタ**: プロジェクトを作成および編集できます。
- **管理者**: Workspace の完全な管理

オプションで [グループ ロール マッピングを構成](/jp/features/business/scim#ロールマッピングの構成) して、アイデンティティ プロバイダー グループに基づいてロールを割り当てることができます。一致するグループ マッピングがない場合は、デフォルトのロールが適用されます。
</Step>
</Steps>

ZenOpus で生成された値を使用して、アイデンティティ プロバイダーで SCIM プロビジョニングを構成します。

|設定 |値 |
| --- | --- |
|ベース URL | https://api.zenopus.dev/scim/v2 |
|認証 |無記名トークン |
| APIキー | __タグ1__ |

## ロールマッピングの構成

SCIM は、IdP グループのメンバーシップに基づいた自動ロール割り当てをサポートしているため、Workspace 権限を一元的に制御できます。

### IdP グループをロールにマッピングする

IdP グループを ZenOpus Workspace ロールにマップして、きめ細かいアクセス制御を実現します。

グループ役割マッピングを追加するには、次の手順を実行します。

1. **[設定] → Workspace → [ID] → [SCIM プロビジョニング] ** に移動します。
1. [**グループ ロール マッピング**] で、**グループ名** を IdP に表示されているとおりに正確に入力します (例: `engineering-admins`)。
1. 割り当てる **役割** を選択します (閲覧者、編集者、または管理者)
1. [**追加**] をクリックして保存します。

<Note>
グループ名では大文字と小文字が区別されません。たとえば、`Engineering-Admins` と `engineering-admins` は両方とも一致します。
</Note>

**マッピングの例**

| IdPグループ | ZenOpus 役割 |
| --- | --- |
|愛すべき管理者 |管理者 |
|エンジニアリング |編集者 |
|請負業者 |ビューア |

ユーザーがプロビジョニングされるとき:

- ZenOpus は、ユーザーがマップされたグループに属しているかどうかを確認します。
- 一致するものが見つかった場合は、対応するロールが割り当てられます。
- 一致するものが見つからない場合は、**デフォルトの役割**が割り当てられます。

## SCIM プロビジョニングを管理する

ID 設定を使用して、SCIM プロビジョニングを長期的に管理します。

### APIキーをローテーションする

API キーが侵害された可能性がある場合、または再生成する必要がある場合は、API キーをローテーションします。

1. **[設定] → Workspace → [ID] → [SCIM プロビジョニング] ** に移動します。
1. API キーの横にある [**回転**] をクリックします。
1. 回転を確認します。
1. 新しい API キーをすぐにコピーして保存します。
1. 新しい API キーを使用して IdP を更新します。

<Warning>
API キーをローテーションすると、以前の API キーはすぐに無効になります。プロビジョニングの中断を避けるために、IdP 構成をすぐに更新してください。
</Warning>

### SCIM プロビジョニングを無効にする

自動プロビジョニングを停止するには:

1. **[設定] → Workspace → [ID] → [SCIM プロビジョニング] ** に移動します。
1. **SCIM プロビジョニング** を無効にします。

<Note>
SCIM を無効にすると自動プロビジョニングは停止しますが、既存の Workspace メンバーは削除されません。以前に SCIM 経由でプロビジョニングされたユーザーは、手動で削除されるまで Workspace に残ります。
</Note>

## トラブルシューティング

<AccordionGroup>
<Accordion title="ユーザープロビジョニングが「ドメインが検証されていません」エラーで失敗する">
SCIM は、Workspace に対して電子メール ドメインが検証されているユーザーのみをプロビジョニングします。修正するには:

1. **設定 → Workspace → ID** に移動します
1. **確認済みのドメイン** でメール ドメインを追加して確認します。
1. IdP からプロビジョニングを再試行します。
</Accordion>
<Accordion title="ユーザーはプロビジョニングされていますが、ログインできません">
次のことを確認します。

- SSO プロバイダーは正しく構成されています。
- ユーザーは IdP の SSO アプリケーションに割り当てられます。

SCIM 経由でプロビジョニングされたユーザーは、SSO 経由で認証する必要があります。
</Accordion>
<Accordion title="ロールマッピングが適用されていない">
次のことを確認してください。

- マッピング内のグループ名は、IdP が送信するものと完全に一致します (大文字と小文字は区別されません)。
- IdP は、SCIM リクエストでグループ メンバーシップ データを送信するように構成されています
- IdP でグループ プッシュが有効になっている
</Accordion>
</AccordionGroup>

## よくある質問

<AccordionGroup>
<Accordion title="SSO なしで SCIM を使用できますか?">
いいえ、SCIM にはアクティブな SSO プロバイダーが必要です。 SCIM 経由でプロビジョニングされたユーザーは、構成された SSO プロバイダーを使用して認証されます。
</Accordion>
<Accordion title="SCIM を有効にすると、既存のユーザーはどうなりますか?">
SCIM を有効にしても、既存の Workspace メンバーは影響を受けません。 SCIM は、IdP を通じてプロビジョニングされたユーザーを管理します。以前に招待されたユーザーは、SCIM によってプロビジョニングされたユーザーとともに引き続き存在します。
</Accordion>
<Accordion title="APIキーを紛失してしまいました。どうすればいいですか？">
API キーは生成時に 1 回だけ表示されます。紛失した場合:

1. **[設定] → Workspace → [ID] → [SCIM プロビジョニング] ** に移動します。
1. API キーの横にある [**回転**] をクリックして、回転を確認します。
1. 新しい API キーを使用して IdP を更新します。
</Accordion>
<Accordion title="ユーザーが複数のマップされたグループに属している場合はどうなりますか?">
ユーザーが複数のマップされたグループに属している場合、ZenOpus はそれらのグループから最も高い特権のロールを割り当てます。
</Accordion>
<Accordion title="SCIM またはジャストインタイム (JIT) プロビジョニングを使用する必要がありますか?">
SCIM は、ユーザーのライフサイクルとアクセスを ID プロバイダーから一元的に制御する必要がある管理環境に推奨されます。

ジャストインタイム (JIT) プロビジョニングは、SSO を通じてサインアップしたユーザーにのみ適用されます。 SSO サインアップ経由でユーザーが作成されると、JIT ロールが適用されます。

ユーザーが SCIM 経由でプロビジョニングされる場合、ユーザーの作成と役割の割り当て (グループベースの役割マッピング、ユーザーのメタデータ、またはデフォルトの SCIM 役割など) が SCIM によって管理されます。この場合、SCIM プロビジョニングと役割の割り当てが JIT よりも優先されます。
</Accordion>
</AccordionGroup>
