<Note>
このページでは、**ZenOpus Cloud** での Google 認証について説明します。ここで説明するマネージド Google 認証フローと独自の認証情報の設定は、[**ZenOpus Cloud アプリのみ**](/jp/integrations/cloud) で利用できます。
</Note>

## 概要

ZenOpus は、業界標準の OAuth 2.0 を使用した **Sign in with Google** をサポートしており、ユーザーはパスワードを作成する代わりに Google アカウントで認証できるようになります。

[ZenOpus Cloud](/jp/integrations/cloud) の Google 認証は、次の 2 つの方法で構成できます。

- **ZenOpus によって管理**: ZenOpus は、OAuth クライアント、資格情報、リダイレクト処理、およびセキュリティ更新を管理します。これはデフォルトの推奨設定です。
- **独自の認証情報** (BYOK とも呼ばれます): 独自の Google Cloud プロジェクトから OAuth 認証情報を接続し、認証情報のセキュリティとローテーションを管理します。

どちらのオプションも、ユーザーに同じサインイン エクスペリエンスを提供します。

1. ユーザーがアプリ内で [**Google でログイン**] をクリックします
1. Googleが同意画面を表示
1. ユーザーがアクセスを確認して承認する
1. Google はユーザーをサインインしたアプリにリダイレクトします

技術的な観点から見ると、OAuth は以下に依存します。

- **クライアント ID** (アプリケーションを識別します)
- **クライアント シークレット** (アプリケーションを承認する)
- **リダイレクト URL** (Google が承認後にユーザーを送信する場所)

ZenOpus は、このフローをアプリケーションと UI に直接統合します。

## Google サインインを使用する理由

Google サインインは、使い慣れた認証エクスペリエンスを提供し、サインアップとログイン時の煩わしさを軽減します。

- パスワードを作成せずにオンボーディングを高速化
- 信頼できる Google の同意エクスペリエンス
- パスワードのリセットとアカウント回復のオーバーヘッドの削減
- Google が提供する事前検証済みメール アドレス
- プロファイル情報を備えた信頼性の高いユーザー ID

## 一般的な使用例

Google サインインは一般的に次の目的で使用されます。

- ソーシャルログインが期待されるコンシューマアプリケーション
- プロユーザーをターゲットとしたSaaS製品
- Google Workspace を使用した内部ツール
- Eコマースアカウントと注文履歴
- 検証済みのメールアドレスが必要なコミュニティまたは製品

## Google認証設定オプション

ZenOpus Cloud は、次の 2 つの方法で Google 認証の設定をサポートします。

- **ZenOpus** によって管理されます
- **あなた自身の資格情報**

どちらの場合でも、エンド ユーザー エクスペリエンスは同じです。違いは、OAuth 認証情報の管理方法です。

<Tabs>
<Tab title="ZenOpus によって管理 (デフォルト)">
ZenOpus は、このフローの OAuth クライアント、資格情報、リダイレクト処理、および関連するセキュリティ更新を管理します。 Google Cloud Console の構成は必要ありません。

次の場合にこのオプションを使用します。

- 最も簡単なセットアップが必要です
- Google Cloud を手動で構成したくない
- OAuth 認証情報の所有権は必要ありません
- デフォルト (電子メールと基本プロファイル) を超えるカスタム OAuth スコープは必要ありません。

**ZenOpus** によって管理されている場合、ユーザーが初めてログインするときに Google **同意画面**が表示されます。

同意画面:

- アプリケーション名を表示します
- アプリが Google 認証経由でアクセスを要求していることを示します
- ユーザーのメールアドレスと基本的なプロフィール情報へのアクセスをリクエストします

ユーザーは一度アクセスを承認するように求められます。承認後はアプリにリダイレクトされ、権限が変更されない限り同意画面が再度表示されることはありません。

<Steps>
<Step title="Prompt ZenOpus : Google 認証を追加します">
```text
ユーザーが自分の Google アカウントでサインインできるように、アプリに Google ログインを追加します
```

ZenOpus は、**クラウド → ユーザー → 認証 → Google** で Google 認証を自動的に有効にし、次のことを行います。

- バックエンドで Google OAuth を構成する
- サインイン UI コンポーネントを作成または更新する
- 認証状態管理のセットアップ
- サインアウト機能を追加する

**詳細** ビューで変更を確認できます。
</Step>
<Step title="フローをテストする">
Google サインインを有効にした後、次のことを確認します。

- [**Google でログイン**] ボタンが表示されます
- ユーザーは認証のために Google にリダイレクトされます
- ユーザーはサインインした状態でアプリに戻ります
- 同意画面は期待どおりに動作します
- ユーザーは正常にサインインおよびサインアウトできます
</Step>
</Steps>
</Tab>
<Tab title="あなた自身の資格情報">
独自の Google Cloud プロジェクトで OAuth 認証情報を作成して管理し、ZenOpus に接続します。資格情報のセキュリティとローテーションを管理するのはあなた自身の責任です。

次の場合にこのオプションを使用します。

- 独自の Google Cloud プロジェクトで OAuth を必要とする
- 同意画面のブランディングを完全に制御する必要がある
- セキュリティまたはコンプライアンスの要件がある
- デフォルトを超えるカスタム OAuth スコープが必要

**独自の認証情報**を使用する場合、Google **同意画面**は、次のような Google Cloud プロジェクトによって完全に制御されます。

- アプリ名とブランディング
- 同意画面の構成
- OAuth スコープ

ZenOpus は、同意画面を変更せずに、提供された資格情報と設定を使用します。

<Steps>
<Step title="Google Cloud で OAuth 認証情報を作成する">
[Google Cloud Console](https://console.cloud.google.com) では、次のことを行う必要があります。

- Google Cloud プロジェクトを選択または作成します
- **OAuth 同意画面**を構成する
- **Web アプリケーション** タイプの **OAuth クライアント ID** を作成します

詳細については、[Google Auth Platform ドキュメント](https://Support.google.com/cloud/topic/15540269) を参照してください。
</Step>
<Step title="ZenOpus で資格情報を構成する">
- **クラウド → ユーザー → 認証 → Google** に移動します。
- **Google サインイン**を有効にし、**自分の認証情報**を選択します。
- **クライアント ID** と **クライアント シークレット** を入力します。
</Step>
<Step title="Google Cloud でリダイレクト URI を構成する">
独自の Google OAuth 資格情報を使用する場合、ZenOpus には **複数の可能なリダイレクト URI** が表示される場合があります。次に例を示します。

- デフォルトの ZenOpus ドメイン (`https://myapp.zenopus.app`)
- 追加した 1 つ以上のカスタム ドメイン (`https://myapp.com`)

Google Cloud Console では、**使用するすべてのリダイレクト URI** を OAuth クライアントの **承認されたリダイレクト URI** として追加する必要があります。

<Note>
リダイレクト URI は正確に一致する必要があります。スキーム、ドメイン、パス、末尾のスラッシュはすべて重要です。
</Note>
</Step>
<Step title="Google OAuth クライアントでどのリダイレクト URI が承認されているかを確認する">
ZenOpus は、**Google OAuth クライアントでどのリダイレクト URI が承認されているか**を知る必要があります。 Google Cloud に追加した後、ZenOpus に戻り、**ホワイトリストに登録したリダイレクト URI を選択**します。

ZenOpus でリダイレクト URI を選択すると、次のことを確認したことになります。

- Google OAuth クライアントでどのドメインが承認されているか
- サインイン後にユーザーをどのドメイン ZenOpus にリダイレクトできるか

これにより、デフォルト ドメインとカスタム ドメイン間で Google 認証が正しく機能し続けることが保証されます。
</Step>
<Step title="Google サインイン設定を保存する">
[**保存**] をクリックして、Google 認証情報と選択したリダイレクト URI を保存して確認します。
</Step>
<Step title="認証 UI をアプリに追加する">
Prompt ZenOpus を使用して、Google サインイン UI をアプリに追加します。

```text
自分の Google OAuth 資格情報を使用して、Google サインインをアプリに追加します。
```

ZenOpus は、カスタム資格情報を使用して UI を生成します。
</Step>
<Step title="フローをテストする">
実稼働環境にデプロイする前に、次のことを確認してください。

- [**Google でログイン**] ボタンが表示されます
- リダイレクト URI が正しく構成されている
- ユーザーはサインインした状態でアプリに戻ります
- 同意画面は期待どおりに動作します
- ユーザーは正常にサインインおよびサインアウトできます
</Step>
</Steps>
</Tab>
</Tabs>

## Google認証の管理

**ZenOpus** によって管理されている認証情報と **独自の認証情報**をいつでも **クラウド → ユーザー → 認証 → Google** から切り替えることができます。

Google 認証設定を切り替えても、既存のユーザー アカウントには影響しません。ユーザーは同じ Google アカウントを使用してサインインし続けることができます。

## 「独自の資格情報」設定のトラブルシューティング

<AccordionGroup>
<Accordion title="リダイレクト URI の不一致">
- ZenOpus で選択したすべてのリダイレクト URL が Google Cloud に追加されていることを確認します
- 末尾のスラッシュが欠落しているか、余分なものがないか確認してください
- スキーム (`https` と `http`) が正確に一致することを確認します。
</Accordion>
<Accordion title="同意画面の問題">
- OAuth 同意画面が Google Cloud で正しく構成されていることを確認する
- アプリが公開されていない場合は、アカウントをテスト ユーザーとして追加します
- 必要なスコープが構成されていることを確認します
</Accordion>
<Accordion title="間違った OAuth クライアント タイプ">
- **Web アプリケーション** OAuth クライアントを使用する
- モバイルまたはデスクトップのクライアント タイプは ZenOpus のリダイレクト フローと互換性がありません
</Accordion>
<Accordion title="認証が機能しない">
- **ユーザー → 認証 → Google** で Google サインインが有効になっていることを確認します
- `Client ID` と `Client secret` の値を確認します (独自の資格情報の設定)
- 認証 UI がバックエンドに接続されていることを確認する
</Accordion>
</AccordionGroup>

## よくある質問

<AccordionGroup>
<Accordion title="ZenOpus で管理される Google 認証はすべての ZenOpus アプリで利用できますか?">
いいえ。**ZenOpus 管理** Google 認証は **ZenOpus Cloud アプリのみ**で利用できます。
</Accordion>
<Accordion title="ZenOpus Cloud を使用していないアプリで Google 認証を使用できますか?">
はい。 Google 認証は ZenOpus Cloud を使用しないアプリに追加できますが、ZenOpus Cloud の外部で手動で設定する必要があります。

たとえば、ZenOpus Cloud を使用しないアプリでは、次のようにすることができます。

- Google OAuth をフロントエンドまたはバックエンド コードに直接統合します
- Google の One Tap または標準の OAuth サインイン フローを使用する
- Supabase Auth などのサードパーティ認証サービスを使用する
- OAuth 認証情報、リダイレクト処理、セッション ロジックを自分で管理する

このページでは、マネージド Google 認証や ZenOpus Cloud に固有の独自の認証情報フローなど、**ZenOpus Cloud の実装**について説明します。
</Accordion>
<Accordion title="ZenOpus によって管理されるオプションを使用するか、それとも自分の資格情報を使用する必要がありますか?">
独自の Google Cloud OAuth 認証情報の管理、同意画面のブランドのカスタマイズ、またはカスタム OAuth スコープの構成が特に必要な場合を除き、**ZenOpus によって管理** を使用します。
</Accordion>
<Accordion title="Google はどのような情報を私のアプリと共有しますか?">
デフォルトでは、Google はユーザーの **メール アドレス** と **基本的なプロフィール情報**を提供します。カスタム OAuth スコープ (独自の資格情報の設定のみ) を構成しない限り、追加のデータは共有されません。
</Accordion>
<Accordion title="ユーザーにはサインインするたびに Google 同意画面が表示されますか?">
いいえ。ユーザーには、**初めて Google にサインインしたとき**、または要求された権限が変更された場合に同意画面が表示されます。
</Accordion>
<Accordion title="後で「管理対象」と「独自の資格情報」を切り替えることはできますか?">
はい。 **ZenOpus** によって管理されている資格情報と **独自の資格情報** をいつでも切り替えることができます。既存のユーザー アカウントは影響を受けません。
</Accordion>
<Accordion title="リダイレクト URI の不一致エラーが表示されるのはなぜですか?">
これは通常、ZenOpus で選択したリダイレクト URL が、Google Cloud で構成された **承認されたリダイレクト URI** と正確に一致しないことを意味します。

- ZenOpus で選択したすべてのリダイレクト URL が Google Cloud に追加されていることを確認します。
- スキーム、ドメイン、パス、末尾のスラッシュを確認してください。
</Accordion>
<Accordion title="Google 認証は電子メール/パスワードによるサインインに代わるものですか?">
いいえ。Google 認証は追加のサインイン方法です。他の認証方法と一緒に Google サインインを提供できます。
</Accordion>
<Accordion title="「ZenOpus によって管理」セットアップを使用する場合、Google Cloud Console を構成する必要がありますか?">
いいえ。Google Cloud Console の設定は、**独自の認証情報**を使用する場合にのみ必要です。
</Accordion>
</AccordionGroup>
