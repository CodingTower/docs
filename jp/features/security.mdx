ZenOpus は、開発中および公開前に一般的なアプリケーションのセキュリティ リスクを軽減するのに役立つ組み込みのセキュリティ ツールを提供します。これらのツールは、アプリケーション開発を Support 安全に行うように設計されていますが、徹底的なセキュリティ レビューに代わるものではありません。

アプリがそのユースケースに適したセキュリティ要件を満たしていることを確認するのは、特に機密データを処理する場合や重要な機能を実行する場合の責任です。

## ZenOpus がアプリを安全に保つ方法

ZenOpus は、主に 2 つの方法でアプリケーションを保護します。1 つは一般的なセキュリティ上の間違いを事前に防ぐことで、もう 1 つは自動分析を通じて潜在的な脆弱性を特定することです。

脆弱性を特定するために、ZenOpus は 4 つの自動セキュリティ スキャナーを使用して、アプリケーションのさまざまな部分をチェックします。

- **RLS 分析**: データベース アクセス ポリシーと行レベルのセキュリティ ルールをレビューします。
- **データベース セキュリティ チェック**: データベース スキーマと RLS 構成を確認します。
- **コード セキュリティ レビュー**: 一般的なセキュリティ脆弱性についてアプリケーション コードを分析します。
- **依存関係の監査**: npm の依存関係における既知の脆弱性を検出します。

<Warning>
これらのスキャナーは一般的なセキュリティ問題の特定に役立ちますが、完全なセキュリティを保証するものではありません。機密データや重要な機能を扱うアプリについては、追加の専門家によるセキュリティ レビューを検討してください。
</Warning>

以下のセクションでは、リスクの主な領域について詳しく説明し、ZenOpus がそれらのリスクへの対処にどのように役立つかについて説明します。

安全なコードを記述し、一般的なセキュリティ上の間違いを回避するための実践的なガイダンスについては、[ZenOpus アプリのセキュリティのベスト プラクティス](/jp/tips-tricks/security-best-practices) を参照してください。

### APIキーの保護

ZenOpus は、ブラウザーで実行されるフロントエンド コードを生成します。これは、API キーやその他のシークレットをクライアント側のコードに安全に保存できないことを意味します。資格情報が公開されると、データ漏洩や不正アクセスが発生する可能性があります。

機密の資格情報が公開されるのを防ぐために、ZenOpus はチャットに貼り付けられた API キーを自動的に検出し、コードにハードコーディングするのではなく、**シークレット**に安全に保存するようにガイドします。

資格情報を直接貼り付けるのではなく、構築したいものを記述します。ZenOpus は、サーバー側の機能とシークレット ストレージを使用して安全なセットアップを実装するのに役立ちます。

**例:**

```text
サービス X API を統合してテキスト応答を生成したい
```

ZenOpus は次のことをガイドします。

1. API キーを Secret に安全に保存します
1. サーバー側で API 呼び出しを行うエッジ関数を作成する
1. フロントエンド コードからその Edge 関数を呼び出す

このアプローチにより、資格情報が安全に保たれ、Web セキュリティのベスト プラクティスに従います。

### 行レベルのセキュリティ (RLS) によるデータ アクセスの保護

行レベル セキュリティ (RLS) ポリシーは、どのユーザーがデータベース内のデータにアクセスまたは変更できるかを制御します。誤って設定された RLS ルールは、データ漏洩の一般的な原因です。

ZenOpus は、RLS ポリシーを検討して推論し、ユーザーがアクセスを許可されているデータのみを表示および変更できるようにするのに役立ちます。 **RLS 分析** スキャンの結果は、機密データが漏洩する可能性のある過度に寛容なルールやアクセス チェックの欠落を浮き彫りにします。

RLS ポリシーを定期的に確認する必要があります。特に公開前、または認証やデータベース構造の変更後には確認してください。

### データベースのセキュリティリスクの特定

データベース スキーマとアクセス ルールは連携して、データの公開方法を決定します。スキーマ設計とアクセス ポリシーの安全でない組み合わせは、個々のルールが正しいように見えてもセキュリティ リスクを引き起こす可能性があります。

ZenOpus は、**データベース セキュリティ チェック**を使用してデータベース スキーマと RLS 構成を一緒にレビューし、不正アクセスやデータ漏洩につながる可能性のあるパターンを特定します。これは、ルールを個別に確認する場合には明らかではない問題を発見するのに役立ちます。

### アプリケーションコードの脆弱性の特定

アプリケーション コードは、不適切な入力処理、安全でない認証フロー、外部サービスの安全でない使用などのパターンを通じてセキュリティ リスクを引き起こす可能性があります。

ZenOpus は、**コード セキュリティ レビュー** を使用してアプリケーション コードを分析し、一般的な脆弱性パターンを特定し、セキュリティを向上させるための推奨事項を提供します。これらのチェックは、本番環境で悪用される前に問題を早期に表面化するのに役立ちます。

### 依存関係の脆弱性の特定

サードパーティのライブラリは、セキュリティ リスクの一般的な原因となります。広く使用されているパッケージでも、新しい問題が発見されると脆弱性が発生する可能性があります。

自動セキュリティ スキャンの一環として、ZenOpus は **依存関係監査** を使用して、プロジェクトの依存関係に既知のセキュリティ脆弱性がないかチェックします。このスキャンにより、脆弱なパッケージが特定され、リスクの評価と修復に役立つ情報が提供されます。

依存関係の脆弱性に関する調査結果が表示される場合があります。

- プロジェクト レベルの [セキュリティ ビュー](/jp/features/security-view) では、詳細を確認し、単一プロジェクトのアクションを実行できます。
- Workspace レベルの [セキュリティ センター](/jp/features/security-center) では、Workspace 管理者と所有者が複数のプロジェクトにわたる脆弱性を監視できます。

調査結果には、影響を受けるパッケージの名前とバージョン、脆弱性の種類、安全なバージョンへの更新などの推奨される修復手順が含まれます。

依存関係を最新の状態に保つことは、アプリケーションを安全に維持するために重要です。重大な脆弱性が公開された場合、セキュリティ センターは影響を受けるプロジェクトを特定して、問題に迅速に対処できるようにします。

## セキュリティスキャンの実行時

ZenOpus は、ワークフローのさまざまなポイントでセキュリティ スキャンを実行し、問題を早期に発見できるようにします。これらのアクションは **無料で、クレジットを消費しません**。

各スキャナーは、プロジェクトに関連する場合、および最後のスキャン以降に変更が発生した場合にのみ実行されます。たとえば、データベース スキャナーはデータベースが接続されているプロジェクトに対してのみ実行され、依存関係の監査は依存関係が変更された場合にのみ実行されます。

<Note>
**コード セキュリティ レビュー**は自動的には実行されません。これは、**セキュリティ ビュー**で**更新**をクリックした場合にのみ実行されます。
</Note>

### 仕事をしながら

ZenOpus は、関連ファイルが変更されたときに特定のスキャナーを自動的にトリガーします。

- **RLS 分析** は、データベースの移行または構成ファイルが変更されたときに実行されます。
- **依存関係の監査**は、プロジェクトの依存関係が変更されたときに実行されます。
- **データベース セキュリティ チェック**は、RLS 分析結果が利用可能になった後のフォローアップとしてセッションごとに 1 回実行されます。

### 公開前

公開ダイアログを開くと、ZenOpus は **RLS 分析**、**データベース セキュリティ チェック**、**依存関係監査** スキャナーを自動的に実行します。重大な問題が見つかった場合は、続行する前にそれらを確認するよう求められます。

アプリを公開する前に、重要な問題をすべて解決してください。未解決の重大な問題を含めて公開することは可能ですが、特に運用アプリや機密性の高いユーザー データを扱うアプリの場合は、行わないことを強くお勧めします。

### オンデマンド

[セキュリティ ビュー](/jp/features/security-view) の **更新** ボタンを使用すると、いつでもオンデマンドでスキャンの更新をトリガーできます。結果が古いスキャナのみが実行されます。

これは **コード セキュリティ レビュー** スキャナーを実行する唯一の方法です。作業中または公開チェック中に自動的に実行されるわけではありません。

オンデマンド スキャンは、コードを大幅に変更した後、またはすべてのスキャン結果がプロジェクトの現在の状態を反映していることを確認したい場合に便利です。

## 会話によるセキュリティレビュー

自動スキャナーとは別に、プロンプトを表示することで、いつでも ZenOpus に **会話型セキュリティ レビュー** を実行するよう依頼できます。これは AI 主導のレビューであり、通常のチャット メッセージと同様にクレジットを消費します。

たとえば:

```text
アプリのセキュリティを確認する
```

ZenOpus はプロジェクトを分析し、詳細な評価を返します。レビューでは通常、次のような領域がカバーされます。

- アプリケーションコードの潜在的な脆弱性
- XSS リスク、入力処理、認証の欠陥などの一般的な問題
- データベース スキーマと行レベル セキュリティ (RLS) ポリシー
- 全体的なセキュリティ体制と具体的な推奨事項

レビューの範囲と深さはプロジェクトによって異なります。 ZenOpus に特定の領域に焦点を当てるように依頼することもできます。たとえば、次のようになります。

```text
アプリの認証フローを確認してください。
```

または

```text
API ルートが安全かどうかを確認してください。
```

会話型レビューは、重要な変更後またはメジャー リリース前に特に役立ちます。現在警告が表示されていない場合でも、運用アプリケーションでは定期的にレビューをリクエストすることをお勧めします。

## セキュリティを管理する場所

ZenOpus は 2 つのセキュリティ サーフェスを提供します。1 つは個々のプロジェクトを保護するためのもので、もう 1 つは Workspace 全体にわたるセキュリティを監視するためのものです。

### プロジェクトのセキュリティ (セキュリティ ビュー)

特定のプロジェクトを構築または保守する場合は、[**セキュリティ ビュー**](/jp/features/security-view) を使用します。

「セキュリティ」ビューは次のことに役立ちます。

- 1 つのプロジェクトのセキュリティ調査結果を確認する
- どの問題が重要であり、なぜそれが重要なのかを理解する
- 自動修正を適用するか、手動で変更を加えます
- プロジェクトが安全に公開できることを確認する

これは、開発中および公開前にセキュリティ問題に対処するための主な場所です。

### Workspace セキュリティ (セキュリティ センター)

Business および Enterprise プランの Workspace 管理者と所有者は、[**セキュリティ センター**](/jp/features/security-center) を使用して、Workspace 内のすべてのプロジェクト全体のセキュリティを監視できます。

セキュリティ センターは次のことに役立ちます。

- どのプロジェクトに重大なセキュリティ問題があるかを確認する
- チーム全体のパターンと再発リスクを特定する
- 複数のプロジェクトに影響を与える依存関係の脆弱性を監視する
- 全体的なセキュリティ体制を Workspace レベルで追跡する

セキュリティ センターはセキュリティ情報を集約しますが、修正は引き続きプロジェクト レベルで適用されます。

## 一般的なセキュリティ構成

プロジェクトと Workspace のセキュリティ レビューに加えて、ZenOpus はアカウント レベルおよび認証関連のセキュリティ設定を提供します。これらの構成は特定のプロジェクトに関連付けられていませんが、ユーザー アカウントを保護し、資格情報が侵害されるリスクを軽減する上で重要な役割を果たします。

認証を設定するとき、またはユーザーにサインインを許可するアプリを起動する前に、これらの設定を確認する必要があります。

### 漏洩したパスワードの保護

アプリで電子メールとパスワードによる認証を使用する場合、ユーザーが既知の侵害されたパスワードを選択するのを防ぐことができます。

この保護は、**HIBP (Have I Been Pwned) データベース**と照合してユーザー パスワードをチェックし、一般的に漏洩したパスワードや推測しやすいパスワードを拒否します。これを有効にすると、認証情報の再利用によるアカウント乗っ取りのリスクを軽減できます。

- **ZenOpus Cloud** の使用: **クラウド → ユーザー → 認証設定 → 電子メール** で **パスワード HIBP チェック**を有効にします
- **Supabase を直接使用する**: 詳細な手順については、[パスワード強度とパスワード漏洩保護](https://supabase.com/docs/guides/auth/password-security#password-strength-and-leaked-password-protection) に関する Supabase ドキュメントを参照してください。

## 費用と使用方法

ZenOpus には、自動セキュリティ ツールと会話型セキュリティ ツールの両方が含まれています。ほとんどのセキュリティ機能は無料で使用できますが、一部のセキュリティ アクションはクレジットを消費します。

自動化された会話型セキュリティ ツールは、連携して動作するように設計されています。自動化されたセキュリティ ツールを使用して、開発中および公開前に継続的に監視し、対象を絞った修正を行います。アプリ全体のセキュリティ体制をより深く探索的に評価したい場合は、会話型セキュリティ レビューを使用します。

### 自動セキュリティアクション (無料)

次の自動セキュリティ アクションはクレジットを消費しません。

- **セキュリティ スキャンの実行**  
[セキュリティ] ビューで [**更新**] をクリックするなど、セキュリティ スキャンの更新は無料です。
- **検出された問題を修正します**  
[セキュリティ] ビューで **すべて修正を試す** を使用するなど、検出結果の自動修正は無料です。
- **セキュリティ ビューでの調査結果の確認と議論**  
**セキュリティ ビュー内のインライン チャット**を使用して、特定のセキュリティ検出結果について質問したり修正をリクエストしたりすることは無料です。

これらのアクションでは、ZenOpus の自動セキュリティ スキャナーまたはセキュリティ固有の修復ツールが使用され、通常のチャットの使用としてカウントされません。

### 会話型セキュリティ アクション (クレジットを使用)

次の会話型セキュリティ アクションは、通常のチャット メッセージと同様にクレジットを消費します。

- **会話型セキュリティレビュー**  
チャットで ZenOpus に「アプリのセキュリティをレビューしてください」と依頼すると、AI 主導の会話型レビューが実行され、通常のチャット メッセージと同様にクレジットが消費されます。
- **チャットでの一般的なセキュリティの質問**  
自動セキュリティ ツールを使用せずに、メイン チャットでセキュリティ関連の質問を直接行うと、標準のチャットの使用として扱われ、クレジットが消費されます。
