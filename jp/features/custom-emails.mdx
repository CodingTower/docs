## 電子メールとは何ですか

電子メールを使用すると、ZenOpus Cloud アプリは、デフォルトの ZenOpus Cloud 認証送信者ではなく、独自の電子メール ドメインから認証電子メール (サインアップ確認、パスワード リセット、マジック リンク、招待状など) を送信できるようになります。

例: `noreply@notify.yourdomain.com` または `noreply@yourdomain.com`

専用の送信サブドメインを使用すると、ブランドの一貫性を維持しながら、ルート ドメインの評判を保護することができます。

以下をカスタマイズできます。

- プロジェクトが使用する電子メール ドメイン
- 送信者のサブドメイン構成
- 電子メールのブランディングとビジュアル スタイル
- 認証メールのテンプレートと件名

ZenOpus は、ドメイン検証、DNS 構成、電子メール認証 (`SPF`、`DKIM`、および `DMARC`)、および配信インフラストラクチャを処理します。外部の電子メール プロバイダー アカウントや API キーは必要ありません。

## サポートされている電子メールの種類

現時点では、メール Support **認証メールのみ**。

認証電子メールは、アカウント アクセスと身元確認に関連するトランザクション電子メールの一種です。ユーザーが特定のセキュリティ関連アクションを実行すると、クラウド認証によって自動的にトリガーされます。

これらの認証メールには次のものが含まれます。

- **サインアップを確認**  
ユーザーがアカウントを作成し、電子メール確認が有効になっているときに送信されます。アカウントをアクティブ化する前に、ユーザーが電子メール アドレスを所有していることを確認します。
- **パスワードのリセット**  
ユーザーがパスワードのリセットを要求したときに送信されます。安全なパスワード回復を可能にします。
- **マジックリンク**  
パスワードなしのログインが有効で、ユーザーがログイン リンクを要求したときに送信されます。パスワードを要求せずにユーザーを認証します。まだ構成されていない場合は、ZenOpus にマジック リンク認証を有効にするよう依頼できます。
- **招待**  
ユーザーをプロジェクトに招待するときに送信されます (クラウド → ユーザーから)。安全な招待リンクを通じてアカウントへのアクセスを許可します。
- **メールアドレスの変更**  
ユーザーが電子メール アドレスを更新すると送信されます。変更を適用する前に、新しいアドレスの所有権を確認します。
- **再認証**  
ユーザーがパスワードの変更などの機密性の高いアクションを実行する前に身元を確認する必要がある場合に送信されます。通常、有効期間が短い検証コードが含まれます。再認証がまだ構成されていない場合は、ZenOpus に再認証を有効にするよう依頼できます。

<Note>
マーケティング電子メール、ニュースレター、注文確認メール、発送通知、その他の取引電子メールや通知電子メールは、現時点ではサポートされていません。
</Note>

## 可用性と使用法

メールは**有料プラン**でご利用いただけます。有料の Workspace には、**毎月 50,000 件の認証メール**が追加料金なしで含まれています。使用量は Workspace 全体にわたって計算されます。

追加の認証メールには、1,000 メールあたり 1 ドルが課金されます。

## 電子メールを使用する理由

- **到達性の向上:** 独自のドメインから送信された電子メールは、スパムではなく受信箱に届く可能性が高くなります。
- **信頼性の向上:** ユーザーには差出人アドレスにあなたのドメインが表示されるため、信頼性が強化されます。
- **一貫したブランディング:** 認証メールは製品 ID と一致します。
- **管理されたセットアップ:** ZenOpus はドメインを検証し、必要な DNS レコードを構成します。
- **認証された送信:** ZenOpus は、メールを認証し、受信トレイの配置を改善するために [SPF、DKIM、および DMARC](#what-are-spf-dkim-and-dmarc-and-do-i-need-to-configure-them) を自動的にセットアップします。
- **継続的な監視:** ZenOpus は、DNS の変更が電子メール配信に影響を与える可能性がある場合に警告します。

## 前提条件

電子メールを使用するには:

- [ZenOpus Cloud](/jp/integrations/cloud) をプロジェクトで有効にする必要があります。
- Workspace は有料プランである必要があります。
- ドメインを所有し、その DNS 設定を管理するためのアクセス権を持っている必要があります。
- ドメインを追加、削除、または確認するには、Workspace 管理者または所有者である必要があります。

## 電子メールの仕組み

電子メール ドメイン ZenOpus を構成する場合:

- ルート ドメインの下にトランザクション送信者サブドメイン (`notify.yourdomain.com` など) を作成します。
- 自動セットアップ フローを使用して DNS 構成をガイドします。
- 必要な DNS レコードをチェックしてドメインを検証します
- ブランド認証メールテンプレートを自動的に生成
- 電子メール ドメイン経由で認証電子メールをルーティングする認証電子メール フックをデプロイします。
- DNS検証が完了したら電子メール配信をアクティブ化します

検証後、認証メールはデフォルトの ZenOpus Cloud 認証送信者の代わりにドメインから送信されます。

### 電子メールドメインの範囲と使用法

電子メール ドメインは Workspace レベルでプロビジョニングされます。

- Workspace には複数の電子メール ドメインを含めることができます
- 検証済みのドメインは複数のプロジェクトで使用できます
- 各プロジェクトは、使用する検証済みドメインを選択します
- プロジェクトごとに一度にアクティブにできる電子メール ドメインは 1 つだけです
- ステータスが **確認済み** のドメインのみが電子メールを送信できます
- 電子メールはプロジェクトごとに有効または無効にできます

プロジェクトで電子メールが無効になっている場合、認証電子メールはデフォルトの ZenOpus Cloud 認証送信者にフォールバックします。

## メールドメインを追加する

メールの送信に使用するドメインを選択します。これは、ユーザーの受信箱に表示されるドメインになります。

開始するには 2 つの方法があります。

- **チャットから**: ZenOpus に電子メールの設定を依頼すると、設定ボタンが表示されます。たとえば:
    ```text
    自分のドメインからパスワードリセットメールを送信したいと考えています。
    ```
- **クラウド設定から**: **クラウド → 電子メール** に移動し、**開始する** をクリックします。  これにより、電子メール設定フローが表示されます。

<Steps>
<Step title="メールドメインを設定する">
- ドロップダウンから既存の Workspace ドメインを選択するか、**新しい電子メール ドメインの追加** を選択して、ルート ドメインまたはサブドメイン (例: `yourdomain.com`) を入力します。
    <Note>
    ドメインには最大 5 つのレベル (`a.b.c.example.com` など) を含めることができます。
    </Note>
- [**続行**] をクリックします。
</Step>
<Step title="送信者のサブドメインを構成する">
認証メールの送信に使用されるサブドメインのプレフィックスを構成します。右側のプレビューには、構成されたドメインからの電子メールの例が含まれるシミュレートされた受信トレイが表示されます。

- デフォルトでは、ZenOpus は配信用に `notify` サブドメイン (`noreply@notify.yourdomain.com` など) を作成しますが、サブドメインは変更できます。
- 必要に応じて、[**@yourdomain.com から送信されたものとして表示**] を有効にすると、トランザクション サブドメインがバックグラウンドで配信を処理している間、受信者には `@yourdomain.com` からのメールが受信トレイに表示されます。
- [**ドメインの設定**] をクリックします。
</Step>
<Step title="完全な DNS 構成">
[**ドメインのセットアップ**] をクリックすると、ZenOpus によって **自動化された DNS 構成フロー** (Entri による) が起動され、ドメインに必要な `NS` ゾーン委任の設定を案内します。

DNS プロバイダーがリストにない場合、または手動制御を希望する場合は、DNS レコードを自分で構成できます。 **ドメイン プロバイダーの選択** エントリ モーダルの一番下までスクロールし、**手動セットアップに進む** を選択します。

自動方法と手動方法の両方で、ドメインの所有権を確認する必要があります。 ZenOpus はドメインをプロビジョニングし、必要な DNS レコードを生成します。

<Tabs>
<Tab title="自動セットアップ">
画面上のプロンプトに従って DNS プロバイダーにログインし、Entri に DNS レコードを更新する権限を与えます。
</Tab>
<Tab title="手動セットアップ">
1. 表示されたDNSレコードをコピーします。
    - ドメイン検証用の `TXT` レコード
    - ZenOpus のネームサーバーを指す `NS` レコード
1. **X/Y レコードをコピーしました** カウンターで進捗状況を追跡します。
1. コピーした DNS レコードをドメイン レジストラーまたは DNS プロバイダーに直接入力します。

他の人が DNS を管理している場合は、これらの指示を同僚に転送することもできます。
</Tab>
</Tabs>
</Step>
<Step title="検証を待ちます">
DNS が構成されると、プロジェクトの [**メール**] ページに戻ります。

ステータス バナーには、ドメインの現在の状態が表示されます。

<Note>
DNS の変更は通常、数時間以内に反映されますが、最大 48 時間かかる場合もあります。
</Note>

DNS 検証の進行中、ZenOpus はすぐに認証電子メール テンプレートを生成して展開します。

自動プロンプトがチャットに表示され、バックグラウンドで実行されます。

```text
Notice.yourdomain.com の ZenOpus 認証メール テンプレートを設定します。 DNS 検証が進行中です。メール テンプレートをアプリのスタイルに合わせます。ブランド カラーを使用し、ロゴがある場合は追加し、アプリと同じトーンと言語でコピーを作成します。
```

ZenOpus は、次の 6 つのブランド認証電子メール テンプレートを自動的に生成して展開します。

- アプリのブランドカラーを適用します
- プロジェクトにロゴが存在する場合はロゴを追加します
- アプリで使用されているトーンと言語を一致させる

DNS 検証が完了する前でも、確認して [電子メール テンプレートをカスタマイズ](#customizing-email-templates) することができます。
</Step>
</Steps>

ZenOpus は、DNS 検証の完了を自動的に検出し、ドメインをアクティブ化します。

ドメインのステータスが **認証済み** に変わると、デフォルトの ZenOpus Cloud 認証送信者ではなく、ドメインから認証メールの送信が開始されます。

検証が完了するまで、認証メールはデフォルトの送信者を使用して送信され続けます。

<Note>
新しくプロビジョニングされたドメインは中立的な評価から始まります。ドメインが一貫した正当な送信を通じて受信トレイ プロバイダーとの信頼を構築している間、初期の配信可能性は変動する可能性があります。詳細については、[電子メール配信のベスト プラクティス](#電子メールの到達性のベスト-プラクティス) を参照してください。
</Note>

## 電子メールドメインのステータス

電子メール ドメインは次のステータスを移行できます。

|ステータス |意味 |
| --- | --- |
|保留中 |セットアップが開始され、DNS 構成が進行中です。
|検証中 | DNS レコードが検出され、検証が進行中です |
|セットアップ |ドメインのプロビジョニングが進行中 |
|検証済み (アクティブ) |ドメインの準備ができており、電子メールの送信に使用できます |
|オフライン |必要な DNS レコードが検証後に変更または削除されました。
|失敗しました |検証またはプロビジョニングに失敗しました |

<Warning>
ドメインは継続的に監視されます。必要な DNS レコードがドリフト、削除、または期限切れになった場合、ドメインは再検証されるまで非アクティブ状態に移行する可能性があります。
</Warning>

## 電子メールドメインの管理

**[クラウド] → [メール]** で、ドメイン ドロップダウンを使用して以下にアクセスします。

- **分析とログ**: 電子メール ドメインの配信メトリックと、そのドメインを使用している現在のプロジェクトの電子メール アクティビティを表示します。
- **ドメインの管理**: すべての Workspace 電子メール ドメインとそのステータス バッジの表示、DNS レコードの検査とコピー、現在のプロジェクトが使用するドメインの切り替え、ドメインの手動検証、ドメインの追加、またはドメインの削除を行います。
    <Warning>
    ドメインの削除は **Workspace 全体のアクション**です。これにより、そのドメインからのメールの送信が停止されます。そのドメインを使用するプロジェクトは、自動的にデフォルトの認証メールに戻ります。
    </Warning>
- **新しいドメインを追加します:** 新しい Workspace 電子メール ドメインを追加します。
- **電子メールを無効にする**: 現在のプロジェクトの電子メールを有効または無効にします。無効にすると、認証メールはカスタム テンプレートではなくデフォルトの Cloud Auth 送信者を使用して送信されます。

### 分析とログ

分析ビューには次のものが含まれます。

- **メトリクス カード**: 電子メール ドメインの送信数、配信数、返送数
- **配信グラフ**: 7、30、または 90 日間にわたるメール ドメインの配信メールと失敗メールの比較
- **電子メール アクティビティ ログ**: 現在のプロジェクトに対して送信された個々の電子メールの拡張可能なリスト。受信者、ステータス (完了、失敗、保留中)、タイプ (認証電子メールまたはテスト電子メール)、タイムスタンプ、および失敗時のエラー メッセージを含む詳細な実行トレースが表示されます。

## 認証メールテンプレートをカスタマイズする

電子メールが有効になると、ZenOpus はプロジェクト用のブランド認証電子メール テンプレートを自動的に生成します。

**メール** ページには、各テンプレートのタブが含まれています。各タブには次の内容が表示されます。

- **差出人**アドレス
- **件名** 行
- iframe でレンダリングされた電子メール テンプレートの **ライブ プレビュー**
- **テストの送信** オプション (アカウントまたはカスタム アドレスへのテスト電子メールの送信など)。
    <Note>
    新しく構成されたドメインから送信された電子メールは、評判が高まるまでに最初はスパムに分類される可能性があります。
    </Note>

### 自動ブランディング

ZenOpus は、次の方法でアプリのブランドを自動的に適用します。

- `src/index.css` からの CSS 変数の抽出 (原色、フォント、境界線の半径)
- `public/` および `src/assets/` のロゴ ファイルの検出
- 検出されたロゴを専用の電子メールアセットストレージバケットにアップロードする
- メールのコピーをアプリのトーンや言語に合わせて調整する

メール本文の外側の背景は常に `#ffffff` であり、メール クライアント全体で一貫したレンダリングが保証されます。内側のコンポーネントにはブランドカラーを使用できます。

### カスタマイズできるもの

ZenOpus に更新を依頼できます。

- コピーとトーン
- ブランドカラー
- レイアウトと構造
- 画像とロゴの配置
- 件名

<Warning>
電子メール クライアント全体で一貫したレンダリングを確保するには、電子メール本文の外側の背景は白 (`#ffffff`) のままにする必要があります。内側のコンポーネントにはブランドカラーを使用できます。
</Warning>

たとえば:

```text
ボタンと見出しに #2563eb を使用してメールを私のブランドに合わせ、上部に私のロゴを追加し、絵文字を削除し、口調をプロフェッショナルだがフレンドリーにし、ボタンをレイアウトの上部に移動し、Support@myapp.com のフッターを追加し、件名を私のアプリ名で始まるように更新します。
```

ZenOpus は、必要な認証変数と安全なコールバック リンクを維持しながら、テンプレートを更新します。

### テンプレートを手動で編集する

コードを直接編集したい場合は、次のようにします。

- 認証テンプレートは `supabase/functions/_shared/email-templates/` にあります。
- 送信ロジックは `supabase/functions/auth-email-hook/` にあります

テンプレートは、インライン スタイルを持つ React Email コンポーネントを使用します (電子メール クライアントは Support 外部 CSS を使用しません)。

必要な認証変数とコールバック リンクはそのままにしておく必要があります。そうしないと、認証フローが中断されます。

テンプレート ファイルまたは件名の編集後、変更を有効にするには、ZenOpus に `auth-email-hook` 関数を再デプロイするように要求する必要があります。再デプロイメントにより、認証電子メールをレンダリングして送信するバックエンド機能が更新されます。

たとえば:

```text
電子メール テンプレートを再展開する
```

## 電子メールの到達性のベスト プラクティス

新しいドメインから認証メールを送信する場合、受信トレイの配置が安定するまでに時間がかかる場合があります。受信トレイ プロバイダーは、エンゲージメント、認証、送信パターンに基づいて、送信者の評判を長期的に評価します。

配信可能性を向上させ、送信者の強力な評判を構築するには、次のプラクティスに従ってください。

<AccordionGroup>
<Accordion title="新しいドメインを徐々にウォームアップする">
新しいメール ドメインには送信履歴がありません。スパム送信者は新しいドメインを使用し、すぐに放棄することが多いため、受信トレイプロバイダーはそれらを慎重に扱います。

最初の配信可能性は、最初の数日または数週間で変動する可能性があります。これは正常です。

実際のユーザーがサインアップやパスワードのリセットなどの認証メールに関与するにつれて、ドメインの評判は徐々に構築されていきます。

専用のトランザクション サブドメインを使用すると、認証トラフィックを分離し、プライマリ ドメインの評判を保護することができます。

**一般的なタイムライン**

- 最初の数日間: テストメールがスパムに分類される可能性があります
- 最初の数週間: 正規のトラフィックによって評判が高まるため、到達性が向上します
- 最初の数か月間: 一貫した熱心な送信により評判が安定します

**役立つこと:**

- 通常の認証トラフィック (ユーザーのサインアップ、パスワードのリセット) で送信を開始します。
- ユーザーベースの拡大に合わせて自然にボリュームを拡大しましょう
- 数週間、数か月にわたって一貫した送信パターンを維持する
- 認証メールが予期されており、実際のユーザーのアクションによってトリガーされることを確認する
- 受信トレイで正規のメールを「スパムではない」としてマークする

**痛いところ:**

- 新しいドメインからのメールの量が突然急増する
- セットアップ直後にテストメールを一括送信する
- 一貫性のない送信 (長いギャップの後に突然のバーストが続く)
- ドメインまたは送信者 ID に対するパニックによる変更

評判は、時間の経過とともに、一貫した正当な送信動作によって自動的に構築されます。
</Accordion>
<Accordion title="ドメイン認証が有効であることを確認してください">
電子メールが受信トレイに到達するには、認証に合格する必要があります。 ZenOpus は、電子メール ドメインを設定するときに、`SPF`、`DKIM`、および `DMARC` を自動的に構成します。

DNS レコードが変更されているか構成が間違っている場合、認証が失敗し、電子メールがスパムに分類される可能性があります。

**クラウド → 電子メール**でドメインのステータスが**認証済み**であることを必ず確認してください。
</Accordion>
<Accordion title="ユーザーがトリガーした正当なメールのみを送信する">
認証メールは、サインアップやパスワードのリセットなど、実際のユーザーのアクションによってのみトリガーされる必要があります。

テストや自動化を目的として人為的なトラフィックを生成することは避けてください。不自然なトラフィック パターンは、ドメインの評判に悪影響を与える可能性があります。
</Accordion>
<Accordion title="認証メールを厳密にトランザクションに保つ">
認証メールはユーザーの最近のアクションと明確に一致し、メールが送信された理由を説明する必要があります。

**良い実践方法**:

- わかりやすい件名を使用してください (「パスワードをリセット」、「メールを確認してください」)
- ユーザーが何をする必要があるかを明確に説明する
- 電子メールとユーザーが実行したアクションを照合します

**避けてください**:

- マーケティング コンテンツまたはアップセルを認証メールに追加する
- 誤解を招くまたはクリックベイトの件名
- 認証メール内の宣伝文言
</Accordion>
<Accordion title="一貫した送信者 ID を維持する">
一貫したものを使用します。

- 送信元アドレス
- 送信者名
- ドメイン

送信者の ID が頻繁に変更されると、受信トレイ プロバイダーとの信頼シグナルが減少します。
</Accordion>
<Accordion title="スパムトリガーとなるフォーマットを回避する">
電子メール テンプレートをカスタマイズする場合:

**実行してください**:

- 明確で説明的な件名を使用する
- フォーマットをシンプルかつクリーンに保つ
- バランスの取れたテキストと画像の比率を使用する
- ブランド名と認識可能な送信者アドレスを含めてください
- 可能な限り、電子メール内のリンクが送信ドメインと一致していることを確認してください

**しないでください**:

- 件名にはすべて大文字を使用してください
- 句読点を過剰に追加する (!!!)
- 誤解を招く件名を作成する
- 画像を多用したレイアウトを使用する
- 無関係または不一致のドメインへのリンク

受信トレイプロバイダーは、送信者ドメインとリンクで使用されるドメイン間の一貫性を探します。電子メールが `yourdomain.com` から送信されている場合、電子メール内のリンクは理想的には `yourdomain.com` またはそのサブドメインを指す必要があります。不一致のドメインはフィッシング攻撃によく使用され、スパム フィルタリングを引き起こす可能性があります。

明確で予測可能な書式設定とドメインの調整により、信頼とエンゲージメントが向上します。
</Accordion>
<Accordion title="直帰率と苦情を監視する">
高い返送率は送信者の評判にダメージを与えます。

**分析とログ**を使用して以下を監視します。

- 送信済み
- 納品済み
- 跳ね返った

無効なアドレスまたは入力ミスによるハード バウンスは特に有害です。無効なアドレスに繰り返し送信すると、受信トレイの配置が減少する可能性があります。

直帰率が増加した場合:

- 電子メールアドレスの収集方法を確認する
- サインアップ時に入力を検証する
- 失敗した配信を繰り返し再試行しないようにする

スパムの苦情も評判に悪影響を及ぼします。認証メールは常に期待され、明確に説明される必要があります。
</Accordion>
<Accordion title="内部テストを制限する">
**[クラウド] → [電子メール]** の [**テストを送信**] オプションは検証には便利ですが、テストは制限されます。

**これが重要な理由**:

- 受信トレイプロバイダーは、送信するすべてのメールからのエンゲージメントシグナルを評価します
- 同じアドレスに繰り返しテストメールを送信すると、人為的なトラフィック パターンが作成されます
- テストメールを偽のアドレスに返送すると評判が傷つきます

**より良いアプローチ**:

- 自分自身またはチームにテストメールをいくつか送信します
- 自分が管理する実際の電子メール アドレスを使用する
- 大量のテストメールの送信を避ける
</Accordion>
<Accordion title="メールがスパムに入った場合">
認証メールがスパムとしてマークされている場合:

1. ドメインのステータスが **確認済み** であることを確認します
1. DNS レコードが正しく構成されていることを確認する
1. 件名と内容を確認する
1. **分析とログ**で直帰率を確認する
1. 送信量の突然の急増を軽減する
1. 我慢してください。新しいドメインは受信トレイプロバイダーとの信頼を構築するのに時間がかかります

一貫した正当な送信動作により、時間の経過とともに配信性が向上します。
</Accordion>
<Accordion title="積極的なエンゲージメントシグナルを奨励する">
受信トレイプロバイダーは、送信者の評判を評価する際にエンゲージメントを考慮します。

認証メールは、ユーザーが次のような場合に自然にポジティブなシグナルを生成します。

- メールを開く
- 確認またはリセットのリンクをクリックします
- 意図したアクションを完了する

明確なメッセージと予想されるフローにより、エンゲージメントが向上し、長期的な受信箱の配置が向上します。
</Accordion>
<Accordion title="受信トレイプロバイダーのアルゴリズムを理解する">
受信トレイ プロバイダーは、スパム フィルタリング基準を公に共有しません (スパマーがシステムを悪用するのを防ぐため)。ただし、一般的には次のように評価します。

- **送信者認証**: `SPF`、`DKIM`、および `DMARC` は正しく構成されていますか? (ZenOpus はこれを自動的に処理します)
- **送信者の評判**: バウンス、苦情、エンゲージメントの履歴はどのようなものですか?
- **コンテンツの品質**: 件名は明確ですか?フォーマットはきれいですか?リンクはドメインと一致していますか?
- **エンゲージメントシグナル**: ユーザーはメールを開きますか?リンクをクリックしますか?返事？またはスパムとしてマークします
- **送信パターン**: 一貫して送信していますか?それとも突然の爆発でしょうか？

「フィッシング詐欺師は何をするでしょうか?」と考えてみましょう。次に、その逆を行います。受信トレイプロバイダーはユーザーを保護したいと考えているため、あらゆる段階で悪意のない正当な送信者であることを証明してください。
</Accordion>
</AccordionGroup>

## トラブルシューティング

<AccordionGroup>
<Accordion title="メールドメインが「検証中」のままになる">
考えられる原因:

- DNS レコードが完全に伝播されていない
- `NS` 委任が正しくありません
- `TXT` 検証レコードがありません
- 間違ったドメイン レベルで構成された DNS

**何をすべきか:**

- **クラウド → 電子メール** に表示される正確な DNS レコードを再確認します。
- レコードが正しいルート ドメインに追加されたことを確認する
- 伝播まで最大 48 時間待ちます
- 検証を再試行する
</Accordion>
<Accordion title="ドメインのステータスがオフラインと表示される">
これは、必要な DNS レコードが検証後に変更、削除、または期限切れになったことを意味します。

**何をすべきか:**

- **クラウド → 電子メール**で DNS レコードを確認する
- 欠落した `NS` または `TXT` レコードを復元する
- ドメインを再検証する
</Accordion>
<Accordion title="メールが送信されない">
確認してください:

- 電子メール ドメインがプロジェクトにリンクされています
- ドメインのステータスは **確認済み** です
- プロジェクトでは電子メールが有効になっています
- クラウド認証が有効になっています
- サポートされている認証メールタイプをトリガーしています

テンプレート タブで **送信テスト** を使用し、配信の失敗やバウンスがないか **分析とログ** を確認します。
</Accordion>
<Accordion title="分析における直帰率が高い">
一般的な原因:

- 無効なメールアドレス
- サインアップ時のタイプミス
- 無効なアドレスへの繰り返しの再試行

サインアップ時に電子メール入力を検証し、ハード バウンスの再試行を回避します。
</Accordion>
<Accordion title="テストメールがスパムに分類される">
これは、新しくプロビジョニングされたドメインでは一般的です。

大量の内部テスト電子メールの送信は避けてください。ユーザーがトリガーしたオーガニックメールが時間の経過とともに評判を築けるようにします。
</Accordion>
<Accordion title="メールテンプレートの変更が表示されない">
テンプレート ファイルを編集したにもかかわらず、ユーザーが依然として古いバージョンを受信している場合:

- `auth-email-hook` 関数が再デプロイされていることを確認してください
- ZenOpus に電子メール テンプレートを再展開するように依頼します

テンプレートの変更は再デプロイ後にのみ有効になります。
</Accordion>
<Accordion title="外部ツールを使用して配信可能性の問題を調査する">
電子メールが常にスパムに分類されたり、拒否されたりする場合は、次のツールを使用してより詳細な診断を行うことができます。

これらのツールはオプションであり、大量の送信者に最も役立ちます。

- [Google ポストマスター ツール](https://postmaster.google.com/)  
送信者の評判、スパム率、Gmail 受信者の認証調整を表示します。
- [Yahoo Sender Hub](https://senders.yahooinc.com/)  
Yahoo の受信トレイの苦情率と送信者の評判を監視します。
- [Spamhaus ブロックリストの検索](https://check.spamhaus.org/)  
ドメイン、IP アドレス、または電子メールの URL が主要なブロックリストに含まれているかどうかを確認します。
- [Google セーフ ブラウジング](https://transparencyreport.google.com/safe-browsing/search)  
電子メールで使用されているリンクをテストして、安全でないというフラグが立てられていないことを確認します。

これらのツールは、受信トレイの配置に影響を与える可能性のある評判やブロックリストの問題を診断するのに役立ちます。
</Accordion>

## よくある質問

<AccordionGroup>
<Accordion title="ZenOpus の認証メールとは何ですか?">
認証メールを使用すると、ZenOpus Cloud アプリは、サインアップ確認、パスワード リセット、マジック リンク、招待状、メール変更確認、再認証コードなどのアカウント関連のメールを、デフォルトの ZenOpus Cloud 認証送信者の代わりに独自のドメインから送信できるようになります。

これらのメールは、ユーザーがセキュリティ関連のアクションを実行すると、Cloud Auth によって自動的にトリガーされます。
</Accordion>
<Accordion title="認証メールはトランザクションメールと同じですか?">
認証メールはトランザクションメールのサブセットです。すべての認証メールはトランザクションメールですが、すべてのトランザクションメールが認証メールであるわけではありません。

電子メール Support **認証電子メールのみ**。
</Accordion>
<Accordion title="マーケティングメールを送信できますか?">
いいえ。認証メールのみがサポートされています (サインアップの確認、パスワードのリセット、マジック リンク、招待、メールの変更、再認証)。

マーケティング電子メール、ニュースレター、注文確認メール、その他の認証されていないトランザクション電子メールは、現時点ではサポートされていません。
</Accordion>
<Accordion title="電子メールには ZenOpus Cloud が必要ですか?">
はい。電子メールには ZenOpus Cloud が必要であり、Cloud Auth と統合して認証電子メールを送信します。
</Accordion>
<Accordion title="電子メール ドメインを使用するにはカスタム Web ドメインが必要ですか?">
いいえ。Web カスタム ドメインは、アプリがホストされる場所を制御します。カスタム電子メール ドメインは、電子メールの送信元のドメインを制御します。

Web ホスティングと電子メール配信では異なる DNS レコードとインフラストラクチャが使用されるため、これらは別個の構成になります。

DNS を制御している限り、電子メール送信にまったく異なるルート ドメインを使用できます。 ただし、ユーザーの信頼性と配信可能性を確保するために、製品ブランドと一致するドメインを使用することをお勧めします。

カスタム ドメインでアプリをホストしても、電子メール送信は自動的に構成されません。
</Accordion>
<Accordion title="メール送信にサブドメインを使用できますか?">
はい。 ZenOpus は、配信用に `notify.yourdomain.com` などのトランザクション サブドメインを自動的に作成します。

オプションで、メールの配信がバックグラウンドでサブドメイン経由で行われる一方で、電子メールをルート ドメインから送信されたものとして表示することができます。

ルート ドメインの評判を保護するのに役立つため、専用の電子メール サブドメインを使用することをお勧めします。
</Accordion>
<Accordion title="SendGrid や Resend などの外部電子メール プロバイダーが必要ですか?">
いいえ。ZenOpus は、ドメイン検証、DNS 構成、`SPF`、`DKIM`、`DMARC` などの認証レコード、配信インフラストラクチャを管理します。 API キーやサードパーティの電子メール プロバイダー アカウントを管理する必要はありません。
</Accordion>
<Accordion title="SPF、DKIM、DMARC とは何ですか? これらを構成する必要がありますか?">
`SPF`、`DKIM`、および `DMARC` は、受信トレイ プロバイダーが電子メールが正規のものであり、なりすましされていないことを確認するのに役立つ電子メール認証標準です。

電子メールを送信すると、Gmail や Outlook などのプロバイダーは次のことをチェックします。

- 電子メールはドメインから送信されることが承認されています
- メッセージは送信中に変更されていません
- ドメイン所有者は、不審な電子メールを処理するためのポリシーを定義しています

認証が失敗すると、電子メールがスパムとしてマークされたり、拒否されたり、ドメインの評判が損なわれたりする可能性があります。

ZenOpus は以下を自動的に構成および維持します。

- **SPF (Sender Policy Framework):** ドメインに電子メールを送信できるサーバーを承認します。
- **DKIM (DomainKeys Identified Mail):** 改ざんを防ぐために電子メールに暗号化署名します。
- **DMARC (ドメインベースのメッセージ認証、レポート、および適合性):** プロバイダーが失敗した認証を処理し、なりすましから保護する方法を定義します。

これらのレコードを手動で構成する必要はありません。 ZenOpus はドメイン検証中にそれらを設定し、継続的に監視します。認証レコードが変更、削除、または期限切れになった場合は、復元できるように通知されます。
</Accordion>
<Accordion title="1 か月あたり何通の認証メールが含まれますか?">
有料の Workspace には、**毎月 50,000 件の認証メール**が追加料金なしで含まれています。

使用量は Workspace 全体にわたって計算されます。

追加のメールには **1,000 メールあたり 1 ドル** が課金されます。
</Accordion>
<Accordion title="無料プランでもメールは利用できますか？">
いいえ。電子メールは **有料プランのみ**で利用できます。 Free プランでは、デフォルトの ZenOpus Cloud 認証送信者が使用されます。
</Accordion>
<Accordion title="DNS 検証にはどのくらい時間がかかりますか?">
DNS の変更は通常、数時間以内に反映されますが、最大 48 時間かかる場合もあります。

**クラウド → 電子メール**でドメインのステータスを確認できます。検証が完了すると、ステータスは自動的に更新されます。
</Accordion>
<Accordion title="私のメールがスパムになってしまうのはなぜですか?">
新しい電子メール ドメインは、送信レピュテーションなしで開始されます。

認証メールがスパムに入った場合:

• ドメインのステータスが **確認済み** であることを確認します。  

 • DNS レコードが変更されていないことを確認する  

 • 送信量の突然の急増を避ける  

 • スパムトリガーとなるフォーマットを避ける  

 • **分析とログ**で直帰率を確認する

一貫した正当なユーザー アクティビティにより、配信可能性は時間の経過とともに向上します。
</Accordion>
<Accordion title="メールテンプレートをカスタマイズできますか?">
はい。以下をカスタマイズできます。

- コピーとトーン
- ブランドカラー
- レイアウトと構造
- ロゴと画像
- 件名

ZenOpus にテンプレートの更新を依頼したり、`supabase/functions/_shared/email-templates/` で直接編集したりすることができます。

必要な認証変数とコールバック リンクはそのままにしておく必要があります。

電子メール クライアント全体で一貫したレンダリングを確保するには、電子メール本文の外側の背景は白 (`#ffffff`) のままにする必要があります。内側のコンポーネントにはブランドカラーを使用できます。
</Accordion>
<Accordion title="電子メールを無効にするとどうなりますか?">
プロジェクトで電子メールが無効になっている場合、認証電子メールはブランド テンプレートの代わりにデフォルトの ZenOpus Cloud 認証送信者を使用して送信を続けます。
</Accordion>
<Accordion title="電子メールのドメインを削除するとどうなりますか?">
電子メール ドメインの削除は、**Workspace 全体のアクション**です。

そのドメインを使用しているすべてのプロジェクトは、すぐにデフォルトの ZenOpus Cloud 認証送信者にフォールバックします。
</Accordion>
</AccordionGroup>
