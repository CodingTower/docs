ZenOpus は、アプリケーションが期待どおりに動作することを確認するいくつかの方法を提供します。これらのツールは、問題を再現し、修正を確認し、フロントエンドとバックエンドの動作全体の回帰を回避するのに役立ちます。

<Tip>
**適切なテストを選択してください**

どのテスト ツールを使用すればよいかわからない場合は、検証したい動作の種類から始めます。

- 問題がユーザーに見える場合は、ブラウザーのテストを使用します。
- UI の動作が長期間にわたって正しい状態を維持する必要がある場合は、フロントエンド テストを使用します。
- 問題がバックエンド ロジックにある場合は、エッジ関数を直接呼び出してから、エッジ テストを追加します。
</Tip>

## ZenOpus が検証中に観察できること

ZenOpus は、エージェントが検証ツールを実行するときに情報を収集します。この情報はこれらのツールの一部として自動的に収集され、アプリで何が起こっているかを理解するために使用されます。

ZenOpus は以下をキャプチャできます:

- ブラウザテスト中のブラウザコンソールログ
- ブラウザのテスト中に行われたネットワーク リクエスト
- テストの失敗とテスト実行によるビルド エラー
- バックエンド関数を直接呼び出す場合のリクエストとレスポンスのデータ

これらの信号は、関連ツールを使用するときはいつでも利用でき、多くの場合、追加の手動デバッグを行わなくても問題を診断するには十分です。

ZenOpus は、それぞれ異なる種類の検証用に設計された複数のテスト アプローチをサポートしています。ほとんどのテストは、次の 2 つのカテゴリのいずれかに分類されます。

- 完全なユーザー フローをチェックする
- 特定のルールや動作を個別にチェックする

## ブラウザのテスト

[ブラウザ テスト](/jp/features/browser-testing) は、ブラウザでアプリを操作することにより、実際のユーザーの動作をチェックします。エージェントは、ページを移動し、ボタンをクリックし、フォームに記入し、リクエストを送信できます。その際、スクリーンショットをキャプチャし、UI 要素を検査し、コンソールとネットワークのアクティビティを観察できます。

次の場合にブラウザ テストを使用します。

- 誰かが壊れた、または予期しない UI の動作を報告しました
- オンボーディングやチェックアウトなどの複数のステップのフローを確認する必要がある
- 問題はルーティング、認証、タイミング、または状態に依存する可能性があります
- ユーザー視点で変更を確認したい

ブラウザーのテストは他の方法に比べて時間がかかりますが、ユーザーが実際に体験していることを最も明確に把握できます。

調査中にエージェントによってブラウザのテストが開始される場合があります。明示的にリクエストすることもできます。

**例**

```text
ブラウザーのテストを使用して、チェックアウト フローを検証します。
```

## フロントエンドのテスト

フロントエンド テストでは、明確なアサーションを使用して UI の動作を個別に検証します。これらは、特定の UI ロジックが長期にわたって期待どおりに動作し続けることを保証するのに役立ちます。

フロントエンド テストはシミュレートされたブラウザ環境で実行され、迅速に実行されて一貫した結果が得られ、通常は `.test.tsx` ファイルとしてコンポーネントの隣に存在します。

**テストスタック**

- ヴィテスト
- React テスト ライブラリ
- ジェスダム

次の場合にフロントエンド テストを使用します。

- 問題は明確なルールまたは期待として説明できます
- UI の回帰を修正しており、それが再発しないようにしたい場合
- 条件付きレンダリングまたはフォーム、テーブル、フィルターなどの複雑な状態を操作している場合
- 完全なアプリを実行せずに迅速なフィードバックが必要な場合

フロントエンド テストは、要求された場合にのみ実行されます。

**例**

```text
ログイン フォームのフロントエンド テストを作成して実行します。
```

## エッジ機能によるバックエンド検証

ZenOpus は、エッジ関数でバックエンドの動作を検証するための 2 つの補完的な方法 (直接呼び出しと自動テスト) を提供します。これらのアプローチは、多くの場合、一緒に使用されます。

### 直接エッジ関数呼び出し

直接呼び出しを使用すると、エージェントは特定の入力を使用してエッジ関数を実行し、結果をすぐに検査できます。これにより、UI 関連の複雑さが回避され、迅速な分離に役立ちます。ログインしている場合は、認証された呼び出しで現在のセッションを使用できます。

次の場合に直接呼び出しを使用します。

- バックエンドの問題が疑われ、それを UI から分離したい場合
- 特定のパラメータまたは入力を確認する必要がある
- 変更前と変更後の動作を比較したい
- 認証された動作をデバッグしています

ダイレクト エッジ関数呼び出しは、要求された場合にのみ実行されます。

**例**

```text
無効な電子メールを使用して、サインアップ エッジ機能を直接呼び出します。
```

### エッジテスト

エッジ テストは、時間をかけてバックエンド ルールをチェックする自動テストです。これらは、重要なロジックが突然壊れないようにするのに役立ちます。

**テストスタック**

- Deno 組み込みテストランナー
- ネイティブ TypeScript Support

次の場合にエッジ テストを使用します。

- エッジ関数を変更し、回帰保護が必要な場合
- ビジネスルールまたは権限を検証している
- 問題は微妙であり、手動で確認するのは困難です

エッジ テストは、要求された場合にのみ実行されます。

**例**

```text
支払い処理機能のエッジ テストを作成します。
```

### バックエンド検証ツールがどのように連携するか

直接呼び出しとエッジ テストは、バックエンドの動作をデバッグするときによく一緒に使用されます。

一般的なシーケンスは次のようになります。

1. エッジ関数を直接呼び出して、特定の入力で問題を再現します。
1. 動作を理解したら、修正を適用します。
1. 同じ入力で関数を再度呼び出して、変更を確認します。
1. エッジ テストを追加して、時間が経っても動作が後退しないことを確認します。

このアプローチでは、長期的な検証を提供しながら、反復を高速に保ちます。

## よくある質問

<AccordionGroup>
<Accordion title="変更ごとにすべてのテストを実行する必要がありますか?">
いいえ。ほとんどの問題は、単一のツールを使用して理解および検証できます。ブラウザー テストはユーザーに見える問題に役立ち、フロントエンド テストは特定の UI 動作をロックダウンするのに役立ち、バックエンド検証はビジネス ロジックに最適です。調査している問題の種類に一致するツールを選択してください。
</Accordion>
<Accordion title="数回試しても問題が解決しない場合はどうすればよいですか?">
数回試しても問題が解決しない場合は、ZenOpus に複数の検証ツールを使用するように依頼できます。 ZenOpus はこれらのツールを実行し、その出力を使用して次のステップをガイドします。

- 以前の実行のログやテストの失敗を確認すると、原因を絞り込むのに役立つことがよくあります。
- 通常、バックエンドの問題は、エッジ関数を直接呼び出すことで簡単に分離できます。
- ユーザーが直面する問題は、ブラウザーのテストで調査するのが最適です。
- 動作を理解したら、自動テストを追加すると、長期にわたって動作を安定させることができます。

それでも問題が解決しない場合は、最新の安定したバージョンに戻してください。
</Accordion>
<Accordion title="ZenOpus は検証ツールを自動的に実行しますか?">
ほとんどの検証ツールは、ユーザーが要求した場合にのみ実行されます。場合によっては、エージェントが調査中にツールを提案または開始することがありますが、検証はバックグラウンドでサイレントに実行されません。
</Accordion>
</AccordionGroup>
