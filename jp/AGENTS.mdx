文書作成エージェントの指示

ZenOpus は、ユーザーが自然言語プロンプトを通じて Web アプリケーションを構築できるフルスタック アプリ開発プラットフォームです。

これらの手順は、Mintlify AI エージェントが docs.zenopus.dev 全体でドキュメントを作成および更新する方法を定義します。特に指示がない限り、一貫してそれらに従ってください。

## 精度と安全性

- 製品の動作、制限、価格設定、プランの利用可能性、または UI 要素を発明したり想定したりしないでください。
- どのプランの機能が利用可能であるかを必ず指定してください (Free、Pro、Business、Enterprise、または「すべてのプラン」)。
- Workspace クレジット (構築用) とクラウド/AI 残高 (デプロイされたアプリ用) を区別します。
- 情報が明確に文書化または提供されていない場合は、推測しないでください。
- 不確実な内容や推測的な内容を追加するよりも、詳細を省略することを好みます。
- 必要に応じて、仮定を立てたり、ギャップを埋めたりするのではなく、説明を求めます。
- 該当する場合は、不可逆的なアクション、制限、不明な点について明確に説明します。

## 必要な情報の流れ（コアルール）

ドキュメント ページを作成または更新するときは、次の説明の順序に従ってください。

1. **内容**: 機能、コネクタ、またはコンセプトが何であるかを説明します。
    - これを 1 ～ 2 つの短い段落にとどめてください。
1. **理由**: なぜそれが重要なのか説明してください。
    - 価値、結果、いつそれを使用するか (または使用しないか) に焦点を当てます。
1. **対象者**: 対象者を説明してください。
    - 具体的なユースケース、サンプルアプリ、シナリオ、またはテーブルを使用します。
1. **方法**: 仕組みや使用方法を説明します。
    - セットアップ手順、構成、要件 (アカウント、資格情報)、請求と使用上の注意、制限事項、および元に戻せないアクションが含まれます。
    - **概念的なコンテンツまたは参照コンテンツ**の場合、「方法」は、段階的な指示ではなく、概念がどのように機能するかを高レベルで説明する場合があります。
1. **FAQ**: よくある質問を追加して、わかりやすさ、AI アシスタントの検索、検索ランキングを向上させます。

## 必須のセクション順序 (デフォルト)

コンテンツが明らかに該当しない場合を除き、この見出し順序を正確に使用してください。

- イントロ（何）
- なぜ、いつ使用するのか
- 一般的な使用例とアプリ例 (誰が)
- 要件と請求
- セットアップ/構成 (方法)
- よくある質問
- 制限事項とトラブルシューティング (オプション)

## 含めるべきもの（内容チェックリスト）

- ユーザーが API キー、アカウント、環境設定、または依存関係を必要とする場合は、**前提条件** を追加します。
- **よくある質問** を追加して、検索エンジンと Mintlify AI アシスタントがコンテンツを理解してランク付けできるようにします。
- Support チケットで見られる一般的な問題について、**トラブルシューティング セクション**を追加します。
- 該当する場合は、**請求の所有権**、**使用量の消費**、**元に戻せないアクション**を明確に説明します。

## よくある質問（必須）

- ほとんどのページに **3 ～ 6 個の FAQ** を含めます (ページ サイズに応じて)。
- ユーザーが実際に尋ねる**自然言語の質問**として FAQ を作成します。
- 以下に関する質問を優先します。
    - 請求と所有権 (ZenOpus と外部プロバイダー)
    - アカウント、認証情報、または API キーの要件
    - 制限、クォータ、またはレート制限
    - 削除、取り消し、または元に戻せないアクション
    - よくある誤解や失敗例
- 回答は簡潔、事実に基づいた、ざっと読み取れるものにしてください。
- マーケティング用語を避けてください。明瞭さと検索精度を最適化します。

## 執筆原則 (グローバル)

- イントロは読みやすく短くしてください。
- 「理由」セクションでは、機能よりも結果を優先します。
- 「誰が」と「どのように」で、具体的で現実的な例を使用します。
- 関連する場合は正確な UI パスを含め、すべての手順を明確にしてください。

## SEOとアクセシビリティ

- ユーザーが検索する内容に合わせて、説明的でキーワードが豊富な見出しを使用します。
- フロントマター `title` と `description` が明確で具体的で、ユーザー中心であることを確認します。
- 見つけやすさを向上させる場合は、frontmatter に `keywords` 配列を追加します (ユーザーが検索するであろう関連性のある特定の用語を使用し、キーワードの詰め込みを避けます)。
- アクション指向のページタイトルを好みます。役に立つ場合は、タイトルを長くしてより説明的なものにすることができます。
- ページ タイトルが長い場合は、ナビゲーション用に短いバージョンの `sidebarTitle` を追加します。
- 具体的で実用的なリンク テキストを使用します (「ここをクリック」は避けてください)。
- 論理的な見出し階層を維持します (レベルをスキップしないでください)。
- すべての画像と図に説明的な代替テキストを追加します。
- 短い段落、リスト、明確なセクション区切りを使用して、読みやすいようにコンテンツを構造化します。

## Mintlify コンポーネント (いつ使用するか)

Mintlify コンポーネントを意図的に使用して、明瞭さと読みやすさを向上させます。コンポーネントを使いすぎないでください。

### コールアウト

- メインコンテンツをサポートする便利なコンテキストには `<Note>` を使用します。
- ベスト プラクティスまたは専門家のアドバイスについては、`<Tip>` を使用してください。
- 破壊的なアクション、元に戻せない手順、セキュリティ上の問題、または重大な制限には `<Warning>` を使用します。
- `<Check>` を使用して成功を確認するか、検証手順を提供します。

### ステップ

- 複数ステップの手順、特にセットアップと構成には、`<Steps>` と `<Step>` を使用します。
- アクションを順番に完了する必要がある場合は、ステップを優先します。
- 可能であれば、検証手順または成功インジケーターを含めます。

### タブ

- プラットフォーム固有の命令または相互に排他的な代替命令には、`<Tabs>` を使用します。
- 連続したステップにはタブを使用しないでください。

### アコーディオン

- トラブルシューティング、FAQ、エッジケース、または高度な構成には `<AccordionGroup>` を使用します。
- コアフローを可視化しておきます。アコーディオンの背後にある詳細を隠します。

### コードグループ

- 同じ例を複数の言語で表示する場合は、`<CodeGroup>` を使用します。
- 関連する言語が 1 つだけの場合は、1 つのコード ブロックを使用します。

## 聴衆とトーン

- 主に **技術者以外の読者**を対象に書きます。
- 単純化しすぎず、自明ではない概念を説明してください。
- **能動態**と**二人称** (「あなた」) を使用します。
- マーケティング用語よりも、明確さ、正確さ、有用性に重点を置きます。

## スタイルと書式設定

- **アクティブな音声**: 誰がアクションを実行するかを明確にするために、アクティブな音声を推奨します。
    - 正解: 「プロジェクトを削除します。」
    - 不正解: 「プロジェクトは削除されました。」
    - 例外: 状態や結果を強調する場合は受動態を使用します。
        - 例: 「プロジェクトが保存されました。」
- **現在形**:
    - 一般的な動作には現在形を使用します。
        - 正解: 「[保存] をクリックして設定を更新します。」
    - 未来時制は将来の結果に対してのみ使用してください。
        - 正解: 「変更内容は失われます。」
- **文例**:
    - すべての見出しに大文字と小文字を使用します。
    - 最初の単語、固有名詞、頭字語のみを大文字にします。
    - 正：「プロジェクト設定」
    - 不正解：「プロジェクト設定」
- **シリアル (オックスフォード) カンマ**:
    - 最後の「and」または「or」の前には必ずカンマを入れてください。
    - 例: 「プロジェクトの編集、共有、削除」。
- **日付**:
    - *月日、年*を使用します (例: *2025 年 7 月 31 日*)。
    - 序数 (1st、2nd、3rd) は使用しないでください。
- **UI リファレンス**:
    - 太字の UI 要素: **設定**、**プロジェクトの削除**。
    - 値、フィールド、または識別子にはインライン コードを使用します: `project_id`。
    - → 区切り記号を使用して正確な UI パスを使用します: **設定 → プライバシーとセキュリティ → デフォルトのプロジェクトの表示設定**

## プロンプトの例

- すべてのプロンプトの例を **プレーンテキスト コード ブロック**に配置します。
- 可読性を維持するには、`text wrap` を使用します。

例:

```text
OpenWeatherMap API を統合します。 
ベース URL: https://api.openweathermap.org/data/2.5。 
認証: クエリ パラメーター appid として渡される API キー。 
現在の天気を取得するためのエンドポイントが必要です: GET /weather?q={city}&units=metric&appid={API_KEY}。 
ドキュメント: https://openweathermap.org/current
```
